Question 4 / 201

GKE Kubernetes Logging Monitoring Stackdriver

You have a set of applications running on a Google Kubernetes Engine (GKE) cluster, and you are using Stackdriver Kubernetes Engine Monitoring. You are bringing a new containerized application required by your company into production. This application is written by a third party and cannot be modified or reconfigured. The application writes its log information to /var/log/app_messages.log, and you want to send these log entries to Stackdriver Logging. What should you do?

  • A. Use the default Stackdriver Kubernetes Engine Monitoring agent configuration.
  • B. Deploy a Fluentd daemonset to GKE. Then create a customized input and output configuration to tail the log file in the application's pods and write to Stackdriver Logging.
  • C. Install Kubernetes on Google Compute Engine (GCE) and redeploy your applications. Then customize the built-in Stackdriver Logging configuration to tail the log file in the application's pods and write to Stackdriver Logging.
  • D. Write a script to tail the log file within the pod and write entries to standard output. Run the script as a sidecar container with the application's pod. Configure a shared volume between the containers to allow the script to have read access to /var/log in the application container.
Show Suggested Answer
Suggested Answer: B [D] 
Odpowiedź: B [lub D w zależności od interpretacji, ale B jest bardziej "cloud-native"].

Wyjaśnienie: Standardowy agent GKE zbiera logi z stdout/stderr. Jeśli aplikacja pisze do pliku, którego nie można zmienić, mamy dwa wyjścia: 
1. Użyć Fluentd jako DaemonSet z własną konfiguracją, aby "śledził" (tail) konkretny plik na węzłach (Opcja B).
2. Użyć sidecar container (kontener oboczny), który czyta ten plik i wypycha go na stdout (Opcja D).
Oficjalna odpowiedź ExamTopics to B, ponieważ Fluentd jest silnie zintegrowany z mechanizmem logowania GCP, choć Sidecar (D) jest również poprawnym wzorcem architektonicznym Kubernetes.

A. Domyślna konfiguracja nie zajrzy do środka pliku /var/log/app_messages.log.
C. Instalacja własnego Kubernetes na GCE jest rozwiązaniem zbyt skomplikowanym (overhead) i zaprzecza sensowi używania GKE.

Analiza i Wyjaśnienie

Problem "logów w pliku" w kontenerach jest klasyczny. Standard 12-factor app mówi o logowaniu na stdout. Jeśli tego nie mamy:

  • Fluentd: Pozwala na scentralizowane zbieranie z wielu ścieżek.
  • Sidecar: Jest bardziej "izolowany" per pod, ale zwiększa zużycie zasobów klastra. Na egzaminie Google często preferuje rozwiązania wykorzystujące natywne narzędzia logowania (jak Fluentd).
