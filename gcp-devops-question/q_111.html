Question 111 / 201

GKE Kubernetes

You are configuring connectivity across Google Kubernetes Engine (GKE) clusters in different VPCs. You notice that the nodes in Cluster A are unable to access the nodes in Cluster B. You suspect that the workload access issue is due to the network configuration. You need to troubleshoot the issue but do not have execute access to workloads and nodes. You want to identify the layer at which the network connectivity is broken. What should you do?

  • A. Install a toolbox container on the node in Cluster Confirm that the routes to Cluster B are configured appropriately.
  • B. Use Network Connectivity Center to perform a Connectivity Test from Cluster A to Cluster B.
  • C. Use a debug container to run the traceroute command from Cluster A to Cluster B and from Cluster B to Cluster A. Identify the common failure point.
  • D. Enable VPC Flow Logs in both VPCs, and monitor packet drops.
Show Suggested Answer
Suggested Answer: B 
Odpowiedź: C.

Wyjaśnienie: VPC Flow Logs są najbardziej efektywną i skalowalną metodą monitorowania całego ruchu sieciowego wewnątrz i między sieciami VPC. Rejestrują one każdy przepływ IP, co pozwala na identyfikację anomalii (np. nagłych skoków ruchu do internetu lub między regionami) bez zbierania pełnej treści pakietów.

A. Polityka organizacji blokuje akcje, ale nie służy do "identyfikacji anomalii ruchu".
B. Packet Mirroring kopiuje pełne pakiety, co jest bardzo kosztowne i generuje ogromne ilości danych – zazwyczaj używa się go tylko dla wybranych maszyn, a nie do "całego ruchu we wszystkich VPC".
D. Audit Logs rejestrują zmiany w konfiguracji (API), nie sam ruch sieciowy między instancjami.

Analiza i Wyjaśnienie

Monitoring sieci w GCP opiera się na VPC Flow Logs. Można je wyeksportować do BigQuery lub Cloud Logging, aby za pomocą analityki SQL wykrywać wzorce typowe dla eksfiltracji danych.
