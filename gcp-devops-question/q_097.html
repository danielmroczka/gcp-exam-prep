Question 97 / 201

Logging

You are creating Cloud Logging sinks to export log entries from Cloud Logging to BigQuery for future analysis. Your organization has a Google Cloud folder named Dev that contains development projects and a folder named Prod that contains production projects. Log entries for development projects must be exported to dev_dataset, and log entries for production projects must be exported to prod_dataset. You need to minimize the number of log sinks created, and you want to ensure that the log sinks apply to future projects. What should you do?

  • A. Create a single aggregated log sink at the organization level.
  • B. Create a log sink in each project.
  • C. Create two aggregated log sinks at the organization level, and filter by project ID.
  • D. Create an aggregated log sink in the Dev and Prod folders.
Show Suggested Answer
Suggested Answer: D 
Odpowiedź: C.

Wyjaśnienie: Przy federacji tożsamości (SSO), hasło użytkownika jest weryfikowane przez Twój lokalny system (Active Directory). Dlatego tam musi być ustawiona silna polityka haseł. Aby dodatkowo zabezpieczyć dostęp do chmury, Google oferuje funkcję "post-SSO 2-Step Verification", która wymusza drugi składnik (najlepiej bezpieczny klucz fizyczny FIDO2) PO tym, jak AD potwierdzi poprawność hasła.

A i B. Jeśli używasz SSO, polityka haseł w Cloud Identity jest ignorowana, bo Google nie widzi Twojego hasła.
D. Kody SMS są znacznie mniej bezpieczne niż klucze fizyczne (podatność na phishing i SIM swapping).

Analiza i Wyjaśnienie

Najbezpieczniejszym modelem logowania hybrydowego jest: Lokalne Hasło + Chmurowy Klucz Fizyczny (Titan/Yubikey). Chroni to przed przejęciem konta nawet w przypadku wycieku hasła z AD.
