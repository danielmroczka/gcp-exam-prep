You need to create a near real-time inventory dashboard that reads the main inventory tables in your BigQuery data warehouse. Historical inventory data is stored as inventory balances by item and location. You have several thousand updates to inventory every hour. You want to maximize performance of the dashboard and ensure that the data is accurate. What should you do?

  • A. Leverage BigQuery UPDATE statements to update the inventory balances as they are changing.
  • B. Partition the inventory balance table by item to reduce the amount of data scanned with each inventory update.
  • C. Use the BigQuery streaming the stream changes into a daily inventory movement table. Calculate balances in a view that joins it to the historical inventory balance table. Update the inventory balance table nightly. Most Voted
  • D. Use the BigQuery bulk loader to batch load inventory changes into a daily inventory movement table. Calculate balances in a view that joins it to the historical inventory balance table. Update the inventory balance table nightly.

Odpowiedź: C

Wyjaśnienie: Wybór C jest najlepszym rozwiązaniem, ponieważ zapewnia optymalną wydajność dla dashboardu, który ma przetwarzać dużą ilość danych w czasie rzeczywistym. Użycie streamingu danych do tabeli ruchu zmian (movement table) pozwala na szybkie i efektywne aktualizowanie danych, minimalizując czas potrzebny na przetwarzanie aktualnych zmian. Partitionowanie tabeli, choć może poprawić wydajność w dłuższej perspektywie, nie jest kluczowe dla tego konkretnego scenariusza, gdzie priorytetem jest szybkie przetwarzanie aktualnych zmian.  Wybór A, B i D jest mniej efektywny, ponieważ wymaga dodatkowych operacji, takich jak tworzenie tabeli ruchu zmian,  i nie jest odpowiedni dla dynamicznego przetwarzania danych w czasie rzeczywistym.  Użycie bulk loaderów, choć potencjalnie szybkie, może generować duże obciążenie serwerów BigQuery, a nie optymalizować wydajność dla dashboardu, który ma być w stanie obsługiwać dużą ilość aktualizacji.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Use BigQuery streaming into a daily movement table. Calculate balances in a view that joins it to the historical balance table. Update nightly.

Dlaczego?

Wymagania: "near real-time inventory dashboard", "thousands of updates every hour", "accurate data". 1. Streaming Inserts: BigQuery jest zoptymalizowany pod kątem dopisywania danych (Append). Strumieniowanie tysięcy zmian na godzinę do pomocniczej tabeli "ruchów" (movement table) jest wydajne i zapewnia świeżość danych. 2. Calculated View: Aby uzyskać aktualny stan magazynowy (balance), dashboard odpytuje widok, który sumuje: [Ostatnie stabilne saldo (history)] + [Suma ruchów z dzisiaj (streaming table)]. To gwarantuje dokładność w czasie rzeczywistym. 3. Nightly Update: Raz na dobę (w oknie o małym ruchu) wykonujemy proces batchowy, który "utrwala" ruchy z całego dnia do tabeli historycznej i czyści tabelę ruchów. To optymalizuje wydajność odczytu na dłuższą metę.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. BigQuery UPDATE: To najczęstszy błąd. BigQuery nie jest bazą OLTP. Instrukcje UPDATE są bardzo ciężkie, mają limity (quota) na liczbę operacji dziennie i są kosztowne. Tysiące update'ów na godzinę szybko zablokowałoby tabelę i przekroczyło limity.
  • B. Partition by item: BigQuery nie wspiera partycjonowania po dowolnej kolumnie tekstowej (jak 'item') – wspiera tylko po czasie, dacie lub liczbach całkowitych (integer range). Nawet jeśli, to nie rozwiązuje problemu wydajności częstych zmian salda.
  • D. Bulk loader: Batchowe ładowanie (co np. godzinę) nie spełnia wymogu "near real-time". Użytkownik widziałby dane opóźnione o cały cykl ładowania.
