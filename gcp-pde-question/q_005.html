An external customer provides you with a daily dump of data from their database. The data flows into Google Cloud Storage GCS as comma-separated values
(CSV) files. You want to analyze this data in Google BigQuery, but the data could have rows that are formatted incorrectly or corrupted. How should you build this pipeline?

  • B. Enable BigQuery monitoring in Google Stackdriver and create an alert.
  • C. Import the data into BigQuery using the gcloud CLI and set max_bad_records to 0.

Odpowiedź: D.

Wyjaśnienie: Użycie Dataflow z mechanizmem obsługi błędów pozwala na przetworzenie danych przy jednoczesnym identyfikowaniu i izolowaniu rekordów, które są niepoprawne lub uszkodzone. Te błędne rekordy mogą zostać przekierowane do osobnej tabeli (tzw. "dead-letter table") do dalszej analizy i debugowania. Jest to najbardziej robustne podejście do zarządzania błędnymi danymi w tym scenariuszu. Takie rozwiązanie zapewnia, że dane, które można przetworzyć, są ładowane do BigQuery, a problemy z danymi są systematycznie rozwiązywane.

A. Użycie źródeł federacyjnych wymagałoby dodatkowej logiki do obsługi błędów danych na poziomie zapytania SQL, co jest mniej efektywne niż przetwarzanie błędów na etapie ładowania.
B. Monitorowanie w Stackdriverze może pomóc w wykrywaniu problemów ogólnych, ale nie zapewnia mechanizmu do aktywnego izolowania i przenoszenia błędnych rekordów podczas procesu importu danych.
C. Ustawienie `max_bad_records` na 0 podczas importu przez gcloud CLI spowoduje zatrzymanie całego zadania importu przy napotkaniu pierwszego błędnego rekordu, co nie jest pożądanym zachowaniem, jeśli chcemy zaimportować jak najwięcej danych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Run a Google Cloud Dataflow batch pipeline....

Dlaczego?

W scenariuszu ETL (Extract, Transform, Load), gdzie jakość danych wejściowych jest niepewna ("data could have rows that are formatted incorrectly or corrupted"), kluczowe jest zastosowanie wzorca, który nie przerywa całego procesu z powodu kilku błędów, ale też nie ignoruje ich po cichu.

Wzorzec Dead Letter Queue (DLQ):

  • Google Cloud Dataflow (oparty na Apache Beam) to idealne narzędzie do budowania skalowalnych potoków przetwarzania danych.
  • Pozwala zdefiniować logikę walidacji (np. "czy ten wiersz CSV ma odpowiednią liczbę kolumn?").
  • Poprawne rekordy są zapisywane do głównej tabeli w BigQuery.
  • Błędne rekordy są kierowane (side output) do osobnej tabeli (tzw. Dead-Letter Table) lub na GCS.
  • To pozwala analitykom na późniejsze zbadanie błędów bez zatrzymywania całego procesu importu ("pipeline robustness").

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Federated data sources: Pozwalają odpytywać pliki CSV bezpośrednio z GCS (External Tables). Choć można to zrobić, wydajność jest niższa niż natywnych tabel, a obsługa błędów formatowania jest trudna – zapytanie po prostu zwróci błąd, jeśli napotka uszkodzony wiersz, zamiast go oddzielić.
  • B. Stackdriver monitoring: Monitoring służy do obserwacji, a nie do naprawy danych czy logiki ETL. Może powiadomić o błędzie, ale nie oddzieli złych danych od dobrych.
  • C. gcloud CLI max_bad_records=0: To ustawienie jest bardzo restrykcyjne. Jeśli napotka choć jeden błąd, cały import zostanie przerwany. Jeśli ustawimy wyższą wartość, błędy będą ignorowane (do limitu), ale niekoniecznie będziemy mieli łatwy sposób ich analizy w osobnej tabeli w ten sam uporządkowany sposób co w Dataflow.
