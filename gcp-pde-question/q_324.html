Your financial services company has a critical daily reconciliation process that involves several distinct steps: fetching data from an external SFTP server, decrypting the files, loading them into Cloud Storage, and finally running a series of BigQuery SQL transformations. Each step has strict dependencies, and the entire process should notify you if not completed by 7:00 AM. You need a highly observable and robust solution that supports easy re-runs of individual steps if errors occur. What should you do?

  • A. Develop a Cloud Composer DAG that includes a single PythonOperator to execute a Python script that runs each step sequentially, incorporating error handling and retries.
  • B. Implement a Cloud Composer DAG, with each step defined as a separate task using appropriate Airflow operators, and schedule the DAG for daily execution.
  • C. Define a Cloud Composer DAG to orchestrate the SFTP fetch and decryption steps, and then use Cloud Scheduler to trigger a separate Dataflow job that handles the Cloud Storage load and BigQuery transformations.
  • D. Create a Cloud Composer DAG that includes a single BashOperator to execute a top-level shell script, which in turn calls individual scripts for each pipeline step.
Show Suggested Answer
Suggested Answer: B

Wyjaśnienie

Correct Answer: B

Cloud Composer (based on Apache Airflow) is the standard GCP tool for orchestrating complex workflows with dependencies. By defining each step as a separate task within a DAG, you gain high observability (you can see which specific task failed in the Airflow UI), robust error handling, and the ability to easily clear and re-run individual tasks without re-running the entire pipeline.

Why other options are incorrect:
A and D. Using a single operator (Python or Bash) to run all steps as one task loses the benefits of Airflow's observability and task-level retries/re-runs. If one step fails, the entire task fails, and you'd have to re-run everything from the start or implement complex internal logic.
C. Dividing the workflow between Composer and Cloud Scheduler/Dataflow creates unnecessary complexity and fragmentation, making it harder to monitor the end-to-end dependency and meet the 7:00 AM notification requirement.
