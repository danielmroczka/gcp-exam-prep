You need to connect multiple applications with dynamic public IP addresses to a Cloud SQL instance. You configured users with strong passwords and enforced the SSL connection to your Cloud SQL instance. You want to use Cloud SQL public IP and ensure that you have secured connections. What should you do?

  • A. Add CIDR 0.0.0.0/0 network to Authorized Network. Use Identity and Access Management (IAM) to add users.
  • B. Add all application networks to Authorized Network and regularly update them.
  • C. Leave the Authorized Network empty. Use Cloud SQL Auth proxy on all applications. Most Voted
  • D. Add CIDR 0.0.0.0/0 network to Authorized Network. Use Cloud SQL Auth proxy on all applications.

Odpowiedź: C.

Pozostawienie pola Authorized Network pustego w połączeniu z użyciem Cloud SQL Auth proxy jest najbezpieczniejszym rozwiązaniem w tej sytuacji. Cloud SQL Auth proxy tworzy zaszyfrowany tunel do instancji Cloud SQL, eliminując potrzebę otwierania instancji na ruch z publicznych adresów IP. Dzięki temu, nawet jeśli aplikacje mają dynamiczne publiczne IP, dostęp do instancji jest zabezpieczony, ponieważ proxy uwierzytelnia połączenie i szyfruje dane. Użycie proxy w połączeniu z silnymi hasłami i wymuszonym połączeniem SSL zapewnia kompleksowe zabezpieczenie.  Nie polegamy na listach kontroli dostępu (ACL) opartych na adresach IP, które mogą być trudne do utrzymania w przypadku dynamicznych adresów.

A. Otwarcie na 0.0.0.0/0 jest bardzo niebezpieczne i naraża instancję na ataki z całego internetu, nawet z silnymi hasłami i SSL.
B. Regularne aktualizowanie listy autoryzowanych sieci jest uciążliwe i podatne na błędy, a także nie eliminuje ryzyka związanego z publicznymi adresami IP.
D. Połączenie otwarcia na 0.0.0.0/0 z proxy jest zbędne i nadal naraża instancję na niepotrzebne ryzyko, ponieważ proxy nie chroni przed atakami skierowanymi bezpośrednio na publiczny IP.

Dodatkowe źródła:

*   **Cloud SQL Auth proxy:** https://cloud.google.com/sql/docs/mysql/connect-external-app
*   **Authorized Networks:** https://cloud.google.com/sql/docs/mysql/configure-public-ip

Sugerowana odpowiedź: C

Odpowiedź C jest zdecydowanie najbardziej poprawna i bezpieczna w tej sytuacji. Pytanie dotyczy połączenia aplikacji z dynamicznymi publicznymi adresami IP do instancji Cloud SQL, przy jednoczesnym zapewnieniu bezpieczeństwa. Użycie Cloud SQL Auth proxy eliminuje potrzebę zarządzania autoryzowanymi sieciami i otwierania instancji na bezpośrednie połączenia z publicznych IP. Proxy tworzy zaszyfrowany tunel, uwierzytelnia połączenie za pomocą IAM i szyfruje dane, co zapewnia wysoki poziom bezpieczeństwa.

Komentarze użytkowników w większości potwierdzają tę odpowiedź. Użytkownik raaad trafnie zauważa, że proxy zapewnia bezpieczny dostęp bez konieczności zarządzania adresami IP. Pukapuiz i Matt_108 również podkreślają zalety użycia Cloud SQL Auth proxy. Scaenruy, autor pytania, również wybrał odpowiedź C.

Użytkownik FreshMind podnosi ważny punkt dotyczący frazy "You want to use Cloud SQL public IP". Należy jednak pamiętać, że użycie publicznego IP jest wymagane do działania proxy – proxy łączy się z publicznym IP instancji Cloud SQL, ale samo połączenie z aplikacji do proxy jest zabezpieczone. Nie oznacza to otwierania instancji na bezpośrednie połączenia z publicznych IP.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A i D: Otwarcie na 0.0.0.0/0 jest bardzo niebezpieczne i naraża instancję na ataki. Nawet w połączeniu z proxy, pozostawia otwartą możliwość ataków bezpośrednio na publiczny IP. Użytkownik JyoGCP słusznie zauważa, że 0.0.0.0/0 jest niezalecane ze względów bezpieczeństwa.
  • B: Regularne aktualizowanie listy autoryzowanych sieci jest uciążliwe, podatne na błędy i nie rozwiązuje problemu dynamicznych adresów IP.

Podsumowując, Cloud SQL Auth proxy jest najlepszym rozwiązaniem dla dynamicznych adresów IP, zapewniając bezpieczeństwo i elastyczność.
