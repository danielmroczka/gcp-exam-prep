You have a table that contains millions of rows of sales data, partitioned by date. Various applications and users query this data many times a minute. The query requires aggregating values by using AVG, MAX, and SUM, and does not require joining to other tables. The required aggregations are only computed over the past year of data, though you need to retain full historical data in the base tables. You want to ensure that the query results always include the latest data from the tables, while also reducing computation cost, maintenance overhead, and duration. What should you do?

  • A. Create a materialized view to aggregate the base table data. Include a filter clause to specify the last one year of partitions. Most Voted
  • B. Create a materialized view to aggregate the base table data. Configure a partition expiration on the base table to retain only the last one year of partitions.
  • C. Create a view to aggregate the base table data. Include a filter clause to specify the last year of partitions.
  • D. Create a new table that aggregates the base table data. Include a filter clause to specify the last year of partitions. Set up a scheduled query to recreate the new table every hour.

Odpowiedź: A.

Materializowane widoki są idealne do pre-agregacji danych, które są często odczytywane, a rzadko modyfikowane. W tym przypadku, agregacje (AVG, MAX, SUM) są wykonywane na danych sprzed roku, co oznacza, że większość danych w tabeli bazowej nie musi być przetwarzana za każdym razem. Użycie klauzuli `WHERE` w definicji materializowanego widoku ogranicza przetwarzanie do ostatnich roku partycji, co znacznie redukuje koszty obliczeniowe i czas wykonywania zapytań. Materializowane widoki są automatycznie odświeżane (można skonfigurować częstotliwość), zapewniając dostęp do najnowszych danych bez konieczności ręcznego zarządzania. Dodatkowo, pełna historia danych jest zachowywana w tabelach bazowych, co spełnia wymagania biznesowe.

A. Jest to najlepsze rozwiązanie, ponieważ wykorzystuje pre-agregację i filtrowanie partycji dla optymalizacji.
B. Konfigurowanie wygasania partycji w tabeli bazowej spowodowałoby utratę historycznych danych, co jest niedopuszczalne.
C. Widoki nie przechowują fizycznie danych, więc za każdym razem, gdy są odpytywane, wymagają pełnego przetworzenia danych, co jest nieefektywne.
D. Tworzenie nowej tabeli i jej regularne odświeżanie jest bardziej kosztowne i skomplikowane w utrzymaniu niż użycie materializowanego widoku.

Sugerowana odpowiedź: A

Odpowiedź A jest prawidłowa, ponieważ najlepiej spełnia wszystkie wymagania zadania. Kluczowe jest tutaj pre-agregowanie danych za pomocą materializowanego widoku, co znacznie redukuje koszty obliczeniowe i czas wykonywania zapytań, szczególnie przy częstych zapytaniach o zagregowane dane. Dodatkowo, użycie klauzuli filtrującej partycje (ostatni rok) ogranicza zakres przetwarzanych danych, optymalizując wydajność. Materializowane widoki są automatycznie odświeżane, zapewniając dostęp do aktualnych danych, a jednocześnie pełna historia danych jest zachowywana w tabelach bazowych, co jest kluczowe zgodnie z wymaganiami zadania.

Dyskusja użytkowników w większości potwierdza tę odpowiedź. Użytkownicy, w tym ci z nowszymi komentarzami i większą liczbą głosów, wskazują na korzyści płynące z użycia materializowanych widoków do pre-agregacji danych i filtrowania partycji. Użytkownik raaad trafnie podkreśla, że materializowane widoki są idealne do tego typu zapytań.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Konfigurowanie wygasania partycji w tabeli bazowej narusza wymóg zachowania pełnej historii danych.
  • C: Widoki nie przechowują fizycznie danych, więc każde zapytanie wymagałoby pełnego przetworzenia danych, co jest nieefektywne.
  • D: Tworzenie nowej tabeli i jej regularne odświeżanie jest bardziej kosztowne i skomplikowane w utrzymaniu niż użycie materializowanego widoku.

Użytkownik et2137 początkowo sugerował odpowiedź C, ale został skorygowany przez innych użytkowników, którzy zwrócili uwagę na fakt, że materializowane widoki zapewniają dostęp do najnowszych danych, a widoki nie przechowują danych fizycznie.
