You are implementing workflow pipeline scheduling using open source-based tools and Google Kubernetes Engine (GKE). You want to use a Google managed service to simplify and automate the task. You also want to accommodate Shared VPC networking considerations. What should you do?

  • A. Use Dataflow for your workflow pipelines. Use Cloud Run triggers for scheduling.
  • B. Use Dataflow for your workflow pipelines. Use shell scripts to schedule workflows.
  • C. Use Cloud Composer in a Shared VPC configuration. Place the Cloud Composer resources in the host project.
  • D. Use Cloud Composer in a Shared VPC configuration. Place the Cloud Composer resources in the service project. Most Voted

Odpowiedź: D.

Cloud Composer to w pełni zarządzana usługa orkiestracji przepływów pracy oparta na Apache Airflow, która idealnie nadaje się do automatyzacji i planowania zadań w GKE. Konfiguracja Shared VPC pozwala na bezpieczne i efektywne współdzielenie sieci pomiędzy projektami, co jest kluczowe w wielu organizacjach. Umieszczenie zasobów Cloud Composer w projekcie serwisowym (service project) jest zalecane, ponieważ pozwala na centralne zarządzanie zasobami i uprawnieniami, jednocześnie umożliwiając dostęp do zasobów w projekcie hosta (host project) poprzez Shared VPC. To rozwiązanie zapewnia elastyczność i kontrolę nad infrastrukturą.

A. Dataflow jest świetny do przetwarzania strumieniowego i wsadowego, ale nie jest tak elastyczny w planowaniu jak Cloud Composer, a Cloud Run triggers nie są idealne do złożonych zależności.
B. Używanie shell scriptów do planowania jest mało skalowalne, trudne w utrzymaniu i nie oferuje wbudowanych funkcji monitorowania i zarządzania błędami.
C. Umieszczenie zasobów Cloud Composer w projekcie hosta może prowadzić do problemów z uprawnieniami i zarządzaniem zasobami, a także utrudniać izolację środowisk.

Dodatkowe źródła:

*   **Cloud Composer Documentation:** https://cloud.google.com/composer/docs
*   **Shared VPC Documentation:** https://cloud.google.com/vpc/docs/shared-vpc
*   **Apache Airflow:** https://airflow.apache.org/

Sugerowana odpowiedź: D

Poprawną odpowiedzią jest D: Użyj Cloud Composer w konfiguracji Shared VPC. Umieść zasoby Cloud Composer w projekcie serwisowym.

Cloud Composer to w pełni zarządzana usługa orkiestracji przepływów pracy oparta na Apache Airflow, idealna do automatyzacji i planowania zadań w GKE. Konfiguracja Shared VPC umożliwia współdzielenie sieci pomiędzy projektami, co jest kluczowe w wielu organizacjach. Umieszczenie zasobów Cloud Composer w projekcie serwisowym jest zalecane, ponieważ zapewnia centralne zarządzanie zasobami i uprawnieniami, a także umożliwia dostęp do zasobów w projekcie hosta poprzez Shared VPC. Zapewnia to lepszą izolację i kontrolę nad infrastrukturą.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość głosów (79%) popiera opcję D, a użytkownicy, jak AWSandeep, zellck i ducc, powołują się na oficjalną dokumentację Google Cloud, która wyraźnie wskazuje, że środowisko Cloud Composer powinno znajdować się w projekcie serwisowym w konfiguracji Shared VPC.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Dataflow jest bardziej odpowiedni do przetwarzania strumieniowego i wsadowego, a Cloud Run triggers nie są wystarczająco elastyczne do złożonych planowań.
  • B: Używanie skryptów powłoki do planowania jest mało skalowalne, trudne w utrzymaniu i brakuje wbudowanych funkcji monitorowania i obsługi błędów.
  • C: Umieszczenie zasobów Cloud Composer w projekcie hosta może prowadzić do problemów z uprawnieniami, zarządzaniem zasobami i utrudniać izolację środowisk. Użytkownicy, jak loki82, słusznie zauważają, że umieszczanie usług w projekcie hosta jest złą praktyką, jeśli usługa obsługuje Shared VPC.

Komentarze użytkowników, którzy początkowo wybierali odpowiedź C, często zmieniali zdanie po zapoznaniu się z dokumentacją, co dodatkowo potwierdza poprawność odpowiedzi D.
