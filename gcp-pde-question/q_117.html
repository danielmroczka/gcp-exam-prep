You are designing a data processing pipeline. The pipeline must be able to scale automatically as load increases. Messages must be processed at least once and must be ordered within windows of 1 hour. How should you design the solution?

  • A. Use Apache Kafka for message ingestion and use Cloud Dataproc for streaming analysis.
  • B. Use Apache Kafka for message ingestion and use Cloud Dataflow for streaming analysis.
  • C. Use Cloud Pub/Sub for message ingestion and Cloud Dataproc for streaming analysis.
  • D. Use Cloud Pub/Sub for message ingestion and Cloud Dataflow for streaming analysis. Most Voted

Odpowiedź: D.

Cloud Pub/Sub jest idealnym rozwiązaniem do ingestii wiadomości, gdy wymagana jest skalowalność i niezawodność. Zapewnia on mechanizm "at least once delivery", gwarantując, że każda wiadomość zostanie przetworzona przynajmniej raz. Cloud Dataflow, z kolei, jest potężnym silnikiem do przetwarzania strumieniowego, który umożliwia przetwarzanie danych w oknach czasowych (w tym przypadku 1-godzinnych) i zachowanie kolejności w ramach tych okien. Połączenie Pub/Sub i Dataflow pozwala na budowę skalowalnego i niezawodnego potoku przetwarzania danych, spełniającego wszystkie wymagania zadania. Dataflow oferuje wbudowane mechanizmy do obsługi kolejności i okien czasowych, co upraszcza implementację.

A. Cloud Dataproc nie jest optymalny do przetwarzania strumieniowego w czasie rzeczywistym, lepiej sprawdza się w przetwarzaniu wsadowym.
B. Chociaż Kafka jest dobrym rozwiązaniem do ingestii, Cloud Dataflow jest bardziej odpowiedni do przetwarzania strumieniowego z gwarancją kolejności w oknach czasowych niż Dataproc.
C. Cloud Pub/Sub jest lepszy od Kafka w kontekście zarządzanej usługi Google Cloud, oferującej łatwiejszą skalowalność i integrację z innymi usługami GCP.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Use Cloud Pub/Sub for message ingestion and Cloud Dataflow for streaming analysis.

Dlaczego?

Wymagania: 1. Scale automatically as load increases: Cloud Pub/Sub (ingestion) i Cloud Dataflow (processing) są usługami w pełni serverless i oferują natywne autoskalowanie bez zarządzania klastrami. 2. Processed at least once: Pub/Sub gwarantuje dostarczenie "at-least-once". 3. Ordered within windows of 1 hour: Cloud Dataflow (Apache Beam) doskonale radzi sobie z porządkowaniem danych w oknach czasowych (Event Time windowing) oraz obsługą spóźnionych danych (late data) za pomocą znaków wodnych (watermarks).

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i B. Apache Kafka: Kafka wymaga zarządzania klastrem (brokerami, Zookeeperem), co utrudnia "skalowanie automatyczne" bez dużego narzutu operacyjnego (chyba że użyjesz Managed Kafka, ale Pub/Sub jest natywnym wyborem serverless w GCP).
  • C. Cloud Dataproc: Dataproc (Spark Streaming) nie skaluje się tak płynnie i szybko "automatycznie" jak Dataflow. Ponadto Dataproc operuje na mikro-batchach, co utrudnia precyzyjne zarządzanie oknami czasowymi i porządkiem w porównaniu do modelu Beam w Dataflow.
