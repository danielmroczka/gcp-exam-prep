Your organization's data assets are stored in BigQuery, Pub/Sub, and a PostgreSQL instance running on Compute Engine. Because there are multiple domains and diverse teams using the data, teams in your organization are unable to discover existing data assets. You need to design a solution to improve data discoverability while keeping development and configuration efforts to a minimum. What should you do?

  • A. Use Data Catalog to automatically catalog BigQuery datasets. Use Data Catalog APIs to manually catalog Pub/Sub topics and PostgreSQL tables.
  • B. Use Data Catalog to automatically catalog BigQuery datasets and Pub/Sub topics. Use Data Catalog APIs to manually catalog PostgreSQL tables. Most Voted
  • C. Use Data Catalog to automatically catalog BigQuery datasets and Pub/Sub topics. Use custom connectors to manually catalog PostgreSQL tables.
  • D. Use customer connectors to manually catalog BigQuery datasets, Pub/Sub topics, and PostgreSQL tables.

Odpowiedź: B.

Data Catalog oferuje automatyczne katalogowanie zasobów BigQuery i tematów Pub/Sub, co znacząco redukuje nakład pracy w porównaniu do ręcznego katalogowania. Wykorzystanie Data Catalog API do ręcznego katalogowania tabel PostgreSQL jest rozsądne, ponieważ nie ma natywnego, automatycznego wsparcia dla tej bazy danych. To podejście minimalizuje wysiłek konfiguracyjny, jednocześnie zapewniając centralne miejsce do odkrywania danych.  Dzięki temu zespoły będą mogły łatwo znaleźć i zrozumieć dostępne zasoby danych, co poprawi efektywność i współpracę.  Data Catalog pozwala na dodawanie metadanych, tagów i opisów, co ułatwia wyszukiwanie i zrozumienie danych.  Wykorzystanie automatyzacji tam, gdzie to możliwe, jest kluczowe dla utrzymania aktualności katalogu danych.

A. Ręczne katalogowanie tematów Pub/Sub jest bardziej czasochłonne niż wykorzystanie automatycznego katalogowania Data Catalog.
C. Custom connectors wymagają więcej pracy programistycznej i utrzymaniowej niż wykorzystanie Data Catalog API.
D. Ręczne katalogowanie wszystkich zasobów jest nieefektywne i trudne do utrzymania w aktualności, szczególnie w dynamicznym środowisku.

Dodatkowe źródła:

*   **Google Cloud Data Catalog:** https://cloud.google.com/data-catalog
*   **Data Catalog API:** https://cloud.google.com/data-catalog/docs/reference/rest
*   **Custom Connectors:** https://cloud.google.com/data-catalog/docs/how-to/create-custom-connector

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Najbardziej prawdopodobną poprawną odpowiedzią jest C. Dyskusja użytkowników, szczególnie te nowsze komentarze (af17139, Abizi, baimus, 987af6b), wskazują, że dla PostgreSQL dostępny jest community connector, który powinien być preferowany nad ręcznym katalogowaniem za pomocą Data Catalog API. Komentarze te odwołują się do dokumentacji Google Cloud, która potwierdza istnienie connectorów dla popularnych baz danych on-premise, a PostgreSQL jest jedną z nich. Użycie connectorów minimalizuje nakład pracy programistycznej i utrzymaniowej w porównaniu do tworzenia własnych skryptów do integracji z API.

Odpowiedź B, choć początkowo popularna, jest mniej optymalna. Ręczne katalogowanie PostgreSQL za pomocą API jest możliwe, ale nieefektywne, gdy dostępne jest dedykowane rozwiązanie w postaci connectorów. Komentarze wskazują, że Data Catalog API powinno być używane tylko wtedy, gdy nie ma dostępnego connector’a.

Odpowiedź A jest nieprawidłowa, ponieważ sugeruje ręczne katalogowanie tematów Pub/Sub, co jest niepotrzebne, ponieważ Data Catalog automatycznie je kataloguje.

Odpowiedź D jest niepraktyczna, ponieważ ręczne katalogowanie wszystkich zasobów jest czasochłonne i trudne do utrzymania w aktualności.

Podsumowując, wykorzystanie automatycznego katalogowania tam, gdzie to możliwe (BigQuery, Pub/Sub) oraz dedykowanych connectorów dla pozostałych źródeł (PostgreSQL) jest najbardziej efektywnym i zgodnym z zasadą minimalizacji nakładu pracy rozwiązaniem.
