You work for a car manufacturer and have set up a data pipeline using Google Cloud Pub/Sub to capture anomalous sensor events. You are using a push subscription in Cloud Pub/Sub that calls a custom HTTPS endpoint that you have created to take action of these anomalous events as they occur. Your custom
HTTPS endpoint keeps getting an inordinate amount of duplicate messages. What is the most likely cause of these duplicate messages?

  • A. The message body for the sensor event is too large.
  • B. Your custom endpoint has an out-of-date SSL certificate.
  • C. The Cloud Pub/Sub topic has too many messages published to it.
  • D. Your custom endpoint is not acknowledging messages within the acknowledgement deadline. Most Voted

Odpowiedź: D.

Twoje niestandardowe punkt końcowe HTTPS nie potwierdza otrzymania wiadomości w ramach terminu potwierdzenia. Cloud Pub/Sub domyślnie zakłada, że wiadomość nie została pomyślnie przetworzona, jeśli nie otrzyma potwierdzenia w określonym czasie. W takim przypadku Pub/Sub ponownie wysyła wiadomość, co prowadzi do duplikatów. Optymalizacja czasu przetwarzania i zapewnienie szybkiego potwierdzania wiadomości jest kluczowe.

A. Wielkość wiadomości może wpływać na wydajność, ale nie jest bezpośrednią przyczyną duplikatów w tym scenariuszu.
B. Nieaktualny certyfikat SSL spowodowałby błędy połączenia, a nie duplikaty wiadomości, jeśli połączenie w ogóle by się nawiązało.
C. Duża liczba publikacji w temacie Pub/Sub nie generuje duplikatów w subskrypcji, chyba że problem leży w samym przetwarzaniu subskrybentów.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Your custom endpoint is not acknowledging messages within the acknowledgement deadline.

Dlaczego?

Google Cloud Pub/Sub gwarantuje dostarczenie wiadomości "at-least-once" (co najmniej raz). Mechanizm działania:

  1. Pub/Sub wysyła wiadomość do subskrybenta (Twój endpoint HTTPS).
  2. Pub/Sub oczekuje na kod sukcesu (HTTP 200/204) w określonym czasie (ack deadline, domyślnie 10 sekund, konfigurowalne do 600s).
  3. Jeśli Twój endpoint mieli dane zbyt długo i nie odeśle odpowiedzi przed upływem deadline'u, Pub/Sub uznaje, że dostarczenie się nie powiodło.
  4. Pub/Sub wysyła wiadomość ponownie (redelivery).

Skutek: Twój endpoint w końcu przetwarza pierwszą wiadomość (po czasie), ale w międzyczasie dostaje ją znowu. To klasyczna przyczyna duplikatów przy wolnym przetwarzaniu.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Message body too large: Pub/Sub ma limity (10MB), ale przekroczenie limitu spowodowałoby błąd przy publikacji, a nie duplikaty przy odbiorze.
  • B. SSL certificate: Błąd SSL spowodowałby brak dostarczenia (handshake failure), a nie wielokrotne dostarczenie (chyba że retry, ale wtedy żadna wiadomość by nie przeszła).
  • C. Too many messages: Pub/Sub skaluje się automatycznie. Wysoki wolumen nie powoduje duplikatów sam w sobie, o ile system nadąża z ich konsumpcją (ACK).
