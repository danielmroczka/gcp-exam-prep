You are designing a system that requires an ACID-compliant database. You must ensure that the system requires minimal human intervention in case of a failure.
What should you do?

  • A. Configure a Cloud SQL for MySQL instance with point-in-time recovery enabled.
  • B. Configure a Cloud SQL for PostgreSQL instance with high availability enabled. Most Voted
  • C. Configure a Bigtable instance with more than one cluster.
  • D. Configure a BigQuery table with a multi-region configuration.

Odpowiedź: B.

Konfiguracja instancji Cloud SQL dla PostgreSQL z włączoną wysoką dostępnością (High Availability - HA) zapewnia automatyczne przełączanie w przypadku awarii, minimalizując potrzebę interwencji człowieka. HA w PostgreSQL replikuje dane do instancji zapasowej, która automatycznie przejmuje rolę głównej w przypadku awarii instancji głównej. To zapewnia ciągłość działania i spełnia wymagania dotyczące ACID (Atomicity, Consistency, Isolation, Durability). PostgreSQL jest również relacyjną bazą danych, która natywnie obsługuje transakcje ACID.  Wysoka dostępność w Cloud SQL dla PostgreSQL jest zaprojektowana tak, aby automatycznie radzić sobie z awariami, co jest kluczowe dla minimalizacji interwencji człowieka.

A. Point-in-time recovery w Cloud SQL dla MySQL pozwala na przywrócenie danych do określonego momentu w czasie, ale nie zapewnia automatycznego przełączania w przypadku awarii.
C. Bigtable jest bazą danych NoSQL, która nie gwarantuje pełnej zgodności z ACID, a konfiguracja wielu klastrów skupia się na skalowalności i dostępności geograficznej, a nie na automatycznym przełączaniu w przypadku awarii.
D. BigQuery jest hurtownią danych, a nie relacyjną bazą danych, i nie zapewnia zgodności z ACID ani wysokiej dostępności w taki sam sposób jak Cloud SQL.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Konfiguracja instancji Cloud SQL dla PostgreSQL z włączoną wysoką dostępnością (High Availability - HA) jest prawidłową odpowiedzią. Kluczowym wymaganiem pytania jest minimalna interwencja człowieka w przypadku awarii, a HA w PostgreSQL zapewnia automatyczne przełączanie na instancję zapasową bez potrzeby ręcznej interwencji. Jak trafnie zauważył użytkownik NicolasN, w przypadku awarii instancji głównej, Cloud SQL automatycznie przełącza się na instancję zapasową, co jest kluczowe dla spełnienia tego wymagania.

PostgreSQL jest relacyjną bazą danych, która natywnie obsługuje transakcje ACID (Atomicity, Consistency, Isolation, Durability), co również jest wymagane w pytaniu. Użytkownik MaxNRG i wielu innych potwierdzają, że PostgreSQL zapewnia pełną zgodność z ACID, w przeciwieństwie do Bigtable.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Point-in-time recovery (PITR) w Cloud SQL dla MySQL pozwala na przywrócenie danych do określonego momentu w czasie, ale wymaga ręcznego zainicjowania procesu przywracania, co narusza wymóg minimalnej interwencji człowieka.
  • C: Bigtable jest bazą danych NoSQL i nie gwarantuje pełnej zgodności z ACID. Konfiguracja wielu klastrów skupia się na skalowalności i dostępności geograficznej, a nie na automatycznym przełączaniu w przypadku awarii.
  • D: BigQuery jest hurtownią danych, a nie relacyjną bazą danych, i nie zapewnia zgodności z ACID ani wysokiej dostępności w taki sam sposób jak Cloud SQL. Dodatkowo, jak zauważył NicolasN, konfiguracja lokalizacji w BigQuery dotyczy zestawu danych (Dataset), a nie pojedynczej tabeli.

Użytkownik [Removed] błędnie wskazuje BigQuery jako rozwiązanie zgodne z ACID. Chociaż BigQuery posiada pewne mechanizmy zapewniające spójność danych, nie oferuje pełnej zgodności z ACID w tradycyjnym sensie relacyjnych baz danych.
