You are building a report-only data warehouse where the data is streamed into BigQuery via the streaming API. Following Google's best practices, you have both a staging and a production table for the data. How should you design your data loading to ensure that there is only one master dataset without affecting performance on either the ingestion or reporting pieces?

  • A. Have a staging table that is an append-only model, and then update the production table every three hours with the changes written to staging.
  • B. Have a staging table that is an append-only model, and then update the production table every ninety minutes with the changes written to staging.
  • C. Have a staging table that moves the staged data over to the production table and deletes the contents of the staging table every three hours. Most Voted
  • D. Have a staging table that moves the staged data over to the production table and deletes the contents of the staging table every thirty minutes.

Odpowiedź: C.

Wykorzystanie staging table i regularne przenoszenie danych do tabeli produkcyjnej z jednoczesnym usuwaniem zawartości staging table jest najlepszą praktyką w przypadku budowy data warehouse w BigQuery, szczególnie przy strumieniowaniu danych. Pozwala to na utrzymanie jednej, spójnej wersji danych (master dataset) bez wpływu na wydajność zarówno procesu ingestii, jak i raportowania. Usuwanie danych ze staging table po przeniesieniu do produkcji zapobiega niepotrzebnemu wzrostowi kosztów przechowywania i utrzymuje ją w optymalnym stanie do kolejnych partii danych. Regularne (co 3 godziny) przenoszenie danych zapewnia aktualność danych w tabeli produkcyjnej, jednocześnie minimalizując obciążenie systemu. Taka strategia pozwala na efektywne zarządzanie danymi i utrzymanie wysokiej wydajności.

A. Aktualizacja co 3 godziny jest zbyt rzadka, aby zapewnić aktualność danych w raportach i może prowadzić do opóźnień w analizie.
B. Aktualizacja co 90 minut jest lepsza niż co 3 godziny, ale nadal może być niewystarczająca dla niektórych zastosowań i generować większe obciążenie niż aktualizacja co 3 godziny z usuwaniem danych ze staging.
D. Aktualizacja co 30 minut może generować zbyt duże obciążenie dla systemu, szczególnie przy dużych wolumenach danych, i nie jest optymalna pod względem kosztów.

Sugerowana odpowiedź: C

Najlepszym rozwiązaniem jest użycie tabeli stagingowej i regularne przenoszenie danych do tabeli produkcyjnej z jednoczesnym usuwaniem zawartości tabeli stagingowej co 3 godziny. Dyskusja użytkowników, a w szczególności linki do oficjalnej dokumentacji Google Cloud (podane przez użytkowników NicolasN, zellck, Atnafu i John_Pongthorn) jednoznacznie wskazują na tę strategię jako zgodną z najlepszymi praktykami ETL w BigQuery, szczególnie przy wykorzystaniu streaming API.

Kluczowe argumenty przemawiające za odpowiedzią C:

  • Zgodność z dokumentacją Google: Dokumentacja Google wyraźnie zaleca usuwanie danych ze staging table po przeniesieniu do produkcji, aby uniknąć problemów z dostępnością danych i kosztami przechowywania.
  • Uwzględnienie opóźnień w streaming API: Streaming API ma bufor, który może opóźnić dostępność danych do 60 minut lub dłużej. Przenoszenie danych co 3 godziny pozwala na uwzględnienie tego opóźnienia i zapewnia, że dane są dostępne do usunięcia ze staging table.
  • Utrzymanie jednej wersji danych: Regularne przenoszenie danych do tabeli produkcyjnej zapewnia, że istnieje tylko jedna, spójna wersja danych (master dataset).
  • Optymalizacja kosztów: Usuwanie danych ze staging table po przeniesieniu do produkcji minimalizuje koszty przechowywania.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A i B: Aktualizacja tabeli produkcyjnej co 3 godziny lub 90 minut, przy zachowaniu append-only staging table, prowadzi do niekontrolowanego wzrostu rozmiaru staging table i potencjalnych problemów z wydajnością.
  • D: Aktualizacja co 30 minut może generować zbyt duże obciążenie dla systemu, szczególnie przy dużych wolumenach danych, i nie jest optymalna pod względem kosztów. Dodatkowo, jak wskazano w dyskusji, częste operacje DELETE mogą być problematyczne ze względu na opóźnienia w streaming API.

Użytkownik MaxNRG w swoich obszernych komentarzach również argumentuje za odpowiedzią A, jednak w kontekście całej dyskusji i oficjalnej dokumentacji, odpowiedź C wydaje się bardziej precyzyjna i zgodna z najlepszymi praktykami.
