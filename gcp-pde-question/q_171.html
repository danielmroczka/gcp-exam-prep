You work for a large real estate firm and are preparing 6 TB of home sales data to be used for machine learning. You will use SQL to transform the data and use
BigQuery ML to create a machine learning model. You plan to use the model for predictions against a raw dataset that has not been transformed. How should you set up your workflow in order to prevent skew at prediction time?

  • A. When creating your model, use BigQuery's TRANSFORM clause to define preprocessing steps. At prediction time, use BigQuery's ML.EVALUATE clause without specifying any transformations on the raw input data. Most Voted
  • B. When creating your model, use BigQuery's TRANSFORM clause to define preprocessing steps. Before requesting predictions, use a saved query to transform your raw input data, and then use ML.EVALUATE.
  • C. Use a BigQuery view to define your preprocessing logic. When creating your model, use the view as your model training data. At prediction time, use BigQuery's ML.EVALUATE clause without specifying any transformations on the raw input data.
  • D. Preprocess all data using Dataflow. At prediction time, use BigQuery's ML.EVALUATE clause without specifying any further transformations on the input data.

Odpowiedź: A.

Wyjaśnienie: Aby uniknąć przesunięcia (skew) w czasie predykcji, kluczowe jest zastosowanie tych samych transformacji zarówno podczas trenowania modelu, jak i podczas generowania predykcji. BigQuery's `TRANSFORM` clause pozwala zdefiniować kroki preprocessingu wewnątrz modelu. Użycie `ML.EVALUATE` bez dodatkowych transformacji na surowych danych wejściowych w czasie predykcji zapewnia, że dane są przetwarzane dokładnie tak samo, jak podczas trenowania, co minimalizuje ryzyko rozbieżności i poprawia dokładność predykcji. Dzięki temu model operuje na danych w spójny sposób, niezależnie od etapu procesu.

A. Jest to gorsze, ponieważ `ML.EVALUATE` służy do oceny modelu, a nie do generowania predykcji.
B. Jest to gorsze, ponieważ użycie oddzielnego zapisanego zapytania do transformacji danych wejściowych wprowadza potencjalne rozbieżności między transformacjami używanymi podczas trenowania a transformacjami używanymi podczas predykcji.
C. Jest to gorsze, ponieważ użycie widoku BigQuery do preprocessingu danych treningowych nie gwarantuje, że te same transformacje zostaną zastosowane do surowych danych wejściowych w czasie predykcji.
D. Jest to gorsze, ponieważ Dataflow jest bardziej złożonym rozwiązaniem niż potrzebne i nie zapewnia automatycznego spójnego stosowania transformacji w czasie predykcji.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Poprawną odpowiedzią jest A. Kluczowym aspektem zapobiegania przesunięciu (skew) w modelach uczenia maszynowego jest zapewnienie spójnego stosowania tych samych kroków preprocessingu zarówno do danych treningowych, jak i do danych używanych do predykcji. BigQuery's TRANSFORM clause pozwala na zdefiniowanie tych kroków preprocessingu wewnątrz modelu. Użycie ML.EVALUATE (a w konsekwencji również funkcji predykcyjnych) bez dodatkowych transformacji na surowych danych wejściowych gwarantuje, że dane są przetwarzane dokładnie tak samo, jak podczas treningu.

Dyskusja użytkowników potwierdza tę odpowiedź. Większość głosów (90%) popiera odpowiedź A. Użytkownicy, tacy jak AWSandeep, zellck, Prudvi3266, Kvk117, jkhong, TNT87 i ducc, wskazują na dokumentację BigQuery ML i podkreślają, że TRANSFORM clause automatycznie stosuje preprocessing podczas fazy predykcji i oceny.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Użycie oddzielnego zapytania do transformacji danych wejściowych wprowadza ryzyko niespójności między transformacjami używanymi podczas treningu a transformacjami używanymi podczas predykcji.
  • C: Użycie widoku BigQuery do preprocessingu danych treningowych nie gwarantuje, że te same transformacje zostaną automatycznie zastosowane do surowych danych wejściowych w czasie predykcji. Wymagałoby to dodatkowej logiki, aby zapewnić spójność.
  • D: Dataflow jest bardziej złożonym rozwiązaniem niż potrzebne w tym przypadku i nie zapewnia automatycznego spójnego stosowania transformacji w czasie predykcji.

Użytkownik f74ca0c sugeruje odpowiedź C, ale jego argumentacja jest błędna. Chociaż widoki BigQuery mogą pomóc w enkapsulacji logiki preprocessingu, nie rozwiązują problemu automatycznego stosowania tych samych transformacji podczas predykcji bez dodatkowej konfiguracji.

Użytkownik Lenifia również błędnie wskazuje odpowiedź B, argumentując, że użycie zapisanego zapytania zapewnia spójność. W rzeczywistości wprowadza to dodatkowy punkt potencjalnych błędów.
