You have terabytes of customer behavioral data streaming from Google Analytics into BigQuery daily. Your customers’ information, such as their preferences, is hosted on a Cloud SQL for MySQL database. Your CRM database is hosted on a Cloud SQL for PostgreSQL instance. The marketing team wants to use your customers’ information from the two databases and the customer behavioral data to create marketing campaigns for yearly active customers. You need to ensure that the marketing team can run the campaigns over 100 times a day on typical days and up to 300 during sales. At the same time, you want to keep the load on the Cloud SQL databases to a minimum. What should you do?

  • A. Create BigQuery connections to both Cloud SQL databases. Use BigQuery federated queries on the two databases and the Google Analytics data on BigQuery to run these queries.
  • B. Create a job on Apache Spark with Dataproc Serverless to query both Cloud SQL databases and the Google Analytics data on BigQuery for these queries.
  • C. Create streams in Datastream to replicate the required tables from both Cloud SQL databases to BigQuery for these queries. Most Voted
  • D. Create a Dataproc cluster with Trino to establish connections to both Cloud SQL databases and BigQuery, to execute the queries.

Odpowiedź: C.

Replikacja danych z Cloud SQL (MySQL i PostgreSQL) do BigQuery za pomocą Datastream jest najlepszym rozwiązaniem, ponieważ pozwala na odciążenie baz danych Cloud SQL od bezpośrednich zapytań marketingowych. Datastream umożliwia ciągłą replikację danych, co zapewnia, że BigQuery zawiera aktualne informacje o klientach. Marketingowy zespół może następnie uruchamiać złożone zapytania na danych w BigQuery, bez wpływu na wydajność baz danych transakcyjnych. To rozwiązanie jest szczególnie korzystne przy dużej częstotliwości zapytań (100-300 dziennie), ponieważ BigQuery jest zoptymalizowany do analizy dużych zbiorów danych. Dodatkowo, replikacja danych pozwala na historyczną analizę zachowań klientów.

A. BigQuery federated queries są wolniejsze i bardziej obciążające dla baz danych źródłowych niż replikacja danych.
B. Apache Spark z Dataproc Serverless jest bardziej skomplikowany w konfiguracji i zarządzaniu niż Datastream, a także może generować większe koszty.
D. Dataproc z Trino wymaga zarządzania klastrem i jest mniej efektywne kosztowo niż Datastream w tym scenariuszu.

Sugerowana odpowiedź: C

Odpowiedź C, czyli replikacja danych z Cloud SQL do BigQuery za pomocą Datastream, jest zdecydowanie najlepszym rozwiązaniem w tym scenariuszu. Kluczowym wymaganiem jest minimalizacja obciążenia baz danych Cloud SQL, przy jednoczesnej obsłudze dużej liczby zapytań (100-300 dziennie). Datastream umożliwia ciągłą replikację danych, co oznacza, że marketingowy zespół może wykonywać zapytania bezpośrednio na BigQuery, bez wpływu na wydajność baz danych transakcyjnych. BigQuery jest zoptymalizowany do analizy dużych zbiorów danych i obsługi wysokiej częstotliwości zapytań.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość osób, w tym ci z nowszymi komentarzami i większą liczbą głosów, wskazuje na C jako prawidłową odpowiedź. Użytkownik joao_01 trafnie zauważa, że zapytania federacyjne (opcja A) obciążają źródłowe bazy danych, co jest sprzeczne z wymaganiami zadania. Użytkownik Smakyel79 podkreśla, że Datastream jest usługą serverless i łatwą w użyciu, co dodatkowo przemawia za jej wyborem.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Zapytania federacyjne (BigQuery federated queries) są wolniejsze i generują większe obciążenie dla baz danych Cloud SQL.
  • B: Apache Spark z Dataproc Serverless jest bardziej złożony w konfiguracji i zarządzaniu niż Datastream, a także może być droższy.
  • D: Dataproc z Trino wymaga zarządzania klastrem i jest mniej efektywne kosztowo niż Datastream w tym konkretnym scenariuszu.

Podsumowując, Datastream jest najbardziej efektywnym, skalowalnym i opłacalnym rozwiązaniem, które spełnia wszystkie wymagania zadania.
