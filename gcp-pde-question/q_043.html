You work for a large fast food restaurant chain with over 400,000 employees. You store employee information in Google BigQuery in a Users table consisting of a FirstName field and a LastName field. A member of IT is building an application and asks you to modify the schema and data in BigQuery so the application can query a FullName field consisting of the value of the FirstName field concatenated with a space, followed by the value of the LastName field for each employee. How can you make that data available while minimizing cost?

  • A. Create a view in BigQuery that concatenates the FirstName and LastName field values to produce the FullName. Most Voted
  • B. Add a new column called FullName to the Users table. Run an UPDATE statement that updates the FullName column for each user with the concatenation of the FirstName and LastName values.
  • C. Create a Google Cloud Dataflow job that queries BigQuery for the entire Users table, concatenates the FirstName value and LastName value for each user, and loads the proper values for FirstName, LastName, and FullName into a new table in BigQuery.
  • D. Use BigQuery to export the data for the table to a CSV file. Create a Google Cloud Dataproc job to process the CSV file and output a new CSV file containing the proper values for FirstName, LastName and FullName. Run a BigQuery load job to load the new CSV file into BigQuery.

Odpowiedź: A.

Wyjaśnienie: Tworzenie widoku (view) w BigQuery jest najbardziej efektywnym kosztowo rozwiązaniem, ponieważ widoki nie przechowują fizycznie danych, a jedynie definicję zapytania. Dane są obliczane na żądanie, gdy zapytanie jest wykonywane na widoku. Minimalizuje to koszty przechowywania i przetwarzania danych, ponieważ nie generuje dodatkowego zestawu danych, który musiałby być przechowywany i aktualizowany. Widok ten zapewni wymagane pole `FullName` bez fizycznego modyfikowania danych w tabeli `Users`.

A. Utworzenie widoku jest najbardziej kosztowo efektywnym rozwiązaniem, ponieważ dane obliczeniowe nie są fizycznie przechowywane.
B. Dodanie nowej kolumny i wykonanie polecenia UPDATE spowoduje fizyczne przechowywanie dodatkowych danych i poniesienie kosztów związanych z ich zapisem i potencjalną aktualizacją, co jest mniej optymalne kosztowo.
C. Stworzenie zadania Dataflow do przetworzenia całej tabeli i załadowania jej do nowej tabeli wymaga znaczących zasobów przetwarzania i przechowywania danych, co jest znacznie droższe niż utworzenie widoku.
D. Eksportowanie danych do CSV, przetwarzanie ich w Dataproc, a następnie ponowne ładowanie do BigQuery jest procesem wieloetapowym, który jest zasobożerny i generuje dodatkowe koszty związane z działaniem Dataproc i przechowywaniem plików pośrednich.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Najbardziej efektywnym kosztowo rozwiązaniem jest utworzenie widoku (view) w BigQuery. Widoki nie przechowują fizycznie danych, a jedynie definicję zapytania, które jest uruchamiane za każdym razem, gdy widok jest odpytywany. Oznacza to, że nie generujemy dodatkowych kosztów związanych z przechowywaniem danych, co jest kluczowe w kontekście pytania. Komentarze użytkowników, szczególnie te nowsze (z ostatnich miesięcy), potwierdzają tę opinię, wskazując, że koszt przechowywania jest minimalny, a obliczenia są wykonywane tylko w momencie zapytania.

Pytanie podkreśla minimalizację kosztów, a widok jest najtańszym rozwiązaniem. Ważne jest również, że pytanie prosi o "udostępnienie" danych, a nie o fizyczną modyfikację schematu. Widok spełnia to wymaganie bez ingerencji w bazową tabelę.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Dodanie nowej kolumny i aktualizacja jej wartości generuje koszty przechowywania i przetwarzania związane z zapisem danych. Dodatkowo, wymagałoby to regularnych aktualizacji przy dodawaniu nowych pracowników.
  • C: Użycie Dataflow jest nadmierne dla tak prostego zadania. Dataflow wiąże się z dużymi kosztami przetwarzania i przechowywania danych.
  • D: Eksport do CSV, przetwarzanie w Dataproc i ponowne ładowanie do BigQuery to skomplikowany i kosztowny proces, obejmujący wiele etapów i generujący dodatkowe koszty związane z transferem danych i działaniem Dataproc.

Użytkownik Ramanaiah słusznie zauważa, że jeśli aplikacja potrzebuje filtrowania po pełnej nazwie, widok jest dobrym rozwiązaniem, ponieważ jest aktualizowany na bieżąco. Użytkownik Mathew106 trafnie podsumowuje, że dodawanie kolumny generuje koszty związane z każdym zapytaniem, podczas gdy widok nie.
