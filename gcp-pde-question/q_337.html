You are building a report-only data warehouse where the data is streamed into BigQuery via the streaming API. Following Google's best practices, you have both a staging and a production table for the data. How should you design your data loading to ensure that there is only one master dataset without affecting performance on either the ingestion or reporting pieces?

  • A. Have a staging table that is an append-only model, and then update the production table every three hours with the changes written to staging.
  • B. Stream data directly into the production table and use authorized views for staging.
  • C. Use BigQuery's built-in table clones to sync data between staging and production in real-time.
  • D. Configure a Dataflow pipeline to read from the staging table's streaming buffer and write to the production table.
Show Suggested Answer
Suggested Answer: A

Wyjaśnienie

Correct Answer: A

A common pattern for large-scale streaming ingestion into BigQuery is to use a staging table to capture the raw stream (append-only) and then periodically (e.g., every 3 hours) perform a "merge" or "update" into the production table. This separates the high-frequency ingestion load from the reporting workload, ensuring that complex queries on the production table aren't affected by constant streaming writes and that the data being reported is consistent and optimized.

Why other options are incorrect:
B. Streaming directly into production can lead to fragmented data and slower query performance for reporting due to the overhead of the streaming buffer.
C. Table clones are for creating lightweight copies of tables, but they don't provide a mechanism for real-time synchronization of streaming data.
D. While Dataflow can process streaming data, reading from a staging table's streaming buffer and writing to another table just to achieve isolation is more complex than a periodic batch update/merge.
