You are deploying 10,000 new Internet of Things devices to collect temperature data in your warehouses globally. You need to process, store and analyze these very large datasets in real time. What should you do?

  • A. Send the data to Google Cloud Datastore and then export to BigQuery.
  • B. Send the data to Google Cloud Pub/Sub, stream Cloud Pub/Sub to Google Cloud Dataflow, and store the data in Google BigQuery. Most Voted
  • C. Send the data to Cloud Storage and then spin up an Apache Hadoop cluster as needed in Google Cloud Dataproc whenever analysis is required.
  • D. Export logs in batch to Google Cloud Storage and then spin up a Google Cloud SQL instance, import the data from Cloud Storage, and run an analysis as needed.

Odpowiedź: B.

Wyjaśnienie: Opcja B jest poprawna, ponieważ Google Cloud Pub/Sub idealnie nadaje się do przyjmowania strumieni danych w czasie rzeczywistym z dużej liczby urządzeń IoT. Następnie Cloud Dataflow pozwala na efektywne przetwarzanie i analizę tych danych w locie. Zapisanie przetworzonych danych w BigQuery umożliwia skalowalne i wydajne przechowywanie oraz dalszą analizę bardzo dużych zbiorów danych. Ten potok danych jest zaprojektowany do obsługi dużego wolumenu i przetwarzania w czasie rzeczywistym.

Dlaczego pozostałe opcje są gorsze/błędne:
A. Google Cloud Datastore nie jest zoptymalizowany do przetwarzania i analizy bardzo dużych zbiorów danych w czasie rzeczywistym, co czyni go gorszym wyborem w porównaniu do BigQuery.
C. Wysyłanie danych do Cloud Storage i uruchamianie klastra Hadoop/Dataproc na żądanie nie zapewnia przetwarzania w czasie rzeczywistym i może wiązać się z większym opóźnieniem i złożonością zarządzania.
D. Eksport logów w trybie wsadowym do Cloud Storage i użycie Cloud SQL jest przeznaczone do przetwarzania wsadowego, a nie do analizy danych w czasie rzeczywistym z urządzeń IoT.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Send the data to Google Cloud Pub/Sub, stream ... to Google Cloud Dataflow, and store ... in Google BigQuery.

Dlaczego?

Jest to kanoniczna architektura przetwarzania strumieniowego (Streaming Analytics) w Google Cloud:

  1. Ingest: Cloud Pub/Sub to globalny, skalowalny bufor dla danych telemetrycznych z IoT (radzi sobie z 10,000 urządzeń).
  2. Process: Cloud Dataflow (Apache Beam) to silnik do przetwarzania danych w czasie rzeczywistym (analiza, okna czasowe, agregacje).
  3. Store & Analyze: BigQuery to hurtownia danych, która pozwala na składowanie i wykonywanie zapytań SQL na petabajtach danych w czasie rzeczywistym.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Datastore -> BigQuery: Datastore nie jest zoptymalizowany do analityki (OLAP) ani do masowego ingestu pod analitykę w taki sposób. Eksport do BQ wprowadzałby opóźnienie (nie "real time").
  • C. Cloud Storage -> Dataproc (Hadoop): To architektura wsadowa (Batch), a nie strumieniowa ("spin up ... as needed"). Nie spełnia wymogu "real time".
  • D. Batch logs -> Cloud SQL: Cloud SQL nie skaluje się tak dobrze do analityki "very large datasets" jak BigQuery i podejście wsadowe narusza wymóg czasu rzeczywistego.
