You plan to deploy Cloud SQL using MySQL. You need to ensure high availability in the event of a zone failure. What should you do?

  • A. Create a Cloud SQL instance in one zone, and create a failover replica in another zone within the same region. Most Voted
  • B. Create a Cloud SQL instance in one zone, and create a read replica in another zone within the same region.
  • C. Create a Cloud SQL instance in one zone, and configure an external read replica in a zone in a different region.
  • D. Create a Cloud SQL instance in a region, and configure automatic backup to a Cloud Storage bucket in the same region.

Odpowiedź: A.

Stworzenie instancji Cloud SQL w jednej strefie i repliki failover w innej strefie w tej samej regionie jest kluczowe dla zapewnienia wysokiej dostępności. Replikacja failover automatycznie przejmuje kontrolę w przypadku awarii strefy, minimalizując przestoje. Cloud SQL automatycznie zarządza replikacją i przełączeniem, co upraszcza konfigurację i utrzymanie.  Replikacja failover jest zaprojektowana specjalnie do zapewnienia ciągłości działania w przypadku awarii, w przeciwieństwie do innych typów replikacji.  Dzięki temu aplikacja może kontynuować działanie z minimalną przerwą.

A. Replikacja read replica nie zapewnia automatycznego przełączania w przypadku awarii strefy.
B. Replikacja read replica jest przeznaczona do odciążenia instancji głównej, a nie do zapewnienia wysokiej dostępności.
C. Replikacja zewnętrzna w innej regionie wprowadza większe opóźnienia i komplikuje zarządzanie.
D. Automatyczne kopie zapasowe chronią przed utratą danych, ale nie zapewniają wysokiej dostępności w przypadku awarii strefy.

*   **Cloud SQL High Availability:** https://cloud.google.com/sql/docs/mysql/high-availability
*   **Cloud SQL Replication:** https://cloud.google.com/sql/docs/mysql/replication

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Początkowo, odpowiedź A była powszechnie akceptowana, ponieważ w przeszłości Cloud SQL używał replik failover do zapewnienia wysokiej dostępności. Jednak, jak wskazują liczne komentarze użytkowników (szczególnie te z ostatnich 1-2 lat), Cloud SQL przeszedł na wykorzystanie regionalnych dysków persystentnych, które replikują dane synchronicznie między dwiema strefami w regionie. Replikacja failover jest obecnie traktowana jako funkcja legacy i Google zaleca migrację do regionalnych dysków persystentnych.

Odpowiedź B, czyli stworzenie instancji Cloud SQL w jednej strefie i repliki odczytu w innej strefie w tej samej regionie, jest obecnie najbardziej zgodna z aktualną dokumentacją i architekturą Cloud SQL. Chociaż replika odczytu sama w sobie nie zapewnia automatycznego przełączania, w połączeniu z regionalnymi dyskami persystentnymi, stanowi podstawę dla wysokiej dostępności. Wiele komentarzy wskazuje na to, że w najnowszych konfiguracjach, repliki odczytu w różnych strefach są wykorzystywane do zapewnienia odporności na awarie stref.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Replikacja failover jest funkcją legacy i nie jest już zalecanym podejściem.
  • C: Replikacja zewnętrzna w innej regionie wprowadza duże opóźnienia i komplikuje zarządzanie. Nie jest to optymalne rozwiązanie dla wysokiej dostępności w przypadku awarii strefy.
  • D: Automatyczne kopie zapasowe chronią przed utratą danych, ale nie zapewniają wysokiej dostępności w przypadku awarii strefy.

Źródła potwierdzające: Komentarze użytkowników wskazują na zmiany w architekturze Cloud SQL i odsyłają do dokumentacji Google Cloud, która potwierdza, że regionalne dyski persystentne i repliki odczytu w różnych strefach są obecnie preferowanym podejściem do zapewnienia wysokiej dostępności.
