The pipeline should write enriched results to BigQuery for analysis. Which job type and transforms should this pipeline use?

Odpowiedź: C

Krótkie wyjaśnienie: Pipeline powinien być typu Streaming job, ponieważ BigQueryIO jest bezpośrednio używany do zapisu danych do BigQuery.  Streaming job jest idealny, ponieważ pobiera dane z Pub/Sub w czasie rzeczywistym, a nie w sposób ciągły, co pozwala na efektywne przetwarzanie danych w czasie rzeczywistym.  Użycie JdbcIO (Database Input/Output) jest mniej efektywne, ponieważ wymaga odczytu danych z bazy danych, co jest mniej wydajne niż bezpośrednie zapisywanie do BigQuery.  Wybór "A" jest nieprawidłowy, ponieważ nie uwzględnia konieczności zapisywania danych do BigQuery.  Wybór "B" jest zbyt niski, ponieważ nie uwzględnia możliwości wykorzystania BigQueryIO.  Wybór "D" jest nieprawidłowy, ponieważ nie jest to odpowiedni typ jobu dla tego zadania.

Pozostałe opcje są gorsze/błędne: A (Batch job) - Batch job nie jest odpowiedni do pobierania danych w czasie rzeczywistym, a B (Streaming job, PubSubIO) - Streaming job nie jest odpowiedni, ponieważ nie ma bezpośredniego połączenia z BigQuery.  C (Streaming job, PubSubIO, BigQueryIO) -  Użycie BigQueryIO bezpośrednio jest najlepszym rozwiązaniem, ponieważ pozwala na zapisywanie danych do BigQuery w czasie rzeczywistym.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Streaming job, PubSubIO, BigQueryIO, side-inputs.

Dlaczego?

1. Job type: Źródłem danych jest Cloud Pub/Sub, co oznacza, że mamy do czynienia ze strumieniem danych (unbounded data). Pipeline musi być typu Streaming. 2. Transforms: - PubSubIO: Standardowy konektor do odczytu z Pub/Sub. - BigQueryIO: Standardowy konektor do zapisu do BigQuery. - Side-inputs: To kluczowy element wzbogacania danych (enrichment). Gdy mamy mały, statyczny (lub rzadko zmienny) zestaw danych referencyjnych, który mieści się w pamięci, Apache Beam pozwala wstrzyknąć go do głównej transformacji (ParDo) jako side-input. Jest to znacznie wydajniejsze niż robienie zewnętrznych wywołań API lub złączeń (joins) dla każdego rekordu w strumieniu.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Batch job: Nieobsługuje w sposób ciągły danych z Pub/Sub.
  • B. JdbcIO, side-outputs: JdbcIO służy do baz relacyjnych (jak Cloud SQL), a tu dane referencyjne są w BigQuery. Side-outputs służą do rozdzielania strumienia (np. błędy do osobnej kolekcji), a nie do wzbogacania.
  • D. side-outputs: Błędna technika wzbogacania (patrz wyżej).
