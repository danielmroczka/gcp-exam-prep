You are developing an Apache Beam pipeline to extract data from a Cloud SQL instance by using JdbcIO. You have two projects running in Google Cloud. The pipeline will be deployed and executed on Dataflow in Project A. The Cloud SQL. instance is running in Project B and does not have a public IP address. After deploying the pipeline, you noticed that the pipeline failed to extract data from the Cloud SQL instance due to connection failure. You verified that VPC Service Controls and shared VPC are not in use in these projects. You want to resolve this error while ensuring that the data does not go through the public internet. What should you do?

  • A. Set up VPC Network Peering between Project A and Project B. Add a firewall rule to allow the peered subnet range to access all instances on the network. Most Voted
  • B. Turn off the external IP addresses on the Dataflow worker. Enable Cloud NAT in Project A.
  • C. Add the external IP addresses of the Dataflow worker as authorized networks in the Cloud SQL instance.
  • D. Set up VPC Network Peering between Project A and Project B. Create a Compute Engine instance without external IP address in Project B on the peered subnet to serve as a proxy server to the Cloud SQL database.

Odpowiedź: A.

Wyjaśnienie: Problem polega na tym, że Dataflow w Projekcie A nie ma bezpośredniego dostępu do Cloud SQL w Projekcie B, ponieważ instancja Cloud SQL nie ma publicznego IP, a VPC Service Controls i Shared VPC nie są używane. VPC Network Peering pozwala na stworzenie prywatnej sieci pomiędzy dwoma projektami, umożliwiając komunikację tak, jakby znajdowały się w tej samej sieci. Dodanie reguły zapory, która zezwala na ruch z zakresu podsieci w Projekcie A do instancji Cloud SQL w Projekcie B, jest niezbędne, aby umożliwić Dataflow dostęp do bazy danych. To rozwiązanie zapewnia, że ruch nie przechodzi przez publiczny internet, co jest wymogiem zadania.

A. Jest to poprawne rozwiązanie, jak wyjaśniono powyżej.
B. Wyłączenie zewnętrznych adresów IP Dataflow worker i włączenie Cloud NAT nadal wymagałoby, aby ruch wychodzący z Dataflow przechodził przez publiczny internet, co jest sprzeczne z wymaganiami zadania.
C. Dodanie zewnętrznych adresów IP Dataflow worker do autoryzowanych sieci w Cloud SQL jest niepraktyczne i niebezpieczne, ponieważ wymagałoby zarządzania i aktualizowania adresów IP Dataflow worker, które mogą się zmieniać, a także narażałoby instancję Cloud SQL na dostęp z publicznego internetu.
D. Użycie instancji Compute Engine jako proxy jest bardziej skomplikowane i mniej wydajne niż bezpośrednie połączenie przez VPC Network Peering, a także wprowadza dodatkowy punkt awarii.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Poprawną odpowiedzią jest A. Problem polega na braku możliwości komunikacji pomiędzy Dataflow w Projekcie A a instancją Cloud SQL w Projekcie B, ponieważ instancja Cloud SQL nie posiada publicznego adresu IP, a nie korzystamy z VPC Service Controls ani Shared VPC. VPC Network Peering umożliwia stworzenie prywatnej sieci pomiędzy tymi projektami, co pozwala na komunikację tak, jakby znajdowały się w tej samej sieci. Dodanie reguły zapory zezwalającej na ruch z zakresu podsieci w Projekcie A do instancji Cloud SQL w Projekcie B jest niezbędne, aby umożliwić Dataflow dostęp do bazy danych. To rozwiązanie spełnia wymaganie, aby dane nie przechodziły przez publiczny internet.

Dyskusja użytkowników potwierdza, że odpowiedź A jest najbardziej bezpośrednia i efektywna. Wielu użytkowników zwraca uwagę na niepotrzebną złożoność odpowiedzi D, która wprowadza dodatkowy element (proxy) bez wyraźnej potrzeby. Argumenty przeciwko odpowiedzi A, dotyczące zezwalania na dostęp do "wszystkich instancji", są słuszne z punktu widzenia bezpieczeństwa, ale w kontekście pytania, które nie wymaga optymalizacji pod kątem bezpieczeństwa, a jedynie rozwiązania problemu połączenia, odpowiedź A jest wystarczająca.

Odpowiedź B jest niepoprawna, ponieważ użycie Cloud NAT nadal wymagałoby przechodzenia ruchu przez publiczny internet. Odpowiedź C jest niepraktyczna i niebezpieczna, ponieważ wymagałaby zarządzania dynamicznymi adresami IP Dataflow worker i narażałaby instancję Cloud SQL na dostęp z publicznego internetu.

Użytkownik Lestrang słusznie zauważa, że VPC Network Peering nie zapewnia routingu tranzytywnego, ale w tym przypadku nie ma to znaczenia, ponieważ mamy bezpośrednie połączenie pomiędzy dwoma projektami (A i B).
