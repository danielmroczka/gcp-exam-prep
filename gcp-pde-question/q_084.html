After migrating ETL jobs to run on BigQuery, you need to verify that the output of the migrated jobs is the same as the output of the original. You've loaded a table containing the output of the original job and want to compare the contents with output from the migrated job to show that they are identical. The tables do not contain a primary key column that would enable you to join them together for comparison.
What should you do?

  • A. Select random samples from the tables using the RAND() function and compare the samples.
  • B. Select random samples from the tables using the HASH() function and compare the samples.
  • C. Use a Dataproc cluster and the BigQuery Hadoop connector to read the data from each table and calculate a hash from non-timestamp columns of the table after sorting. Compare the hashes of each table. Most Voted
  • D. Create stratified random samples using the OVER() function and compare equivalent samples from each table.

Odpowiedź: C

Krótkie wyjaśnienie: Porównanie danych z tabeli migrowanej do BigQuery z tabelą oryginalnej jest kluczowe, aby upewnić się, że migracja przebiegła poprawnie i dane są identyczne.  Użycie hashów pozwala na szybkie porównanie danych bez konieczności analizy każdego kolumny, co jest szczególnie przydatne, gdy nie ma klucza głównego.  Wybór metody z wykorzystaniem hashów jest najbardziej efektywny i unika problemów z kluczami, co czyni ją najlepszą opcją.  Wybór random samples jest mniej odpowiedni, ponieważ nie uwzględnia różnic w strukturze danych.  Opcja z użyciem stratified random samples jest zbyt złożona i wymaga dodatkowych obliczeń, a opcja z użyciem OVER() może być mniej efektywna.

Pozostałe opcje są gorsze/błędne: A.  Rand() -  Użycie rand() jest nieefektywne i nie pozwala na porównanie danych, ponieważ nie uwzględnia różnic w strukturze danych.  B.  HASH() -  Hashowanie nie jest odpowiednie, ponieważ nie uwzględnia różnic w strukturze danych i może nie być efektywne w porównaniu do użycia hashów.  C.  Użycie hashów -  Użycie hashów jest odpowiednie, ponieważ pozwala na szybkie i efektywne porównanie danych bez konieczności analizy każdej kolumny.  D.  Stratified random samples -  To rozwiązanie jest zbyt złożone i wymaga dodatkowych obliczeń, a nie jest najlepszym sposobem na porównanie danych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Use a Dataproc cluster ... calculate a hash from non-timestamp columns of the table after sorting.

Dlaczego?

Problem: Trzeba udowodnić, że dwie ogromne tabele w BigQuery (oryginalna i po migracji ETL) są identyczne, ale nie mają klucza głównego (Primary Key), który ułatwiłby złączenie (JOIN).

Strategia weryfikacji danych (Data Validation): 1. Sorting: Ponieważ tabele w BigQuery są nieuporządkowane, musimy je posortować według stałych kolumn, aby móc porównywać wiersze w tej samej kolejności. 2. Hashing: Obliczenie skrótu (hash) dla każdego wiersza (z pominięciem kolumn zmiennych jak timestampy ładowania) pozwala na zredukowanie całego wiersza do jednej wartości. 3. Dataproc: Przy petabajtach danych, użycie zewnętrznego silnika (Spark na Dataproc) z konektorem BigQuery pozwala na wykonanie tej ciężkiej operacji porównania (np. obliczenie sumy kontrolnej całego datasetu) bez przeciążania slotów BQ lub w sposób bardziej elastyczny programowo.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A, B, D. Sampling (Próbkowanie): Wybranie losowych próbek (RAND, HASH, Stratified) nie udowadnia, że tabele są identyczne. Może jedynie z dużym prawdopodobieństwem stwierdzić, że nie ma błędów, ale w audycie migracji ETL często wymagana jest 100% pewność ("show that they are identical"). Tylko pełny skan (C) to zapewnia.
