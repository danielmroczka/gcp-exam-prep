You have designed an Apache Beam processing pipeline that reads from a Pub/Sub topic. The topic has a message retention duration of one day, and writes to a Cloud Storage bucket. You need to select a bucket location and processing strategy to prevent data loss in case of a regional outage with an RPO of 15 minutes. What should you do?

  • A. 1. Use a dual-region Cloud Storage bucket.
    2. Monitor Dataflow metrics with Cloud Monitoring to determine when an outage occurs.
    3. Seek the subscription back in time by 15 minutes to recover the acknowledged messages.
    4. Start the Dataflow job in a secondary region.
  • B. 1. Use a multi-regional Cloud Storage bucket.
    2. Monitor Dataflow metrics with Cloud Monitoring to determine when an outage occurs.
    3. Seek the subscription back in time by 60 minutes to recover the acknowledged messages.
    4. Start the Dataflow job in a secondary region.
  • C. 1. Use a regional Cloud Storage bucket.
    2. Monitor Dataflow metrics with Cloud Monitoring to determine when an outage occurs.
    3. Seek the subscription back in time by one day to recover the acknowledged messages.
    4. Start the Dataflow job in a secondary region and write in a bucket in the same region.
  • D. 1. Use a dual-region Cloud Storage bucket with turbo replication enabled.
    2. Monitor Dataflow metrics with Cloud Monitoring to determine when an outage occurs.

Odpowiedź: D.

Opcja D jest poprawna, ponieważ łączy kilka kluczowych elementów zapewniających odporność na awarie i minimalizację utraty danych (RPO 15 minut). Użycie dual-region Cloud Storage bucket z turbo replication zapewnia wysoką dostępność i replikację danych w dwóch regionach, chroniąc przed utratą danych w przypadku awarii jednego z nich. Monitorowanie metryk Dataflow pozwala na szybkie wykrycie problemów. Seek w Pub/Sub na 60 minut pozwala odzyskać wiadomości, które mogły zostać przetworzone, ale nie zapisane w Cloud Storage przed awarią (uwzględniając potencjalne opóźnienia). Uruchomienie Dataflow w regionie wtórnym zapewnia kontynuację przetwarzania po awarii w regionie podstawowym.

A. Regional Cloud Storage nie zapewnia odporności na awarie regionalne.
B. Multi-regional Cloud Storage oferuje wysoką dostępność, ale seek na 60 minut może być niewystarczający dla RPO 15 minut.
C. Regional Cloud Storage jest niewystarczający, a seek na jeden dzień jest zbyt długi w stosunku do RPO.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Odpowiedź D jest najbardziej kompleksowa i poprawna, ponieważ uwzględnia wszystkie aspekty wymagane przez pytanie, w szczególności RPO (Recovery Point Objective) na poziomie 15 minut. Kluczowym elementem jest użycie dual-region Cloud Storage bucket z włączoną turbo replikacją. Turbo replikacja, jak wskazują liczne komentarze użytkowników (lipa31, datapassionate, SVGoogle89), gwarantuje replikację danych w obu regionach w ciągu 15 minut, spełniając wymagany RPO.

Dodatkowe elementy odpowiedzi D, takie jak monitorowanie metryk Dataflow, cofanie subskrypcji Pub/Sub o 60 minut oraz uruchomienie Dataflow w regionie wtórnym, stanowią dodatkowe warstwy zabezpieczeń. Cofanie subskrypcji o 60 minut zapewnia pewność odzyskania danych w przypadku opóźnień w replikacji lub problemów z zapisem do Cloud Storage. Uruchomienie Dataflow w regionie wtórnym zapewnia ciągłość przetwarzania w przypadku awarii regionu podstawowego.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Chociaż dual-region bucket jest dobrym rozwiązaniem, brak turbo replikacji oznacza, że replikacja może trwać dłużej niż 15 minut, naruszając RPO. Dodatkowo, cofanie subskrypcji o 15 minut może być niewystarczające w przypadku opóźnień.
  • B: Multi-regional bucket oferuje wysoką dostępność, ale nie gwarantuje tak niskiego RPO jak dual-region z turbo replikacją. Cofanie subskrypcji o 60 minut jest zbędne, jeśli turbo replikacja działa poprawnie.
  • C: Regional bucket nie zapewnia odporności na awarie regionalne, co jest kluczowym wymaganiem pytania. Cofanie subskrypcji o jeden dzień jest zupełnie niepraktyczne i nie spełnia RPO.

Dyskusja użytkowników potwierdza, że turbo replikacja jest kluczowa (0dd4e0c, shangning007, qq589539483084gfrgrgfr). Chociaż niektórzy użytkownicy (m_a_p_s, raaad) początkowo wskazywali odpowiedź A, późniejsze komentarze i linki do dokumentacji Google Cloud (datapassionate, SVGoogle89) wskazują na przewagę odpowiedzi D.
