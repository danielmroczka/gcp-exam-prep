You have a BigQuery dataset named “customers”. All tables will be tagged by using a Data Catalog tag template named “gdpr”. The template contains one mandatory field, “has_sensitive_data”, with a boolean value. All employees must be able to do a simple search and find tables in the dataset that have either true or false in the “has_sensitive_data’ field. However, only the Human Resources (HR) group should be able to see the data inside the tables for which “has_sensitive data” is true. You give the all employees group the bigquery.metadataViewer and bigquery.connectionUser roles on the dataset. You want to minimize configuration overhead. What should you do next?

  • A. Create the “gdpr” tag template with private visibility. Assign the bigquery.dataViewer role to the HR group on the tables that contain sensitive data.
  • B. Create the “gdpr” tag template with private visibility. Assign the datacatalog.tagTemplateViewer role on this tag to the all employees group, and assign the bigquery.dataViewer role to the HR group on the tables that contain sensitive data.
  • C. Create the “gdpr” tag template with public visibility. Assign the bigquery.dataViewer role to the HR group on the tables that contain sensitive data. Most Voted
  • D. Create the “gdpr” tag template with public visibility. Assign the datacatalog.tagTemplateViewer role on this tag to the all employees group, and assign the bigquery.dataViewer role to the HR group on the tables that contain sensitive data.

Odpowiedź: C.

Stworzenie tag template "gdpr" z publiczną widocznością pozwala wszystkim pracownikom na wyszukiwanie tabel z użyciem Data Catalog i filtrowanie po wartości "has_sensitive_data". Rola `bigquery.metadataViewer` pozwala na przeglądanie metadanych, w tym tagów, a `bigquery.connectionUser` na łączenie się z BigQuery. Ograniczenie dostępu do danych w tabelach z "has_sensitive_data" ustawionym na true, poprzez nadanie roli `bigquery.dataViewer` tylko grupie HR, zapewnia, że tylko uprawnieni użytkownicy mogą przeglądać te dane. To rozwiązanie minimalizuje konfigurację, ponieważ nie wymaga skomplikowanych ustawień widoczności tagów ani dodatkowych ról dla wszystkich pracowników.

A. Prywatna widoczność tag template uniemożliwiłaby wszystkim pracownikom wyszukiwanie tabel po tagu.
B. Nadanie `datacatalog.tagTemplateViewer` wszystkim pracownikom jest niepotrzebne, ponieważ `bigquery.metadataViewer` już pozwala na przeglądanie tagów.
D. Podobnie jak w B, nadanie `datacatalog.tagTemplateViewer` jest zbędne, a dodatkowo nie rozwiązuje problemu ograniczenia dostępu do danych.

Sugerowana odpowiedź: C

Odpowiedź C jest najbardziej poprawna i efektywna w kontekście postawionego problemu. Kluczowe jest zrozumienie, że celem jest umożliwienie wszystkim pracownikom wyszukiwania tabel oznaczonych tagiem "gdpr" (zarówno tych z "has_sensitive_data" ustawionym na true, jak i false), a jednocześnie ograniczenie dostępu do danych w tabelach zawierających dane wrażliwe tylko do grupy HR.

Stworzenie tag template z publiczną widocznością pozwala wszystkim pracownikom na przeglądanie metadanych tabel, w tym tagów, dzięki roli bigquery.metadataViewer, którą już posiadają. Rola bigquery.connectionUser pozwala na łączenie się z BigQuery, ale nie daje dostępu do danych. Następnie, nadanie roli bigquery.dataViewer grupie HR tylko na tabelach z "has_sensitive_data" ustawionym na true, skutecznie ogranicza dostęp do danych wrażliwych.

Dyskusja użytkowników potwierdza tę logikę. Użytkownik raaad trafnie zauważa, że jest to najbardziej bezpośrednie rozwiązanie z minimalnym nakładem konfiguracji. Użytkownik d11379b również podkreśla, że rola bigquery.metadataViewer już pozwala na przeglądanie tagów, co eliminuje potrzebę dodatkowych uprawnień.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Prywatna widoczność tag template uniemożliwiłaby wszystkim pracownikom wyszukiwanie tabel po tagu, co jest sprzeczne z wymaganiami zadania.
  • B i D: Nadanie roli datacatalog.tagTemplateViewer wszystkim pracownikom jest zbędne. Rola bigquery.metadataViewer już pozwala na przeglądanie tagów. Dodatkowo, w przypadku odpowiedzi D, nadanie tej roli nie rozwiązuje problemu ograniczenia dostępu do danych. Użytkownik ML6 i d11379b trafnie wskazują, że ta rola dotyczy szablonów tagów, a nie samych tagów.

Podsumowując, odpowiedź C jest najbardziej efektywna, minimalizuje konfigurację i spełnia wszystkie wymagania zadania.
