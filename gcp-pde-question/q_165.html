You work for a large bank that operates in locations throughout North America. You are setting up a data storage system that will handle bank account transactions. You require ACID compliance and the ability to access data with SQL. Which solution is appropriate?

  • A. Store transaction data in Cloud Spanner. Enable stale reads to reduce latency.
  • B. Store transaction in Cloud Spanner. Use locking read-write transactions. Most Voted
  • C. Store transaction data in BigQuery. Disabled the query cache to ensure consistency.
  • D. Store transaction data in Cloud SQL. Use a federated query BigQuery for analysis.

Odpowiedź: B.

Cloud Spanner to w pełni zarządzana, globalnie rozproszona baza danych relacyjna, która zapewnia silną spójność ACID, nawet w skali globalnej. Jest to kluczowe dla banku, który przetwarza krytyczne transakcje finansowe. Użycie blokowanych transakcji odczytu-zapisu (locking read-write transactions) gwarantuje, że transakcje są wykonywane w sposób atomowy, spójny, izolowany i trwały, co jest niezbędne dla integralności danych finansowych. Spanner oferuje również dostęp do danych za pomocą SQL, co spełnia to wymaganie.  Stale reads (opcja A) naruszają wymaganie ACID, ponieważ mogą zwracać nieaktualne dane.

A. Stale reads obniżają spójność danych, co jest niedopuszczalne dla transakcji bankowych.
C. BigQuery jest hurtownią danych zoptymalizowaną pod kątem analizy, a nie transakcyjnego przetwarzania, i nie gwarantuje ACID.
D. Cloud SQL jest dobrym rozwiązaniem dla aplikacji, ale federated query z BigQuery nie zapewnia natywnej spójności transakcyjnej.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Poprawną odpowiedzią jest B: Przechowywanie transakcji w Cloud Spanner i używanie blokowanych transakcji odczytu-zapisu. Cloud Spanner to w pełni zarządzana, globalnie rozproszona baza danych relacyjna, zaprojektowana do obsługi dużych obciążeń transakcyjnych z silną spójnością ACID. Jest to kluczowe dla banku, który przetwarza krytyczne transakcje finansowe, gdzie integralność danych jest najważniejsza.

Blokowane transakcje odczytu-zapisu gwarantują, że każda transakcja jest wykonywana w sposób atomowy, spójny, izolowany i trwały (ACID). Wymaganie dostępu do danych za pomocą SQL jest również spełnione przez Cloud Spanner.

Opcja A jest niepoprawna, ponieważ użycie "stale reads" narusza wymaganie ACID, zwracając potencjalnie nieaktualne dane. Opcja C jest nieodpowiednia, ponieważ BigQuery to hurtownia danych zoptymalizowana pod kątem analizy, a nie transakcyjnego przetwarzania. Nie gwarantuje ona również ACID. Opcja D, choć Cloud SQL zapewnia ACID, nie skaluje się tak dobrze jak Cloud Spanner, a użycie zapytań federacyjnych z BigQuery nie zapewnia natywnej spójności transakcyjnej.

Dyskusja użytkowników w większości zgadza się z odpowiedzią B, podkreślając, że Cloud Spanner jest bardziej odpowiedni dla dużych banków ze względu na jego skalowalność i silną spójność. Niektórzy użytkownicy początkowo rozważali odpowiedź D, ale ostatecznie uznali, że Cloud Spanner jest lepszym rozwiązaniem ze względu na wymagania dotyczące globalnej spójności i potencjalnie dużego wolumenu transakcji.

Komentarze wskazują również, że "large bank" sugeruje potrzebę skalowalności, którą Cloud Spanner zapewnia, a Cloud SQL może nie być w stanie obsłużyć.
