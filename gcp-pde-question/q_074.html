Your financial services company is moving to cloud technology and wants to store 50 TB of financial time-series data in the cloud. This data is updated frequently and new data will be streaming in all the time. Your company also wants to move their existing Apache Hadoop jobs to the cloud to get insights into this data.
Which product should they use to store the data?

  • A. Cloud Bigtable Most Voted
  • B. Google BigQuery
  • C. Google Cloud Storage
  • D. Google Cloud Datastore

Odpowiedź: A

Wyjaśnienie: Cloud Bigtable jest idealnym rozwiązaniem dla tej sytuacji, ponieważ jest zaprojektowany do przechowywania dużych ilości danych, w tym danych time-series, w sposób wydajny i skalowalny.  Bigtable oferuje szybkie zapytania i wysoką wydajność, co jest kluczowe dla aktualizacji danych w czasie rzeczywistym.  Ponadto, Bigtable jest zaprojektowany do pracy z dużymi ilościami danych, co czyni go odpowiednim dla danych finansowych, które mogą być bardzo duże.  Google BigQuery, choć potężny, może być mniej wydajny w operacjach na dużych zbiorach danych, a Google Cloud Storage może być niewystarczający ze względu na ograniczenia dotyczące rozmiaru plików i formatów danych.  Ostatecznie, Cloud Datastore jest mniej skalowalnym i może być mniej odpowiednim dla dynamicznego wzrostu danych.

Pozostałe opcje są mniej odpowiednie: Google BigQuery jest przeznaczony do analizy danych, a Google Cloud Storage jest przeznaczony do przechowywania plików, a Cloud Datastore jest bardziej odpowiedni do przechowywania danych, które są często aktualizowane.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Prawidłową odpowiedzią jest A. Cloud Bigtable.

Dlaczego?

Wymagania: 1. 50 TB of financial time-series data: Bigtable jest idealne dla danych szeregów czasowych (Time-Series) o dużej skali (TB/PB). 2. Updated frequently and new data streaming: Bigtable ma ekstremalnie wysoką przepustowość zapisu (high write throughput). 3. Existing Apache Hadoop jobs: To kluczowy wskaźnik. Bigtable implementuje interfejs Apache HBase API. Dzięki temu istniejące zadania Hadoop/Spark napisane pod HBase mogą działać na Bigtable bez (lub z minimalnymi) zmianami. Jest to standardowa ścieżka migracji z Hadoop/HBase do GCP.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • B. BigQuery: Świetne do analityki (OLAP), ale gorsze do "frequent updates" (BigQuery jest append-only w naturze, modyfikacje są kosztowne i limitowane). Nie jest kompatybilne z API Hadoop/HBase w ten sam sposób co Bigtable.
  • C. Cloud Storage: Dobry magazyn (Blob storage), ale nie baza danych dla "frequent updates". Nie wspiera API HBase.
  • D. Cloud Datastore: Nie skaluje się tak dobrze przy 50TB i intensywnym zapisie time-series (problemy z hotspotami na indeksach). Brak kompatybilności z Hadoop.
