An online brokerage company requires a high volume trade processing architecture. You need to create a secure queuing system that triggers jobs. The jobs will run in Google Cloud and call the company's Python API to execute trades. You need to efficiently implement a solution. What should you do?

  • A. Use a Pub/Sub push subscription to trigger a Cloud Function to pass the data to the Python API. Most Voted
  • B. Write an application hosted on a Compute Engine instance that makes a push subscription to the Pub/Sub topic.
  • C. Write an application that makes a queue in a NoSQL database.
  • D. Use Cloud Composer to subscribe to a Pub/Sub topic and call the Python API.

Odpowiedź: A.

Pub/Sub z push subscription jest idealnym rozwiązaniem dla wysokiej przepustowości przetwarzania transakcji. Push subscription pozwala na automatyczne dostarczanie wiadomości do Cloud Functions bez konieczności ciągłego sprawdzania kolejki przez aplikację. Cloud Functions są bezserwerowe, skalują się automatycznie w odpowiedzi na obciążenie i są kosztowo efektywne. Połączenie Pub/Sub i Cloud Functions zapewnia szybkie i bezpieczne uruchamianie zadań, które następnie wywołują Python API do realizacji transakcji. To rozwiązanie minimalizuje opóźnienia i zapewnia wysoką dostępność systemu.

A. Opcja B wymaga utrzymania i skalowania instancji Compute Engine, co jest mniej efektywne niż bezserwerowe Cloud Functions.
B. Opcja C, użycie NoSQL do kolejkowania, nie jest dedykowanym rozwiązaniem do kolejkowania wiadomości i może być mniej wydajne i niezawodne niż Pub/Sub.
C. Opcja D, Cloud Composer, jest narzędziem do orkiestracji workflow, a nie do prostego wyzwalania zadań w odpowiedzi na wiadomości, co czyni go nadmiernie skomplikowanym dla tego przypadku użycia.

Dodatkowe źródła:

*   **Pub/Sub Documentation:** https://cloud.google.com/pubsub/docs
*   **Cloud Functions Documentation:** https://cloud.google.com/functions/docs
*   **Cloud Composer Documentation:** https://cloud.google.com/composer/docs

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Najbardziej odpowiednią odpowiedzią jest A. Use a Pub/Sub push subscription to trigger a Cloud Function to pass the data to the Python API. Dyskusja użytkowników, choć początkowo podzielona, z czasem skłania się ku tej odpowiedzi. Wiele komentarzy, szczególnie te nowsze (z ostatnich miesięcy), wskazuje, że opcja A jest bardziej efektywna i skalowalna dla wysokiej przepustowości przetwarzania transakcji. Cloud Functions są bezserwerowe, co oznacza automatyczne skalowanie i optymalizację kosztów, co jest kluczowe w przypadku wysokiego wolumenu transakcji.

Pub/Sub z push subscription umożliwia natychmiastowe wyzwalanie Cloud Functions w momencie pojawienia się nowych wiadomości, minimalizując opóźnienia. Bezpośrednie przekazywanie danych do Python API przez Cloud Function jest prostym i wydajnym rozwiązaniem.

Dlaczego inne odpowiedzi są mniej odpowiednie:

  • B: Użycie Compute Engine wymaga zarządzania infrastrukturą i skalowania, co jest mniej efektywne niż bezserwerowe Cloud Functions.
  • C: NoSQL database nie jest dedykowanym rozwiązaniem do kolejkowania wiadomości i może być mniej niezawodne i wydajne niż Pub/Sub.
  • D: Cloud Composer jest narzędziem do orkiestracji workflow, a nie do prostego wyzwalania zadań. Jest to nadmiernie skomplikowane rozwiązanie dla tego konkretnego przypadku użycia. Użytkownicy słusznie zauważają, że Cloud Composer wprowadza niepotrzebne opóźnienia (latency).

Komentarze użytkowników, którzy pracują w branży (np. squishy_fishy), potwierdzają, że opcja A jest odpowiednia dla mniejszego wolumenu, a opcja D dla większego, co dodatkowo wzmacnia argument za odpowiedzią A w kontekście pytania o wysoką przepustowość.
