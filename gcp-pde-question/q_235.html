You want to schedule a number of sequential load and transformation jobs. Data files will be added to a Cloud Storage bucket by an upstream process. There is no fixed schedule for when the new data arrives. Next, a Dataproc job is triggered to perform some transformations and write the data to BigQuery. You then need to run additional transformation jobs in BigQuery. The transformation jobs are different for every table. These jobs might take hours to complete. You need to determine the most efficient and maintainable workflow to process hundreds of tables and provide the freshest data to your end users. What should you do?

  • A. 1. Create an Apache Airflow directed acyclic graph (DAG) in Cloud Composer with sequential tasks by using the Cloud Storage, Dataproc, and BigQuery operators.
    2. Use a single shared DAG for all tables that need to go through the pipeline.
    3. Schedule the DAG to run hourly.
  • B. 1. Create an Apache Airflow directed acyclic graph (DAG) in Cloud Composer with sequential tasks by using the Cloud Storage, Dataproc, and BigQuery operators.
    2. Create a separate DAG for each table that needs to go through the pipeline.

    2. Use a single shared DAG for all tables that need to go through the pipeline.
    3. Use a Cloud Storage object trigger to launch a Cloud Function that triggers the DAG.

Odpowiedź: D.

Opcja D jest najbardziej efektywna i skalowalna, ponieważ wykorzystuje kombinację DAG-ów dla każdej tabeli oraz wyzwalaczy Cloud Storage. Tworzenie oddzielnych DAG-ów dla każdej tabeli pozwala na równoległe przetwarzanie i uniknięcie wąskich gardeł, które mogłyby wystąpić przy użyciu jednego DAG-a dla wszystkich tabel. Wyzwalacz Cloud Storage uruchamiający Cloud Function, który z kolei uruchamia DAG, zapewnia reaktywne przetwarzanie danych – pipeline uruchamia się tylko wtedy, gdy pojawią się nowe pliki w Cloud Storage, co eliminuje niepotrzebne uruchomienia. To podejście jest szczególnie ważne, gdy nie ma ustalonego harmonogramu dla przychodzących danych. Użycie Cloud Functions jako pośrednika pozwala na elastyczne zarządzanie uruchamianiem DAG-ów i potencjalne dodawanie logiki filtrowania lub walidacji.

A. Użycie pojedynczego DAG-a dla wszystkich tabel spowoduje wąskie gardło i wydłuży czas przetwarzania, a godzinowe uruchamianie DAG-a jest nieefektywne, gdy dane nie zawsze są dostępne.
B. Tworzenie oddzielnych DAG-ów jest dobre, ale brak wyzwalacza Cloud Storage powoduje niepotrzebne uruchamianie DAG-ów, nawet gdy nie ma nowych danych.
C. Pomijanie operatora Cloud Storage w DAG-u jest nieprawidłowe, ponieważ pipeline musi reagować na pojawienie się nowych plików w Cloud Storage.

Sugerowana odpowiedź: D

Odpowiedź D jest najbardziej efektywna i skalowalna w kontekście opisanego problemu. Kluczowe aspekty, które przemawiają za nią to:

  • Reaktywność na nowe dane: Wykorzystanie wyzwalacza Cloud Storage (object trigger) i Cloud Function zapewnia, że pipeline uruchomi się automatycznie tylko wtedy, gdy pojawią się nowe pliki danych. Eliminuje to niepotrzebne uruchomienia DAG-ów, co jest istotne, ponieważ nie ma ustalonego harmonogramu przychodzenia danych.
  • Równoległość i skalowalność: Tworzenie oddzielnego DAG-a dla każdej tabeli umożliwia równoległe przetwarzanie. Każda tabela może być transformowana niezależnie, co znacznie skraca całkowity czas przetwarzania, szczególnie przy dużej liczbie tabel (setki).
  • Utrzymywalność: Chociaż tworzenie wielu DAG-ów może wydawać się skomplikowane, w praktyce można je generować programowo (jak wskazano w dyskusji przez użytkownika 22c1725 i plum21), co znacznie upraszcza zarządzanie. Oddzielne DAG-i dla każdej tabeli oznaczają, że zmiany w transformacjach jednej tabeli nie wpłyną na inne.

Dyskusja użytkowników w większości skłania się ku odpowiedzi D, a nowsze komentarze (np. od af17139) potwierdzają jej zasadność. Użytkownik scaenruy, autor pytania, również potwierdził odpowiedź D.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A i B: Użycie pojedynczego DAG-a dla wszystkich tabel stworzyłoby wąskie gardło i utrudniło skalowanie. Godzinowe uruchamianie DAG-a jest nieefektywne, ponieważ pipeline uruchamiałby się nawet wtedy, gdy nie byłoby nowych danych.
  • C: Pominięcie operatora Cloud Storage w DAG-u jest błędem, ponieważ pipeline musi reagować na pojawienie się nowych plików w Cloud Storage. Chociaż pomysł z jednym DAG-iem i parametrami jest teoretycznie możliwy, to przy tak dużej liczbie tabel o różnych transformacjach staje się on niepraktyczny i trudny w utrzymaniu (jak zauważył af17139).
