You decided to use Cloud Datastore to ingest vehicle telemetry data in real time. You want to build a storage system that will account for the long-term data growth, while keeping the costs low. You also want to create snapshots of the data periodically, so that you can make a point-in-time (PIT) recovery, or clone a copy of the data for Cloud Datastore in a different environment. You want to archive these snapshots for a long time. Which two methods can accomplish this?
(Choose two.)

  • A. Use managed export, and store the data in a Cloud Storage bucket using Nearline or Coldline class. Most Voted
  • B. Use managed export, and then import to Cloud Datastore in a separate project under a unique namespace reserved for that export. Most Voted
  • C. Use managed export, and then import the data into a BigQuery table created just for that export, and delete temporary export files.
  • D. Write an application that uses Cloud Datastore client libraries to read all the entities. Treat each entity as a BigQuery table row via BigQuery streaming insert. Assign an export timestamp for each export, and attach it as an extra column for each row. Make sure that the BigQuery table is partitioned using the export timestamp column.
  • E. Write an application that uses Cloud Datastore client libraries to read all the entities. Format the exported data into a JSON file. Apply compression before storing the data in Cloud Source Repositories.

Odpowiedź: A, B.

Opcje A i B pozwalają na efektywne i ekonomiczne archiwizowanie danych z Cloud Datastore z możliwością odzyskiwania i klonowania. Managed Export umożliwia wyeksportowanie danych do Cloud Storage, a następnie przechowywanie ich w klasach Nearline lub Coldline, co znacznie obniża koszty przechowywania dla danych archiwalnych. Opcja B dodatkowo pozwala na importowanie tych eksportów do oddzielnego projektu Datastore z unikalną przestrzenią nazw, co ułatwia zarządzanie i izolację kopii danych, a także umożliwia tworzenie kopii zapasowych w różnych środowiskach. Połączenie tych dwóch metod zapewnia zarówno niskie koszty przechowywania, jak i elastyczność w zakresie odzyskiwania i klonowania danych.

A. Opcja C jest gorsza, ponieważ BigQuery, choć potężne narzędzie analityczne, nie jest optymalnym miejscem do długoterminowego archiwizowania danych, które mają być potencjalnie przywracane do Datastore.
B. Opcja D jest nieefektywna i kosztowna, ponieważ wymaga pisania i utrzymywania aplikacji do odczytu i zapisu danych, a także generuje wysokie koszty związane z BigQuery Streaming Insert.
C. Opcja E jest niepraktyczna i nieefektywna, ponieważ przechowywanie dużych ilości danych w Cloud Source Repositories (repozytorium kodu) nie jest przeznaczeniem tego narzędzia i generuje wysokie koszty.

Sugerowana odpowiedź: A, B

Prawidłowe odpowiedzi to A, B.

Dlaczego?

Wymagania: "snapshots... periodically", "point-in-time recovery", "clone copy to different environment", "archive snapshots for long time", "low cost". 1. Managed Export: Cloud Datastore posiada funkcję Managed Export/Import. Jest to najprostszy i natywny sposób na robienie kopii zapasowych (snapshotów) całej bazy lub jej części. 2. A. GCS Nearline/Coldline: Managed Export zapisuje dane do Cloud Storage. Przechowywanie tych plików w klasach o niskim koszcie (Nearline/Coldline) idealnie spełnia wymóg taniej, długoterminowej archiwizacji. 3. B. Clone to different environment: Aby spełnić wymóg PITR lub klonowania do innego środowiska (np. staging), używamy funkcji Managed Import, wskazując na pliki eksportu z GCS. Możemy zaimportować dane do innego projektu GCP, co zapewnia pełną izolację kopii.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • C. Import into BigQuery: BigQuery służy do analityki, a nie jako backup dla Datastore. Choć technicznie można zaimportować tam dane, proces "recovery" (powrót do Datastore) byłby bardzo skomplikowany.
  • D. Client libraries + Streaming insert: Pisanie własnej aplikacji do kopiowania miliardów rekordów jest nieefektywne, drogie (koszty odczytu z Datastore + koszty streaming insertu do BQ) i podatne na błędy. Należy używać wbudowanych narzędzi (Managed Export).
  • E. JSON in Source Repositories: Repozytoria kodu nie służą do składowania petabajtów danych binarnych czy JSON. Mają limity rozmiaru i są drogie w tym zastosowaniu.
