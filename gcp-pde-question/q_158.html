You need to deploy additional dependencies to all nodes of a Cloud Dataproc cluster at startup using an existing initialization action. Company security policies require that Cloud Dataproc nodes do not have access to the Internet so public initialization actions cannot fetch resources. What should you do?

  • A. Deploy the Cloud SQL Proxy on the Cloud Dataproc master
  • B. Use an SSH tunnel to give the Cloud Dataproc cluster access to the Internet
  • C. Copy all dependencies to a Cloud Storage bucket within your VPC security perimeter Most Voted
  • D. Use Resource Manager to add the service account used by the Cloud Dataproc cluster to the Network User role

Odpowiedź: C.

Wyjaśnienie: Firma ma politykę bezpieczeństwa, która zabrania klastrowi Cloud Dataproc dostępu do Internetu. Oznacza to, że nie można używać publicznych akcji inicjalizacyjnych, które pobierają zasoby z Internetu. Aby wdrożyć dodatkowe zależności na wszystkich węzłach klastra podczas uruchamiania, używając istniejącej akcji inicjalizacyjnej, należy skopiować wszystkie zależności do bucketu Cloud Storage w obrębie perymetru bezpieczeństwa VPC. Akcja inicjalizacyjna może następnie pobrać zależności z tego bucketu, bez konieczności dostępu do Internetu. To rozwiązanie spełnia wymagania bezpieczeństwa firmy i umożliwia wdrożenie potrzebnych zależności.

A. Wdrożenie Cloud SQL Proxy na węźle głównym nie rozwiązuje problemu pobierania zależności, ponieważ nadal wymaga dostępu do zasobów zewnętrznych.
B. Użycie tunelu SSH w celu zapewnienia klastrowi Cloud Dataproc dostępu do Internetu narusza politykę bezpieczeństwa firmy.
D. Dodanie konta usługi klastra Cloud Dataproc do roli Network User w Resource Manager nie pozwala na pobieranie zależności, a jedynie na dostęp do sieci.

Dodatkowe źródła:

*   **Cloud Dataproc Initialization Actions:** https://cloud.google.com/dataproc/docs/concepts/configuring-clusters/init-actions
*   **Cloud Storage:** https://cloud.google.com/storage
*   **VPC Security Perimeter:** https://cloud.google.com/vpc-service-controls

Sugerowana odpowiedź: C

Poprawną odpowiedzią jest C. Firma narzuciła politykę bezpieczeństwa, która zabrania klastrowi Cloud Dataproc dostępu do Internetu. W związku z tym, nie można używać publicznych akcji inicjalizacyjnych, które pobierają zasoby z zewnętrznych źródeł. Aby wdrożyć dodatkowe zależności, należy je umieścić w Cloud Storage bucket w obrębie perymetru bezpieczeństwa VPC. Akcja inicjalizacyjna może następnie pobrać te zależności z bucketu bez konieczności dostępu do Internetu.

Wiele komentarzy użytkowników potwierdza tę odpowiedź, wskazując na dokumentację Google Cloud (https://cloud.google.com/dataproc/docs/concepts/configuring-clusters/init-actions oraz https://cloud.google.com/dataproc/docs/concepts/configuring-clusters/network#create_a_cloud_dataproc_cluster_with_internal_ip_address_only). Użytkownik musumusu słusznie zauważa, że praktyczne doświadczenie pomaga zrozumieć tę kwestię, a zależności (np. pakiety Pythona, pliki JAR) mogą być przechowywane w bucketach i konfigurowane bez dostępu do UI.

Dlaczego inne odpowiedzi są niepoprawne:

  • A: Wdrożenie Cloud SQL Proxy nie rozwiązuje problemu pobierania zależności, ponieważ nadal wymaga dostępu do zasobów zewnętrznych lub przechowywania ich w Cloud SQL, co nie jest optymalne.
  • B: Użycie tunelu SSH narusza politykę bezpieczeństwa firmy, która zabrania dostępu do Internetu.
  • D: Dodanie konta usługi do roli Network User nie umożliwia pobierania zależności, a jedynie dostęp do sieci.
