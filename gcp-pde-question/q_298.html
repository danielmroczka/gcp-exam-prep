One of your encryption keys stored in Cloud Key Management Service (Cloud KMS) was exposed. You need to re- encrypt all of your CMEK-protected Cloud Storage data that used that key, and then delete the compromised key. You also want to reduce the risk of objects getting written without customer-managed encryption key (CMEK) protection in the future. What should you do?

  • A. Rotate the Cloud KMS key version. Continue to use the same Cloud Storage bucket.
  • B. Create a new Cloud KMS key. Set the default CMEK key on the existing Cloud Storage bucket to the new one.
  • C. Create a new Cloud KMS key. Create a new Cloud Storage bucket. Copy all objects from the old bucket to the new one bucket while specifying the new Cloud KMS key in the copy command.
  • D. Create a new Cloud KMS key. Create a new Cloud Storage bucket configured to use the new key as the default CMEK key. Copy all objects from the old bucket to the new bucket without specifying a key. Most Voted

Odpowiedź: D.

Opcja D jest prawidłowa, ponieważ zapewnia kompleksowe rozwiązanie problemu. Najpierw tworzy nowy klucz KMS, eliminując ryzyko związane z użyciem skompromitowanego klucza. Następnie konfiguruje nowy bucket Cloud Storage z nowym kluczem jako domyślnym CMEK, co gwarantuje, że wszystkie nowo zapisywane obiekty będą automatycznie szyfrowane przy użyciu nowego klucza. Kopiowanie danych z starego bucketu do nowego bez podawania klucza szyfrowania podczas kopiowania wykorzystuje domyślny klucz CMEK skonfigurowany w nowym bucketcie, co re-encryptuje wszystkie istniejące dane. To podejście minimalizuje ryzyko przyszłych błędów w konfiguracji szyfrowania.

A. Rotacja wersji klucza nie re-encryptuje istniejących danych, a jedynie wpływa na nowe operacje szyfrowania.
B. Samo utworzenie nowego klucza i ustawienie go jako domyślnego nie re-encryptuje istniejących danych w starym bucketcie.
C. Tworzenie nowego bucketu i ręczne podawanie klucza podczas kopiowania jest czasochłonne i podatne na błędy ludzkie, a także nie zapewnia automatycznej ochrony dla nowych obiektów.

Sugerowana odpowiedź: D

Odpowiedź D jest prawidłowa i najlepiej odpowiada na postawiony problem. Kluczowe jest zrozumienie, że po naruszeniu klucza szyfrującego, konieczne jest ponowne zaszyfrowanie wszystkich danych, które zostały za pomocą niego zabezpieczone. Opcja D realizuje to poprzez:

  • Utworzenie nowego klucza KMS: Eliminuje ryzyko dalszego wykorzystania skompromitowanego klucza.
  • Utworzenie nowego bucketu Cloud Storage z nowym kluczem jako domyślnym CMEK: Zapewnia, że wszystkie nowe obiekty będą szyfrowane nowym kluczem.
  • Skopiowanie danych ze starego bucketu do nowego bez podawania klucza: Wykorzystuje domyślny klucz CMEK skonfigurowany w nowym bucketcie, co automatycznie re-encryptuje wszystkie istniejące dane.

Dyskusja użytkowników w większości potwierdza poprawność odpowiedzi D (89% głosów). Komentarze użytkowników, takie jak raaad, chickenwingz, JyoGCP i ML6, trafnie argumentują, dlaczego inne opcje są nieprawidłowe.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Rotacja wersji klucza nie re-encryptuje istniejących danych. Dotyczy jedynie nowych operacji szyfrowania. (Potwierdzone przez chickenwingz i ML6 z linkami do dokumentacji Google Cloud).
  • B: Samo utworzenie nowego klucza i ustawienie go jako domyślnego nie re-encryptuje danych w starym bucketcie.
  • C: Chociaż kopiowanie z podaniem klucza re-encryptuje dane podczas kopiowania, nie rozwiązuje problemu przyszłych obiektów zapisywanych bez klucza, a także jest bardziej podatne na błędy ludzkie. Komentarze desertlotus1211 i judy_data wskazują na ryzyko powrotu do szyfrowania zarządzanego przez Google w przypadku błędnej konfiguracji.

Użytkownik Medmah sugeruje opcję A, ale ML6 poprawnie wyjaśnia, że rotacja klucza nie powoduje re-encryptacji danych.

Podsumowując, odpowiedź D jest najbardziej kompleksowym i bezpiecznym rozwiązaniem, które uwzględnia zarówno re-encryptację istniejących danych, jak i zabezpieczenie przed przyszłymi problemami.
