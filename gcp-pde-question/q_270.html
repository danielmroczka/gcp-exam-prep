You need to create a SQL pipeline. The pipeline runs an aggregate SQL transformation on a BigQuery table every two hours and appends the result to another existing BigQuery table. You need to configure the pipeline to retry if errors occur. You want the pipeline to send an email notification after three consecutive failures. What should you do?

  • A. Use the BigQueryUpsertTableOperator in Cloud Composer, set the retry parameter to three, and set the email_on_failure parameter to true.
  • B. Use the BigQueryInsertJobOperator in Cloud Composer, set the retry parameter to three, and set the email_on_failure parameter to true. Most Voted
  • C. Create a BigQuery scheduled query to run the SQL transformation with schedule options that repeats every two hours, and enable email notifications.
  • D. Create a BigQuery scheduled query to run the SQL transformation with schedule options that repeats every two hours, and enable notification to Pub/Sub topic. Use Pub/Sub and Cloud Functions to send an email after three failed executions.

Odpowiedź: B.

Użycie `BigQueryInsertJobOperator` w Cloud Composer jest najlepszym rozwiązaniem, ponieważ pozwala na uruchamianie zadań BigQuery, w tym transformacji SQL, w sposób zaplanowany. Ustawienie parametru `retry` na trzy zapewnia ponawianie prób w przypadku błędów, a `email_on_failure` na `true` powoduje wysłanie powiadomienia e-mail po niepowodzeniu zadania.  W tym przypadku, ponieważ wymagane są ponowne próby i powiadomienia e-mail, `BigQueryInsertJobOperator` oferuje bezpośrednią konfigurację tych funkcji.  Operator ten jest przeznaczony do uruchamiania pojedynczych zadań BigQuery, co idealnie pasuje do opisu transformacji i dołączania wyników.  Dzięki temu pipeline będzie niezawodny i łatwy w monitorowaniu.

A. `BigQueryUpsertTableOperator` służy do aktualizacji lub wstawiania danych do tabeli, a nie do uruchamiania transformacji i dołączania wyników.
C. Scheduled queries nie oferują wbudowanego mechanizmu ponawiania prób, co jest kluczowe w wymaganiach.
D. Użycie Pub/Sub i Cloud Functions jest bardziej skomplikowane niż bezpośrednie wykorzystanie `email_on_failure` w `BigQueryInsertJobOperator` i wymaga dodatkowej konfiguracji.

Sugerowana odpowiedź: D

Analizując dyskusję użytkowników oraz treść pytania, najbardziej precyzyjną odpowiedzią wydaje się być D. Kluczowym elementem pytania jest wysłanie powiadomienia e-mail po trzech kolejnych niepowodzeniach całego pipeline'u, a nie pojedynczego zadania. Operatorzy `BigQueryInsertJobOperator` (odpowiedź B) i `BigQueryUpsertTableOperator` (odpowiedź A) oferują mechanizm ponawiania prób i powiadomień o błędach, ale dotyczą one pojedynczych zadań w DAG-u Cloud Composer. Powiadomienie `email_on_failure` w tych operatorach jest wysyłane po wyczerpaniu limitu ponownych prób dla danego zadania, a nie po trzech kolejnych nieudanych uruchomieniach całego pipeline'u.

Odpowiedź D wykorzystuje BigQuery Scheduled Queries do zaplanowania uruchomienia transformacji co dwie godziny. Następnie, w przypadku niepowodzenia, wykorzystuje Pub/Sub i Cloud Functions do monitorowania i wysyłania powiadomień e-mail po trzech kolejnych nieudanych próbach. To podejście pozwala na monitorowanie całego pipeline'u, a nie tylko pojedynczych zadań, co jest zgodne z wymaganiami pytania.

Wiele komentarzy użytkowników (np. 56d02cd, gabbferreira, plum21, b3e59c2, datapassionate) wskazuje na to, że odpowiedź B nie spełnia warunku dotyczącego powiadomienia po trzech kolejnych niepowodzeniach. Retries w Cloud Composer dotyczą pojedynczych zadań, a nie całego DAG-u.

Odpowiedź C jest niepoprawna, ponieważ BigQuery Scheduled Queries nie oferują wbudowanego mechanizmu ponawiania prób.

Chociaż odpowiedź B jest często wybierana przez użytkowników, dokładna analiza wymagań pytania wskazuje, że odpowiedź D jest bardziej precyzyjna i kompleksowa.
