A data scientist has created a BigQuery ML model and asks you to create an ML pipeline to serve predictions. You have a REST API application with the requirement to serve predictions for an individual user ID with latency under 100 milliseconds. You use the following query to generate predictions: SELECT predicted_label, user_id FROM ML.PREDICT (MODEL 'dataset.model', table user_features). How should you create the ML pipeline?

  • A. Add a WHERE clause to the query, and grant the BigQuery Data Viewer role to the application service account.
  • B. Create an Authorized View with the provided query. Share the dataset that contains the view with the application service account.
  • C. Create a Dataflow pipeline using BigQueryIO to read results from the query. Grant the Dataflow Worker role to the application service account.
  • D. Create a Dataflow pipeline using BigQueryIO to read predictions for all users from the query. Write the results to Bigtable using BigtableIO. Grant the Bigtable Reader role to the application service account so that the application can read predictions for individual users from Bigtable. Most Voted

Odpowiedź: D.

Aby spełnić wymagania niskiej latencji (poniżej 100ms) dla predykcji dla pojedynczego użytkownika, konieczne jest przechowywanie predykcji w bazie danych o niskiej latencji, takiej jak Bigtable. Bezpośrednie wykonywanie zapytania `ML.PREDICT` dla każdego żądania API spowoduje przekroczenie limitu czasu. Dataflow pipeline pozwala na przetworzenie predykcji dla wszystkich użytkowników w sposób wsadowy i zapisanie ich w Bigtable. Następnie aplikacja REST API może szybko pobierać predykcje dla konkretnego użytkownika z Bigtable. Przyznanie roli Bigtable Reader uprawnia aplikację do odczytu tych predykcji.

A. Dodanie klauzuli WHERE nie rozwiązuje problemu latencji, ponieważ każde żądanie nadal będzie wymagało zapytania do BigQuery.
B. Authorized View nie poprawi znacząco latencji, ponieważ nadal będzie wymagało zapytania do BigQuery przy każdym żądaniu.
C. Użycie Dataflow do odczytu wyników zapytania i zwracanie ich bezpośrednio nie spełnia wymogu niskiej latencji, ponieważ Dataflow nie jest przeznaczony do obsługi żądań w czasie rzeczywistym.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Poprawną odpowiedzią jest D. Kluczowym wymaganiem w tym pytaniu jest zapewnienie niskiej latencji (poniżej 100ms) dla predykcji dla pojedynczego użytkownika. Bezpośrednie wykonywanie zapytania ML.PREDICT w BigQuery dla każdego żądania API byłoby zbyt wolne i prawdopodobnie przekroczyłoby limit czasu. BigQuery, mimo swojej mocy, nie jest zoptymalizowany do obsługi żądań o bardzo niskiej latencji dla pojedynczych rekordów.

Rozwiązanie polega na wstępnym obliczeniu predykcji dla wszystkich użytkowników (lub przynajmniej dla tych, których predykcje są często żądane) i zapisaniu ich w bazie danych o niskiej latencji, takiej jak Bigtable. Dataflow pipeline umożliwia efektywne przetworzenie danych z BigQuery i zapisanie ich w Bigtable. Aplikacja REST API może następnie szybko pobierać predykcje dla konkretnego użytkownika z Bigtable, spełniając wymaganie 100ms. Przyznanie roli Bigtable Reader uprawnia aplikację do odczytu tych predykcji.

Większość użytkowników w dyskusji (88% głosów) poprawnie wskazuje odpowiedź D, podkreślając znaczenie wymagania dotyczącego latencji. Komentarze takie jak te od MaxNRG, Lanro i midgoo trafnie wskazują na potrzebę cachowania predykcji i wykorzystania Bigtable jako punktu końcowego dla aplikacji klienckich.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Dodanie klauzuli WHERE do zapytania nie eliminuje problemu latencji, ponieważ każde żądanie nadal wymaga zapytania do BigQuery.
  • B: Authorized View nadal wymaga zapytania do BigQuery przy każdym żądaniu, co nie spełnia wymogu niskiej latencji. Chociaż Authorized View upraszcza dostęp do danych, nie poprawia wydajności zapytań.
  • C: Użycie Dataflow do odczytu wyników zapytania i zwracanie ich bezpośrednio nie spełnia wymogu niskiej latencji, ponieważ Dataflow nie jest przeznaczony do obsługi żądań w czasie rzeczywistym. Dodatkowo, Dataflow sam w sobie wprowadza opóźnienie.
