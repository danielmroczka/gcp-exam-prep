Your infrastructure includes a set of YouTube channels. You have been tasked with creating a process for sending the YouTube channel data to Google Cloud for analysis. You want to design a solution that allows your world-wide marketing teams to perform ANSI SQL and other types of analysis on up-to-date YouTube channels log data. How should you set up the log data transfer into Google Cloud?

  • A. Use Storage Transfer Service to transfer the offsite backup files to a Cloud Storage Multi-Regional storage bucket as a final destination.
  • B. Use Storage Transfer Service to transfer the offsite backup files to a Cloud Storage Regional bucket as a final destination.

Odpowiedź: **C**.

**Wyjaśnienie:**
BigQuery Data Transfer Service (DTS) jest dedykowanym, w pełni zarządzanym rozwiązaniem Google Cloud do automatycznego importowania danych z usług takich jak YouTube (zarówno raportów kanałów, jak i właścicieli treści). Narzędzie to eliminuje konieczność tworzenia własnych skryptów ETL, co zapewnia, że dane są zawsze aktualne i gotowe do analizy. Wybór lokalizacji **Multi-Regional** jest kluczowy w tym scenariuszu, ponieważ infrastruktura musi obsługiwać zespoły marketingowe działające na całym świecie („world-wide”), zapewniając im wysoką dostępność i wydajność dostępu do danych. Dzięki temu analitycy mogą natychmiastowo wykonywać zapytania ANSI SQL bezpośrednio w środowisku BigQuery.

**Dlaczego pozostałe opcje są gorsze/błędne:**
*   **A:** Storage Transfer Service służy do przesyłania dużych zbiorów danych między chmurami lub systemami plików, a nie do natywnego pobierania raportów analitycznych z YouTube.
*   **B:** Ta opcja jest błędna, ponieważ Storage Transfer Service nie jest właściwym narzędziem dla YouTube, a lokalizacja regionalna (Regional) nie spełnia wymagań wysokiej dostępności dla zespołów globalnych.
*   **C:** (Jest to poprawna opcja, ponieważ łączy automatyzację procesu przez DTS z globalnym zasięgiem Multi-Regional).
*   **D:** Choć wykorzystuje odpowiednie narzędzie (DTS), wybór lokalizacji Regional jest gorszy ze względu na ograniczenie dostępności danych do jednego regionu geograficznego dla światowych zespołów.

**Dodatkowe źródła:**
*   BigQuery Data Transfer Service for YouTube
*   BigQuery Dataset Locations

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Use BigQuery Data Transfer Service....

Dlaczego?

1. YouTube Channel Data: BigQuery Data Transfer Service (DTS) ma wbudowany, natywny konektor do "YouTube Channel Reports". Automatyzuje on pobieranie raportów z YouTube bezpośrednio do BigQuery. 2. ANSI SQL Analysis: BigQuery obsługuje standard ANSI SQL. 3. World-wide marketing teams: Sugeruje potrzebę wysokiej dostępności i wydajności w różnych regionach. Chociaż BigQuery dataset jest technicznie "lokalizacją" (np. US, EU), odpowiedź C wspomina o "Cloud Storage Multi-Regional storage bucket as a final destination" - tu jest pewna nieścisłość w treści pytania/odpowiedzi, ponieważ DTS zazwyczaj ładuje prosto do BQ. Jednakże, kontekst "offsite backup files" w odpowiedziach sugeruje, że może chodzić o importowanie logów, które gdzieś leżą. Korekta: Oryginalne pytanie mówi o "sending YouTube channel data". DTS to najlepsze narzędzie. Odpowiedzi A i B (Storage Transfer Service) są do przenoszenia plików z S3/HTTP do GCS, nie do API YouTube. Dlaczego C (Multi-Regional) a nie D (Regional)? Dla "world-wide teams", Multi-Regional (np. US lub EU) zapewnia lepszą dostępność i georedundancję danych.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i B. Storage Transfer Service: Służy do migracji danych z innych chmur (AWS S3) lub on-premise, nie integruje się z YouTube API.
  • D. Regional bucket: Mniej odporny na awarie regionu niż Multi-Regional, co dla globalnego zespołu może być istotne.
