Your startup has a web application that currently serves customers out of a single region in Asia. You are targeting funding that will allow your startup to serve customers globally. Your current goal is to optimize for cost, and your post-funding goal is to optimize for global presence and performance. You must use a native
JDBC driver. What should you do?

  • A. Use Cloud Spanner to configure a single region instance initially, and then configure multi-region Cloud Spanner instances after securing funding. Most Voted
  • B. Use a Cloud SQL for PostgreSQL highly available instance first, and Bigtable with US, Europe, and Asia replication after securing funding.
  • C. Use a Cloud SQL for PostgreSQL zonal instance first, and Bigtable with US, Europe, and Asia after securing funding.
  • D. Use a Cloud SQL for PostgreSQL zonal instance first, and Cloud SQL for PostgreSQL with highly available configuration after securing funding.

Odpowiedź: A.

Cloud Spanner, mimo wyższego kosztu początkowego, jest najlepszym rozwiązaniem w tym scenariuszu ze względu na jego skalowalność globalną i silną spójność danych. Rozpoczęcie od instancji jednoregionowej pozwala na optymalizację kosztów przed pozyskaniem finansowania. Po uzyskaniu środków, łatwo można rozszerzyć instancję Spanner do konfiguracji multi-region, zapewniając globalną obecność i wysoką wydajność, bez konieczności migracji danych czy zmian w kodzie aplikacji (dzięki natywnemu sterownikowi JDBC). Spanner jest zaprojektowany do obsługi transakcji na globalną skalę, co jest kluczowe dla aplikacji obsługujących klientów na całym świecie.  Dodatkowo, Spanner automatycznie zarządza replikacją i failoverem, minimalizując obciążenie operacyjne.

A. Cloud SQL for PostgreSQL, nawet w konfiguracji wysokiej dostępności, nie oferuje tak łatwej skalowalności globalnej jak Spanner.
B. Bigtable jest bazą NoSQL, co może wymagać zmian w architekturze aplikacji i nie jest idealne, jeśli aplikacja wymaga silnej spójności danych.
C. Zonalna instancja Cloud SQL for PostgreSQL oferuje najmniejszą odporność na awarie i nie jest odpowiednia dla aplikacji, która ma się skalować globalnie.
D. Przejście z instancji zonalnej Cloud SQL for PostgreSQL do konfiguracji wysokiej dostępności jest poprawą, ale nadal nie zapewnia globalnej obecności i wydajności, jaką oferuje Spanner.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Odpowiedź A jest najbardziej adekwatna do przedstawionego scenariusza. Kluczowe wymagania to optymalizacja kosztów na początku oraz możliwość skalowania do globalnej obecności i wydajności po pozyskaniu finansowania, przy jednoczesnym użyciu natywnego sterownika JDBC. Cloud Spanner, choć droższy w początkowej fazie, oferuje bezproblemową skalowalność globalną i silną spójność danych, co jest kluczowe dla aplikacji obsługujących klientów na całym świecie. Możliwość rozpoczęcia od instancji jednoregionowej minimalizuje koszty początkowe, a późniejsze przejście na konfigurację multi-region jest proste i nie wymaga migracji danych ani zmian w kodzie aplikacji.

Dyskusja użytkowników jest podzielona, ale nowsze komentarze (np. od TNT87) oraz te z większą liczbą głosów (np. od TNT87, odacir, ducc) popierają odpowiedź A, podkreślając jej skalowalność globalną i zgodność z wymaganiem JDBC. Wielu użytkowników zwraca uwagę na koszt Spanner, ale podkreśla, że pytanie wyraźnie wskazuje na zmianę priorytetów po pozyskaniu finansowania.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B i C: Bigtable nie obsługuje natywnie sterowników JDBC, co wyklucza te opcje ze względu na wymagania zadania.
  • D: Cloud SQL for PostgreSQL, nawet w konfiguracji wysokiej dostępności, nie zapewnia tak łatwej skalowalności globalnej jak Cloud Spanner. Przejście z instancji zonalnej do HA jest poprawą, ale nie spełnia wymogu globalnej obecności i wydajności po pozyskaniu finansowania.

Podsumowując, Cloud Spanner jest najlepszym rozwiązaniem, ponieważ spełnia wszystkie wymagania zadania, oferując optymalizację kosztów na początku i skalowalność globalną po pozyskaniu finansowania, przy jednoczesnym wsparciu dla natywnego sterownika JDBC.
