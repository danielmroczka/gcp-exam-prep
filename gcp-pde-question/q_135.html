You are building a new application that you need to collect data from in a scalable way. Data arrives continuously from the application throughout the day, and you expect to generate approximately 150 GB of JSON data per day by the end of the year. Your requirements are:
✑ Decoupling producer from consumer
✑ Space and cost-efficient storage of the raw ingested data, which is to be stored indefinitely
✑ Near real-time SQL query
✑ Maintain at least 2 years of historical data, which will be queried with SQL
Which pipeline should you use to meet these requirements?

  • A. Create an application that provides an API. Write a tool to poll the API and write data to Cloud Storage as gzipped JSON files.
  • B. Create an application that writes to a Cloud SQL database to store the data. Set up periodic exports of the database to write to Cloud Storage and load into BigQuery.
  • C. Create an application that publishes events to Cloud Pub/Sub, and create Spark jobs on Cloud Dataproc to convert the JSON data to Avro format, stored on HDFS on Persistent Disk.
  • D. Create an application that publishes events to Cloud Pub/Sub, and create a Cloud Dataflow pipeline that transforms the JSON event payloads to Avro, writing the data to Cloud Storage and BigQuery. Most Voted

Odpowiedź: D.

Opcja D jest najlepszym rozwiązaniem, ponieważ spełnia wszystkie wymagania zadania. Cloud Pub/Sub zapewnia decoupling między aplikacją generującą dane a systemem przetwarzającym. Dataflow umożliwia transformację danych JSON do formatu Avro, który jest bardziej efektywny pod względem przestrzeni i kosztów przechowywania niż JSON. Zapis do Cloud Storage zapewnia trwałe, tanie przechowywanie surowych danych, a jednoczesny zapis do BigQuery umożliwia wykonywanie zapytań SQL w czasie zbliżonym do rzeczywistego, spełniając wymaganie near real-time SQL query i przechowywania 2 lat danych historycznych.  Avro jest również schematyzowany, co poprawia niezawodność i wydajność zapytań.

A. Polling API jest nieefektywny i nie zapewnia decoupling, a gzipped JSON nie jest tak efektywny jak Avro.
B. Cloud SQL nie jest skalowalny dla 150 GB dziennie i eksport do BigQuery jest opóźniony, nie spełniając wymagania near real-time.
C. HDFS na Persistent Disk jest droższy i mniej skalowalny niż Cloud Storage, a Spark na Dataproc jest bardziej złożony niż Dataflow do tego zadania.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Odpowiedź D jest zdecydowanie najlepszym rozwiązaniem, spełniającym wszystkie podane wymagania. Kluczowe elementy to:

  • Decoupling producenta od konsumenta: Cloud Pub/Sub zapewnia asynchroniczną komunikację, oddzielając aplikację generującą dane od systemu przetwarzającego.
  • Efektywne przechowywanie danych: Transformacja danych do formatu Avro przed zapisem w Cloud Storage znacząco redukuje przestrzeń dyskową i koszty przechowywania w porównaniu do JSON.
  • Zapytania SQL w czasie zbliżonym do rzeczywistego: Zapis danych do BigQuery umożliwia wykonywanie szybkich zapytań SQL na danych historycznych.
  • Przechowywanie danych historycznych: BigQuery jest idealny do przechowywania i analizy dużych zbiorów danych historycznych (minimum 2 lata).

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość głosów (100%) popiera opcję D, a komentarze takie jak te od barnac1es, juliorevk i wielu innych szczegółowo wyjaśniają, dlaczego jest to optymalne rozwiązanie.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Polling API jest nieefektywny i nie zapewnia decoupling. Gzipped JSON jest mniej efektywny niż Avro.
  • B: Cloud SQL nie jest skalowalny dla tak dużego wolumenu danych (150 GB dziennie). Eksport do BigQuery wprowadza opóźnienie, naruszając wymaganie near real-time.
  • C: HDFS na Persistent Disk jest droższy i mniej skalowalny niż Cloud Storage. Spark na Dataproc jest bardziej złożony niż Dataflow do tego konkretnego zadania.

Należy zauważyć, że forepick zwrócił uwagę na fakt, że format przechowywania powinien być JSON, ale Avro jest bardziej efektywne pod względem przestrzeni i kosztów, co jest ważniejsze w kontekście wymagań zadania.
