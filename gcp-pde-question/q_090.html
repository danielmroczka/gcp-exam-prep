You are deploying MariaDB SQL databases on GCE VM Instances and need to configure monitoring and alerting. You want to collect metrics including network connections, disk IO and replication status from MariaDB with minimal development effort and use StackDriver for dashboards and alerts.
What should you do?

  • A. Install the OpenCensus Agent and create a custom metric collection application with a StackDriver exporter.
  • B. Place the MariaDB instances in an Instance Group with a Health Check.
  • C. Install the StackDriver Logging Agent and configure fluentd in_tail plugin to read MariaDB logs.
  • D. Install the StackDriver Agent and configure the MySQL plugin. Most Voted

Odpowiedź: D

Wyjaśnienie: Wybór D jest najlepszy, ponieważ zapewnia minimalny nakład pracy przy konfiguracji monitoringu i ostrzeżeń dla MariaDB w środowisku GCE VM.  Instalacja OpenCensus Agent pozwala na zbieranie metryk sieciowych, IO dysku i statusu replikacji z jednego miejsca, co ułatwia zarządzanie i monitorowanie.  Użycie StackDriver do tworzenia dashboardów i alertów jest idealne do wizualizacji i automatycznego ostrzegania o potencjalnych problemach.  Zainstalowanie StackDriver Logging Agent pozwala na zbieranie logów, a konfiguracja Fluentd in_tail plugin jest odpowiednia do odczytu i analizy tych logów.  Opcje A, B i C wymagają więcej konfiguracji i złożoności, co z kolei zwiększa czas potrzebny na wdrażanie i utrzymanie systemu.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Install the StackDriver Agent and configure the MySQL plugin (obecnie usługa ta nazywa się Cloud Operations / Cloud Monitoring, a agent to Ops Agent).

Dlaczego?

Wymagania: 1. MariaDB on GCE: MariaDB jest w pełni kompatybilna z MySQL, więc wtyczki (plugins) dedykowane dla MySQL działają również z MariaDB. 2. Collect metrics (network, disk, replication): To są standardowe metryki systemowe i bazodanowe. 3. Minimal development effort: Instalacja gotowego agenta (Stackdriver/Ops Agent) i włączenie wbudowanej wtyczki MySQL to kwestia kilku komend i prostej konfiguracji YAML/TOML. Nie wymaga pisania własnego kodu (jak w opcji A) ani skomplikowanego parsowania logów (jak w opcji C). 4. StackDriver for dashboards: Agent automatycznie wysyła metryki do Cloud Monitoring, gdzie są dostępne natychmiast.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. OpenCensus + custom application: To podejście wymaga dużego nakładu pracy programistycznej (zgodnie z nazwą – "custom metric collection application"). Jest to sprzeczne z wymogiem "minimal development effort".
  • B. Instance Group + Health Check: Health check sprawdza tylko czy proces/port żyje (up/down). Nie zbiera szczegółowych metryk wydajności takich jak "replication status" czy "disk IO".
  • C. Logging Agent + fluentd: Służy do zbierania logów (tekstu), a nie metryk (liczb). Choć można generować metryki z logów (log-based metrics), jest to znacznie trudniejsze i mniej wydajne niż bezpośrednie pobieranie metryk z bazy przez wtyczkę agenta monitorującego.
