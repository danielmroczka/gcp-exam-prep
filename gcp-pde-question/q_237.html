You are planning to load some of your existing on-premises data into BigQuery on Google Cloud. You want to either stream or batch-load data, depending on your use case. Additionally, you want to mask some sensitive data before loading into BigQuery. You need to do this in a programmatic way while keeping costs to a minimum. What should you do?

  • A. Use Cloud Data Fusion to design your pipeline, use the Cloud DLP plug-in to de-identify data within your pipeline, and then move the data into BigQuery.
  • B. Use the BigQuery Data Transfer Service to schedule your migration. After the data is populated in BigQuery, use the connection to the Cloud Data Loss Prevention (Cloud DLP) API to de-identify the necessary data.
  • C. Create your pipeline with Dataflow through the Apache Beam SDK for Python, customizing separate options within your code for streaming, batch processing, and Cloud DLP. Select BigQuery as your data sink. Most Voted
  • D. Set up Datastream to replicate your on-premise data on BigQuery.

Odpowiedź: C.

Dataflow z Apache Beam SDK dla Pythona oferuje największą elastyczność i kontrolę nad całym procesem ładowania danych. Pozwala na programowe zdefiniowanie logiki zarówno dla przetwarzania strumieniowego, jak i wsadowego, a także integrację z Cloud DLP w celu maskowania danych wrażliwych bezpośrednio w potoku. Dzięki temu można zminimalizować koszty, unikając konieczności przetwarzania danych wrażliwych już w BigQuery.  Apache Beam umożliwia również łatwe skalowanie i optymalizację potoku w zależności od potrzeb. Wykorzystanie Dataflow pozwala na pełną kontrolę nad transformacjami danych i integrację z różnymi źródłami i ujściami danych.

A. Cloud Data Fusion jest bardziej wizualnym narzędziem ETL, co może być mniej efektywne kosztowo i mniej elastyczne niż programistyczne podejście Dataflow.
B. BigQuery Data Transfer Service skupia się na transferze danych, a nie na ich transformacji i maskowaniu, co wymagałoby dodatkowego przetwarzania po załadowaniu.
D. Datastream służy do replikacji danych, a nie do ich transformacji i maskowania, co nie spełnia wymagań zadania.

Dodatkowe źródła:

*   **Apache Beam:** https://beam.apache.org/
*   **Google Cloud Dataflow:** https://cloud.google.com/dataflow
*   **Cloud DLP:** https://cloud.google.com/dlp

Sugerowana odpowiedź: C

Odpowiedź C jest najbardziej trafna, ponieważ spełnia wszystkie wymagania zadania. Kluczowe jest tutaj zapotrzebowanie na programatyczne podejście do maskowania danych. Dataflow z Apache Beam SDK dla Pythona umożliwia pełną kontrolę nad potokiem danych poprzez kod, włączając w to integrację z Cloud DLP do maskowania danych wrażliwych. Pozwala to na elastyczne przetwarzanie strumieniowe i wsadowe, a także optymalizację kosztów dzięki skalowalności Dataflow.

Dyskusja użytkowników w większości potwierdza poprawność odpowiedzi C. Użytkownicy zwracają uwagę na elastyczność, wsparcie dla przetwarzania strumieniowego i wsadowego oraz integrację z Cloud DLP. Nowsze komentarze, takie jak te od chrissamharris, podkreślają, że Data Fusion (odpowiedź A) jest rozwiązaniem "low-code", co stoi w sprzeczności z wymaganiem "programatycznego" podejścia.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Cloud Data Fusion jest narzędziem ETL opartego na interfejsie graficznym (low-code), a nie programistycznym.
  • B: BigQuery Data Transfer Service służy do transferu danych, a nie do ich transformacji i maskowania. Maskowanie danych po załadowaniu do BigQuery jest mniej efektywne kosztowo.
  • D: Datastream służy do replikacji danych, a nie do ich transformacji i maskowania.

Podsumowując, Dataflow z Apache Beam SDK dla Pythona oferuje najlepsze połączenie elastyczności, kontroli, efektywności kosztowej i możliwości programatycznego maskowania danych, co czyni go idealnym rozwiązaniem dla tego scenariusza.
