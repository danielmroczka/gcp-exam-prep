You are monitoring your organization’s data lake hosted on BigQuery. The ingestion pipelines read data from Pub/Sub and write the data into tables on BigQuery. After a new version of the ingestion pipelines is deployed, the daily stored data increased by 50%. The volumes of data in Pub/Sub remained the same and only some tables had their daily partition data size doubled. You need to investigate and fix the cause of the data increase. What should you do?

  • A. 1. Check for duplicate rows in the BigQuery tables that have the daily partition data size doubled.
    2. Schedule daily SQL jobs to deduplicate the affected tables.
    3. Share the deduplication script with the other operational teams to reuse if this occurs to other tables.
  • B. 1. Check for code errors in the deployed pipelines.
    2. Check for multiple writing to pipeline BigQuery sink.
    3. Check for errors in Cloud Logging during the day of the release of the new pipelines.
    4. If no errors, restore the BigQuery tables to their content before the last release by using time travel.
  • C. 1. Check for duplicate rows in the BigQuery tables that have the daily partition data size doubled.
    2. Check the BigQuery Audit logs to find job IDs.
    3. Use Cloud Monitoring to determine when the identified Dataflow jobs started and the pipeline code version.
    4. When more than one pipeline ingests data into a table, stop all versions except the latest one. Most Voted
  • D. 1. Roll back the last deployment.
    2. Restore the BigQuery tables to their content before the last release by using time travel.
    3. Restart the Dataflow jobs and replay the messages by seeking the subscription to the timestamp of the release.

Odpowiedź: C.

Opcja C jest najbardziej kompleksowa i ukierunkowana na znalezienie przyczyny problemu oraz jego naprawę w sposób kontrolowany. Sprawdzenie duplikatów w tabelach, które doświadczyły wzrostu, jest logicznym pierwszym krokiem. Następnie analiza logów audytowych BigQuery pozwoli zidentyfikować konkretne zadania, które mogły spowodować problem. Powiązanie tych zadań z Cloud Monitoring i wersją kodu pipeline'u da wgląd w to, co mogło pójść nie tak. Wreszcie, jeśli okaże się, że wiele pipeline'ów zapisuje dane do tej samej tabeli, zatrzymanie starszych wersji zapobiegnie dalszym problemom. To podejście pozwala na dokładną diagnostykę i naprawę bez od razu sięgania po drastyczne środki, takie jak rollback.

A. Samo deduplikowanie to leczenie objawów, a nie znalezienie przyczyny problemu.
B. Chociaż sprawdzanie kodu i logów jest ważne, brak uwzględnienia logów audytowych BigQuery i monitoringu Dataflow sprawia, że to podejście jest niepełne.
D. Rollback i replay to drastyczne kroki, które powinny być rozważane dopiero po dokładnej analizie i wykluczeniu innych możliwości.

Sugerowana odpowiedź: C

Odpowiedź C jest najbardziej kompleksowa i adekwatna do postawionego problemu. Pytanie wymaga zbadania i naprawy przyczyny wzrostu danych, a nie tylko doraźnego rozwiązania problemu (np. deduplikacji). Odpowiedź C systematycznie prowadzi przez proces diagnostyczny:

  • Sprawdzenie duplikatów w tabelach, które doświadczyły wzrostu, jest dobrym pierwszym krokiem, aby szybko zidentyfikować potencjalny problem.
  • Analiza logów audytowych BigQuery pozwala zidentyfikować konkretne zadania, które mogły spowodować wzrost danych.
  • Korelacja z Cloud Monitoring i wersją kodu pipeline'u pozwala ustalić, czy konkretna wersja pipeline'u jest odpowiedzialna za problem.
  • Zarządzanie wieloma wersjami pipeline'ów zapobiega dalszym problemom, jeśli okaże się, że starsze wersje nadal zapisują dane.

Ta odpowiedź skupia się na znalezieniu źródła problemu, a nie tylko na jego objawach.

Dlaczego inne odpowiedzi są mniej odpowiednie:

  • A: Samo deduplikowanie jest leczeniem objawów, a nie przyczyną. Nie rozwiązuje problemu, tylko go maskuje.
  • B: Chociaż sprawdzanie kodu i logów jest ważne, pomija kluczowy element – logi audytowe BigQuery, które mogą precyzyjnie wskazać, które zadania spowodowały wzrost danych.
  • D: Rollback i replay to drastyczne kroki, które powinny być rozważane dopiero po dokładnej analizie i wykluczeniu innych możliwości. Są to rozwiązania awaryjne, a nie standardowa procedura diagnostyczna.
