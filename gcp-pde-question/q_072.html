You are designing storage for 20 TB of text files as part of deploying a data pipeline on Google Cloud. Your input data is in CSV format. You want to minimize the cost of querying aggregate values for multiple users who will query the data in Cloud Storage with multiple engines. Which storage service and schema design should you use?

  • A. Use Cloud Bigtable for storage. Install the HBase shell on a Compute Engine instance to query the Cloud Bigtable data.
  • B. Use Cloud Bigtable for storage. Link as permanent tables in BigQuery for query.
  • C. Use Cloud Storage for storage. Link as permanent tables in BigQuery for query. Most Voted
  • D. Use Cloud Storage for storage. Link as temporary tables in BigQuery for query.

Odpowiedź: B

Wyjaśnienie: Wybór Cloud Bigtable dla przechowywania danych o 20 TB jest optymalny, ponieważ zapewnia odpowiednią skalowalność i wydajność dla dużych zbiorów danych. Bigtable jest zaprojektowany do obsługi dużych ilości danych i oferuje szybkie zapytania, co jest kluczowe dla aplikacji przetwarzania danych w czasie rzeczywistym.  Użycie BigQuery dla linków permanentnych pozwala na efektywne wykonywanie zapytań na danych przechowywanych w Bigtable, co jest szczególnie przydatne przy analizie danych w dużych zbiorach.  W porównaniu do innych opcji, Cloud Storage jest mniej odpowiedni dla dużych zbiorów danych i nie oferuje możliwości szybkiego wykonywania zapytań, co czyni Cloud Bigtable lepszym wyborem dla tego scenariusza.
Link do dodatkowych żródeł: https://cloud.google.com/bigtable/docs/

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Use Cloud Storage for storage. Link as permanent tables in BigQuery for query..

Dlaczego?

Wymagania: 1. 20 TB of text files (CSV): Duży wolumen danych niestrukturalnych/półstrukturalnych. 2. Minimize cost: To klucz. Cloud Storage (GCS) jest najtańszym magazynem dla plików ("Coldline" lub "Standard" storage). BigQuery External Tables pozwalają odpytywać dane bezpośrednio w GCS bez ich importowania (co oszczędza koszty storage w BQ, choć zapytania mogą być wolniejsze i droższe niż na danych natywnych, ale w kontekście pytania o "minimize storage cost of text files" i okazjonalne query, to dobre podejście. Ważniejsze jest to, że Bigtable (opcje A i B) jest BARDZO drogim rozwiązaniem (wymaga stałych węzłów)). 3. Query with multiple engines: Pliki leżące w GCS mogą być czytane przez BigQuery, Dataproc, Dataflow itp. 4. Permanent tables: Pozwalają zdefiniować schemat raz i udostępnić go użytkownikom ("schema design").

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i B. Cloud Bigtable: Bigtable jest drogie (płacisz za węzły 24/7, ok. $0.65/node/hour). Przechowywanie 20 TB w Bigtable w celu "minimalizacji kosztów" dla plików CSV, które można trzymać w GCS za grosze, jest błędem architektonicznym. Bigtable służy do niskich opóźnień (ms), a nie jako tani magazyn plików.
  • D. Temporary tables: Tabele tymczasowe znikają. Jeśli "multiple users" mają korzystać z danych, musieliby za każdym razem definiować tabelę na nowo. Permanent External Table jest właściwym obiektem w BQ.
