You want to rebuild your batch pipeline for structured data on Google Cloud. You are using PySpark to conduct data transformations at scale, but your pipelines are taking over twelve hours to run. To expedite development and pipeline run time, you want to use a serverless tool and SOL syntax. You have already moved your raw data into Cloud Storage. How should you build the pipeline on Google Cloud while meeting speed and processing requirements?

  • A. Convert your PySpark commands into SparkSQL queries to transform the data, and then run your pipeline on Dataproc to write the data into BigQuery.
  • B. Ingest your data into Cloud SQL, convert your PySpark commands into SparkSQL queries to transform the data, and then use federated quenes from BigQuery for machine learning.
  • C. Ingest your data into BigQuery from Cloud Storage, convert your PySpark commands into BigQuery SQL queries to transform the data, and then write the transformations to a new table. Most Voted
  • D. Use Apache Beam Python SDK to build the transformation pipelines, and write the data into BigQuery.

Odpowiedź: C.

Opcja C jest najbardziej odpowiednia, ponieważ wykorzystuje BigQuery do transformacji danych bezpośrednio po załadowaniu ich z Cloud Storage. BigQuery jest serwerless, co eliminuje potrzebę zarządzania infrastrukturą, a jego SQL syntax jest zgodny z wymaganiami zadania. Przekształcenie istniejących transformacji PySpark na zapytania BigQuery SQL pozwoli na wykorzystanie mocy obliczeniowej BigQuery do szybkiego przetwarzania danych. BigQuery jest zoptymalizowany do analizy dużych zbiorów danych, co powinno znacznie skrócić czas wykonania pipeline w porównaniu do 12 godzin. Dodatkowo, zapisywanie transformacji do nowej tabeli w BigQuery zapewnia jasny podział danych i ułatwia dalszą analizę.

A. Dataproc wymaga zarządzania klastrem, co nie jest serwerless i nie spełnia wymogu przyspieszenia rozwoju.
B. Cloud SQL nie jest optymalny do przetwarzania dużych zbiorów danych w porównaniu do BigQuery, a federated queries mogą wprowadzać dodatkowe opóźnienia.
D. Apache Beam, choć potężne, wymaga więcej konfiguracji i nie jest tak bezpośrednio związane z wykorzystaniem SQL do transformacji, jak BigQuery.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Najbardziej odpowiednią odpowiedzią jest C. Pytanie wyraźnie wskazuje na potrzebę rozwiązania serverless oraz wykorzystania składni SQL. BigQuery idealnie spełnia te kryteria. Bezpośrednie załadowanie danych z Cloud Storage do BigQuery i transformacja ich za pomocą zapytań SQL jest najszybszym i najbardziej efektywnym sposobem na przyspieszenie pipeline'u. Przekształcenie istniejącego kodu PySpark na zapytania BigQuery SQL jest rozsądnym krokiem, a BigQuery jest zoptymalizowany do przetwarzania dużych zbiorów danych.

Dyskusja użytkowników potwierdza tę odpowiedź. Wiele osób wskazuje na kluczowe wymaganie "serverless" jako główny powód wyboru BigQuery. Użytkownik AWSandeep początkowo sugerował odpowiedź A, ale później zmienił ją na C, zauważając, że pytanie odnosi się do standardowego Dataproc, a nie Dataproc Serverless. Użytkownik MoeHaydar również podkreśla, że standardowy Dataproc nie jest rozwiązaniem serverless.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Dataproc wymaga zarządzania klastrem, co nie spełnia wymogu "serverless".
  • B: Cloud SQL nie jest optymalny do przetwarzania dużych zbiorów danych, a federated queries mogą wprowadzać dodatkowe opóźnienia.
  • D: Apache Beam, choć potężne, wymaga więcej konfiguracji i nie jest tak bezpośrednio związane z wykorzystaniem SQL do transformacji, jak BigQuery.

Podsumowując, BigQuery oferuje najprostsze, najszybsze i najbardziej skalowalne rozwiązanie spełniające wszystkie wymagania zadania.
