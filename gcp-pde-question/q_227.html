You stream order data by using a Dataflow pipeline, and write the aggregated result to Memorystore. You provisioned a Memorystore for Redis instance with Basic Tier, 4 GB capacity, which is used by 40 clients for read-only access. You are expecting the number of read-only clients to increase significantly to a few hundred and you need to be able to support the demand. You want to ensure that read and write access availability is not impacted, and any changes you make can be deployed quickly. What should you do?

  • A. Create a new Memorystore for Redis instance with Standard Tier. Set capacity to 4 GB and read replica to No read replicas (high availability only). Delete the old instance.
  • B. Create a new Memorystore for Redis instance with Standard Tier. Set capacity to 5 GB and create multiple read replicas. Delete the old instance. Most Voted
  • C. Create a new Memorystore for Memcached instance. Set a minimum of three nodes, and memory per node to 4 GB. Modify the Dataflow pipeline and all clients to use the Memcached instance. Delete the old instance.
  • D. Create multiple new Memorystore for Redis instances with Basic Tier (4 GB capacity). Modify the Dataflow pipeline and new clients to use all instances.

Odpowiedź: B.

Wybrana opcja (B) jest najlepsza, ponieważ pozwala na skalowanie Memorystore for Redis w sposób, który odpowiada na rosnące zapotrzebowanie na odczyty, jednocześnie minimalizując wpływ na dostępność i umożliwiając szybkie wdrożenie. Przejście na Standard Tier zapewnia lepszą wydajność i niezawodność niż Basic Tier. Zwiększenie pojemności do 5 GB daje pewien margines bezpieczeństwa. Kluczowe jest jednak utworzenie wielu read replicas, które rozkładają obciążenie odczytów na kilka instancji, co pozwala obsłużyć znacznie większą liczbę klientów bez obciążania głównej instancji. Usunięcie starej instancji po weryfikacji nowej zapewnia płynne przejście.

A. Basic Tier nie jest wystarczający dla dużej liczby klientów, a brak read replicas oznacza, że główna instancja będzie przeciążona.
C. Przejście na Memcached wymaga znaczących zmian w kodzie Dataflow i klientów, co jest czasochłonne i ryzykowne.
D. Użycie wielu instancji Basic Tier nie zapewnia centralnego punktu zarządzania i może prowadzić do problemów z synchronizacją danych i złożonością zarządzania.

Sugerowana odpowiedź: B

Odpowiedź B jest prawidłowa, ponieważ najlepiej adresuje problem skalowalności przy jednoczesnym zachowaniu wysokiej dostępności i minimalizującym wpływ na istniejący kod. Zwiększenie liczby klientów odczytujących dane wymaga skalowania możliwości odczytu Memorystore for Redis. Przejście na Standard Tier zapewnia wyższą wydajność i niezawodność niż Basic Tier. Kluczowym elementem jest jednak dodanie read replicas. Read replicas rozkładają obciążenie odczytów na wiele instancji, co pozwala obsłużyć znacznie większą liczbę klientów bez przeciążania głównej instancji. Zwiększenie pojemności do 5 GB daje dodatkowy margines bezpieczeństwa.

Komentarze użytkowników, szczególnie te nowsze (SamuelTsch, e70ea9e) potwierdzają, że odpowiedź B jest najbardziej sensowna ze względu na wykorzystanie read replicas. Użytkownik e70ea9e trafnie zauważa, że read replicas pozwalają na skalowanie bez konieczności dużych zmian w kodzie klienta.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Basic Tier nie jest wystarczający dla dużej liczby klientów, a brak read replicas oznacza, że główna instancja będzie przeciążona.
  • C: Przejście na Memcached wymagałoby znaczących zmian w kodzie Dataflow i klientów, co jest czasochłonne i ryzykowne. Ponadto, pytanie dotyczy skalowania istniejącego rozwiązania Redis, a nie migracji do innej technologii.
  • D: Użycie wielu instancji Basic Tier nie zapewnia centralnego punktu zarządzania i może prowadzić do problemów z synchronizacją danych i złożonością zarządzania. Ponadto, Basic Tier nadal nie jest wystarczający dla tak dużej liczby klientów.

Podsumowując, odpowiedź B jest najbardziej efektywnym i praktycznym rozwiązaniem, które spełnia wszystkie wymagania zadania.
