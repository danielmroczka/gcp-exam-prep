You have a data pipeline that writes data to Cloud Bigtable using well-designed row keys. You want to monitor your pipeline to determine when to increase the size of your Cloud Bigtable cluster. Which two actions can you take to accomplish this? (Choose two.)

  • A. Review Key Visualizer metrics. Increase the size of the Cloud Bigtable cluster when the Read pressure index is above 100.
  • B. Review Key Visualizer metrics. Increase the size of the Cloud Bigtable cluster when the Write pressure index is above 100.
  • C. Monitor the latency of write operations. Increase the size of the Cloud Bigtable cluster when there is a sustained increase in write latency. Most Voted
  • D. Monitor storage utilization. Increase the size of the Cloud Bigtable cluster when utilization increases above 70% of max capacity. Most Voted
  • E. Monitor latency of read operations. Increase the size of the Cloud Bigtable cluster of read operations take longer than 100 ms.

Odpowiedź: B

Wyjaśnienie: Monitorowanie Cloud Bigtable, aby określić, kiedy zwiększyć rozmiar klastera, wymaga analizy zarówno wpływu na czytanie, jak i pisanie danych.  Zwiększenie rozmiaru klastera, gdy wskaźniki czytania są powyżej 100, jest kluczowe dla utrzymania wydajności, ponieważ zwiększa przepustowość i zmniejsza opóźnienia. Z kolei, zwiększenie rozmiaru klastera, gdy wskaźniki pisania są powyżej 100, jest niezbędne do obsługi dużych ilości danych i zapewnienia odpowiedniej przepustowości.  Monitorowanie latencji operacji czytania jest ważne, ale nie jest bezpośrednio związane z procesem zwiększania rozmiaru klastera.  Monitorowanie wykorzystania przestrzeni dyskowej, choć istotne, nie jest bezpośrednio związane z procesem optymalizacji rozmiaru klastera.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C, D

Prawidłowe odpowiedzi to C, D.

Dlaczego?

W Bigtable skalowanie klastra (dodawanie węzłów) jest zalecane w dwóch głównych sytuacjach:

  1. C. Monitor write latency: Jeśli latencja zapisu (lub odczytu) stale rośnie, oznacza to, że obecna liczba węzłów nie nadąża z przetwarzaniem obciążenia (CPU/IO bound). Zwiększenie klastra rozkłada obciążenie i obniża latencję.
  2. D. Monitor storage utilization: Każdy węzeł Bigtable ma ograniczoną pojemność dyskową (np. 2.5 TB dla SSD lub 16 TB dla HDD). Google rekomenduje utrzymywanie zużycia dysku poniżej 70%, aby zapewnić stabilność procesów systemowych (jak kompresja/compaction). Jeśli przekroczysz 70%, musisz dodać węzły, aby zwiększyć dostępną przestrzeń, nawet jeśli CPU jest mało obciążone.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i B. Key Visualizer (Pressure Index > 100): Key Visualizer służy do diagnozowania źle zaprojektowanych kluczy (hotspotów). Jeśli masz hotspot, dodanie węzłów niewiele pomoże (pressure nadal będzie na jednym węźle). Należy wtedy poprawić schemat, a nie skalować klaster.
  • E. Read latency > 100 ms: Pytanie dotyczy potoku, który pisze dane ("pipeline that writes data"). Choć latencja odczytu jest ważna, w kontekście tego konkretnego pipeline'u kluczowa jest latencja zapisu (C). Ponadto próg 100ms jest arbitralny – Bigtable normalnie oferuje <10ms.
