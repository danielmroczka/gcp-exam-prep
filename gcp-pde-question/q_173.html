You are designing a pipeline that publishes application events to a Pub/Sub topic. Although message ordering is not important, you need to be able to aggregate events across disjoint hourly intervals before loading the results to BigQuery for analysis. What technology should you use to process and load this data to
BigQuery while ensuring that it will scale with large volumes of events?

  • A. Create a Cloud Function to perform the necessary data processing that executes using the Pub/Sub trigger every time a new message is published to the topic.
  • B. Schedule a Cloud Function to run hourly, pulling all available messages from the Pub/Sub topic and performing the necessary aggregations.
  • C. Schedule a batch Dataflow job to run hourly, pulling all available messages from the Pub/Sub topic and performing the necessary aggregations.
  • D. Create a streaming Dataflow job that reads continually from the Pub/Sub topic and performs the necessary aggregations using tumbling windows. Most Voted

Odpowiedź: D.

Streaming Dataflow z tumbling windows jest najlepszym rozwiązaniem, ponieważ pozwala na ciągłe przetwarzanie danych z Pub/Sub i agregację ich w określonych, nie nakładających się na siebie przedziałach czasowych (godzinnych w tym przypadku). Tumbling windows zapewniają, że dane z każdego przedziału godzinowego są przetwarzane oddzielnie, co jest kluczowe dla agregacji. Dataflow jest zaprojektowany do skalowania w celu obsługi dużych wolumenów danych, co jest wymagane w scenariuszu. Ponadto, streaming Dataflow minimalizuje opóźnienia w przetwarzaniu, co jest korzystne, nawet jeśli kolejność wiadomości nie jest krytyczna.

A. Cloud Functions uruchamiane na triggerze Pub/Sub nie są skalowalne dla dużych wolumenów danych i mogą prowadzić do ograniczeń przepustowości.
B. Planowanie Cloud Function do uruchamiania co godzinę może powodować opóźnienia w przetwarzaniu i nie jest tak efektywne jak ciągłe przetwarzanie strumieniowe.
C. Batch Dataflow job uruchamiany co godzinę jest mniej efektywny niż streaming Dataflow, ponieważ wymaga zebrania wszystkich wiadomości z godziny przed rozpoczęciem przetwarzania, co wprowadza opóźnienia.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Odpowiedź D, czyli stworzenie strumieniowego zadania Dataflow z wykorzystaniem tumbling windows, jest najbardziej odpowiednim rozwiązaniem dla tego problemu. Kluczowe wymagania to skalowalność przy dużych wolumenach danych oraz agregacja zdarzeń w godzinnych interwałach. Strumieniowe przetwarzanie danych w Dataflow pozwala na ciągłe przetwarzanie danych z Pub/Sub, a tumbling windows zapewniają agregację danych w stałych, nie nakładających się na siebie przedziałach czasowych (w tym przypadku godzinowych).

Dlaczego odpowiedź D jest poprawna:

  • Skalowalność: Dataflow jest zaprojektowany do obsługi dużych wolumenów danych, co jest kluczowe w tym scenariuszu.
  • Agregacja w interwałach czasowych: Tumbling windows idealnie nadają się do agregacji danych w stałych, nie nakładających się na siebie przedziałach czasowych.
  • Minimalizacja opóźnień: Strumieniowe przetwarzanie minimalizuje opóźnienia, co jest korzystne, nawet jeśli kolejność wiadomości nie jest krytyczna.

Analiza innych odpowiedzi:

  • A: Cloud Function z triggerem Pub/Sub: Cloud Functions nie są skalowalne dla dużych wolumenów danych i mogą prowadzić do ograniczeń przepustowości.
  • B: Planowane Cloud Function: Uruchamianie Cloud Function co godzinę wprowadza opóźnienia i nie jest tak efektywne jak ciągłe przetwarzanie strumieniowe.
  • C: Batch Dataflow job: Chociaż teoretycznie możliwe, uruchamianie zadania wsadowego co godzinę jest mniej efektywne niż strumieniowe przetwarzanie. Dodatkowo, jak zauważyli użytkownicy (baimus), Pub/Sub ma ograniczenia dotyczące wielkości pojedynczej partii wiadomości (10MB lub 1000 wiadomości), co może uniemożliwić pobranie wszystkich wiadomości w jednym zadaniu wsadowym.

Komentarze użytkowników, szczególnie te nowsze (np. baimus), potwierdzają, że ograniczenia Pub/Sub dotyczące wielkości partii wiadomości sprawiają, że strumieniowe przetwarzanie jest bardziej niezawodne i skalowalne.
