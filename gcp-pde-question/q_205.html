You have a data processing application that runs on Google Kubernetes Engine (GKE). Containers need to be launched with their latest available configurations from a container registry. Your GKE nodes need to have GPUs, local SSDs, and 8 Gbps bandwidth. You want to efficiently provision the data processing infrastructure and manage the deployment process. What should you do?

  • A. Use Compute Engine startup scripts to pull container images, and use gcloud commands to provision the infrastructure.
  • B. Use Cloud Build to schedule a job using Terraform build to provision the infrastructure and launch with the most current container images. Most Voted
  • C. Use GKE to autoscale containers, and use gcloud commands to provision the infrastructure.
  • D. Use Dataflow to provision the data pipeline, and use Cloud Scheduler to run the job.

Odpowiedź: B.

Użycie Cloud Build do uruchomienia zadania Terraform jest najbardziej efektywnym rozwiązaniem, ponieważ Terraform pozwala na zdefiniowanie infrastruktury jako kodu (IaC), co umożliwia automatyczne i powtarzalne provisioning zasobów, w tym GKE z odpowiednimi specyfikacjami (GPU, SSD, przepustowość). Cloud Build automatyzuje proces budowania, testowania i wdrażania, w tym pobieranie najnowszych obrazów kontenerów z rejestru. Połączenie tych dwóch narzędzi zapewnia spójne i skalowalne wdrożenie infrastruktury i aplikacji. Terraform może być zaplanowany do regularnego uruchamiania, zapewniając, że infrastruktura jest zawsze aktualna i zgodna z definicją. To podejście minimalizuje ręczną interwencję i ryzyko błędów.

A. Użycie skryptów startupowych Compute Engine jest mniej efektywne i trudniejsze w utrzymaniu niż IaC, a ręczne używanie `gcloud` jest podatne na błędy.
C. GKE autoscale zarządza skalowaniem, ale nie rozwiązuje problemu provisioning infrastruktury z konkretnymi wymaganiami sprzętowymi.
D. Dataflow jest przeznaczone do przetwarzania strumieniowego i wsadowego danych, a nie do provisioning infrastruktury.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Odpowiedź B jest najbardziej adekwatna do przedstawionego problemu. Pytanie dotyczy efektywnego provisioning infrastruktury GKE z konkretnymi wymaganiami (GPU, SSD, przepustowość) oraz wdrażania kontenerów z najnowszymi konfiguracjami. Połączenie Cloud Build i Terraform idealnie odpowiada na te potrzeby.

  • Terraform umożliwia definiowanie infrastruktury jako kodu (IaC), co zapewnia automatyzację, powtarzalność i kontrolę wersji. Można w nim zdefiniować zasoby GKE z wymaganymi specyfikacjami sprzętowymi.
  • Cloud Build automatyzuje proces budowania, testowania i wdrażania. W tym przypadku, może uruchamiać zadania Terraform w celu provisioning infrastruktury i pobierania najnowszych obrazów kontenerów z rejestru.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość głosów (100% w podziale głosów społeczności) wskazuje na odpowiedź B. Użytkownicy, tacy jak MaxNRG, szczegółowo wyjaśniają korzyści płynące z użycia Cloud Build i Terraform w kontekście tego zadania. Nawet komentarz spicebits, choć wyraża pewne zastrzeżenia co do szczegółów implementacji (Terraform Apply vs. Terraform build), ostatecznie zgadza się, że jest to najlepsza opcja spośród dostępnych.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Użycie skryptów startupowych Compute Engine jest mniej efektywne i trudniejsze w utrzymaniu niż IaC. Ręczne używanie `gcloud` jest podatne na błędy i nie zapewnia automatyzacji.
  • C: GKE autoscale zarządza skalowaniem, ale nie rozwiązuje problemu provisioning infrastruktury z konkretnymi wymaganiami sprzętowymi.
  • D: Dataflow jest przeznaczone do przetwarzania strumieniowego i wsadowego danych, a nie do provisioning infrastruktury.

Należy zauważyć, że początkowo Atnafu wskazywał odpowiedź C jako poprawną, ale później ją poprawił na B, co świadczy o zrozumieniu problemu i prawidłowym doborze rozwiązania.
