You have a Standard Tier Memorystore for Redis instance deployed in a production environment. You need to simulate a Redis instance failover in the most accurate disaster recovery situation, and ensure that the failover has no impact on production data. What should you do?

  • A. Create a Standard Tier Memorystore for Redis instance in the development environment. Initiate a manual failover by using the limited-data-loss data protection mode.
  • B. Create a Standard Tier Memorystore for Redis instance in a development environment. Initiate a manual failover by using the force-data-loss data protection mode. Most Voted
  • C. Increase one replica to Redis instance in production environment. Initiate a manual failover by using the force-data-loss data protection mode.
  • D. Initiate a manual failover by using the limited-data-loss data protection mode to the Memorystore for Redis instance in the production environment.

Odpowiedź: B.

Aby symulować failover Redis w najbardziej realistyczny sposób, bez wpływu na produkcyjne dane, należy przeprowadzić test w środowisku developerskim. Użycie trybu `force-data-loss` podczas failoveru w środowisku developerskim pozwala na szybkie przetestowanie procedury bez ryzyka utraty danych produkcyjnych. Ten tryb symuluje sytuację, w której dane z głównego węzła są nieodzyskiwalne, co pozwala sprawdzić, jak aplikacja radzi sobie z taką sytuacją.  Testowanie w środowisku produkcyjnym jest ryzykowne, a zwiększanie replik w środowisku produkcyjnym nie symuluje pełnego failoveru.  Stworzenie instancji w środowisku developerskim pozwala na bezpieczne eksperymentowanie z różnymi scenariuszami failoveru.

A. Testowanie failoveru w środowisku produkcyjnym, nawet z trybem `limited-data-loss`, jest zbyt ryzykowne.
C. Zwiększenie replik nie symuluje pełnego failoveru i nie testuje procedur odzyskiwania po awarii.
D. Inicjowanie failoveru w środowisku produkcyjnym, nawet z `limited-data-loss`, jest niebezpieczne i może wpłynąć na dostępność aplikacji.

Sugerowana odpowiedź: B

Najlepszym rozwiązaniem jest stworzenie instancji Memorystore for Redis w środowisku developerskim i zainicjowanie manualnego failoveru z użyciem trybu force-data-loss. Kluczowe jest, aby test failoveru odbywał się poza środowiskiem produkcyjnym, aby uniknąć wpływu na rzeczywiste dane. Tryb force-data-loss symuluje pełny failover, co jest najbardziej dokładnym testem procedur odzyskiwania po awarii. Umożliwia to sprawdzenie, jak aplikacja radzi sobie z sytuacją, w której dane z głównego węzła są nieodzyskiwalne.

Dyskusja użytkowników potwierdza tę odpowiedź. Użytkownicy MaxNRG, SamuelTsch, datapassionate i Matt_108 wskazują, że opcja B jest najlepsza, ponieważ izoluje test od środowiska produkcyjnego i wykorzystuje najbardziej rygorystyczny tryb failoveru. Użytkownik MaxNRG w późniejszym komentarzu podkreśla, że opcje C i D naruszają warunek "no impact on production data".

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Testowanie failoveru w środowisku produkcyjnym, nawet z trybem limited-data-loss, jest zbyt ryzykowne.
  • C: Zwiększenie liczby replik w środowisku produkcyjnym nie symuluje pełnego failoveru i nie testuje procedur odzyskiwania po awarii.
  • D: Inicjowanie failoveru w środowisku produkcyjnym, nawet z limited-data-loss, jest niebezpieczne i może wpłynąć na dostępność aplikacji.

Użytkownik desertlotus1211 sugeruje, że odpowiedź A jest lepsza, ale argumentuje to w oparciu o błędne założenie, że limited-data-loss jest rekomendowanym trybem symulacji failoveru. W rzeczywistości, dla celów testowania procedur odzyskiwania po awarii, force-data-loss jest bardziej odpowiedni, ponieważ symuluje bardziej ekstremalny scenariusz.
