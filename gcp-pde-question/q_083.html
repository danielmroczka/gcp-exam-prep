Flowlogistic Case Study -

Company Overview -
Flowlogistic is a leading logistics and supply chain provider. They help businesses throughout the world manage their resources and transport them to their final destination. The company has grown rapidly, expanding their offerings to include rail, truck, aircraft, and oceanic shipping.

Company Background -
The company started as a regional trucking company, and then expanded into other logistics market. Because they have not updated their infrastructure, managing and tracking orders and shipments has become a bottleneck. To improve operations, Flowlogistic developed proprietary technology for tracking shipments in real time at the parcel level. However, they are unable to deploy it because their technology stack, based on Apache Kafka, cannot support the processing volume. In addition, Flowlogistic wants to further analyze their orders and shipments to determine how best to deploy their resources.

Solution Concept -
Flowlogistic wants to implement two concepts using the cloud:
✑ Use their proprietary technology in a real-time inventory-tracking system that indicates the location of their loads
✑ Perform analytics on all their orders and shipment logs, which contain both structured and unstructured data, to determine how best to deploy resources, which markets to expand info. They also want to use predictive analytics to learn earlier when a shipment will be delayed.

Okay, let's analyze this question and provide a detailed response.

**Answer:** A

**Explanation:**

Flowlogistic’s situation is a classic case of a data lake bottleneck, exacerbated by an outdated infrastructure. The core problem isn’t just the lack of a real-time tracking system, but the inability to process the massive volume of data generated by their global operations.  Let’s break down why the other options are less suitable:

*   **B. Cloud Pub/Sub, Cloud Dataflow, and Local SSD:** While Pub/Sub is useful for event streaming, it’s not the primary solution for processing the sheer volume of data. Local SSD is a storage solution, not a processing engine.  It’s a component, not the core of the solution.
*   **C. Cloud Pub/Sub, Cloud SQL, and Cloud Storage:** Cloud SQL is a database, not a data processing engine. Cloud Storage is for object storage, not for real-time analytics or data transformation.
*   **D. Cloud Load Balancing, Cloud Dataflow, and Cloud Storage:**  Load balancing is for distributing traffic, not for processing data. Dataflow is a processing engine, and storage is for data persistence.
*   **E. Cloud Dataflow, Cloud SQL, and Cloud Storage:**  This option is a bit too broad. While Dataflow is a powerful tool, it's not the *only* solution for this problem.  The need for real-time processing and a centralized data lake is the key driver.

**Why A is the Best Choice:**

Cloud Pub/Sub is the ideal choice because it’s designed for real-time data ingestion and streaming. It can handle the high volume of data from global sources, and it can be easily integrated with the proprietary tracking software.  It’s the foundation for the entire data lake architecture, enabling the analytics and predictive modeling needed to optimize operations.  It’s a critical component for feeding the data to the tracking software and for the overall data governance.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Prawidłową odpowiedzią jest A. Cloud Pub/Sub, Cloud Dataflow, and Cloud Storage.

Dlaczego?

Analiza przypadku Flowlogistic wskazuje na następujące potrzeby: 1. Ingest data from global sources: Potrzebny jest globalny, skalowalny system kolejek. Cloud Pub/Sub idealnie zastępuje przeciążoną Kafkę. 2. Process and query in real-time: Cloud Dataflow (Apache Beam) to natywne narzędzie GCP do przetwarzania strumieniowego (streaming) w czasie rzeczywistym. 3. Store the data reliably / Centralized Data Lake: Wymagania biznesowe Flowlogistic mówią o "centralized Data Lake for analysis". Cloud Storage (GCS) jest fundamentem każdego Data Lake w Google Cloud ze względu na trwałość (11 dziewiątek), skalowalność i niską cenę.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • B. Local SSD: Jest to magazyn tymczasowy (ephemeral). Dane zostaną utracone po wyłączeniu maszyny. Nie spełnia wymogu "store the data reliably".
  • C & E. Cloud SQL: Bazy relacyjne (OLTP) nie są przeznaczone do przyjmowania ogromnych wolumenów surowych danych IoT w czasie rzeczywistym (problem skalowalności zapisu).
  • D. Cloud Load Balancing: To usługa sieciowa do rozkładania ruchu HTTP/TCP, nie służy do ingestu i buforowania wiadomości jak Pub/Sub.
