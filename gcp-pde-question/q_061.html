Your analytics team wants to build a simple statistical model to determine which customers are most likely to work with your company again, based on a few different metrics. They want to run the model on Apache Spark, using data housed in Google Cloud Storage, and you have recommended using Google Cloud
Dataproc to execute this job. Testing has shown that this workload can run in approximately 30 minutes on a 15-node cluster, outputting the results into Google
BigQuery. The plan is to run this workload weekly. How should you optimize the cluster for cost?

  • A. Migrate the workload to Google Cloud Dataflow
  • B. Use pre-emptible virtual machines (VMs) for the cluster Most Voted
  • C. Use a higher-memory node so that the job runs faster
  • D. Use SSDs on the worker nodes so that the job can run faster

Odpowiedź: B.

**Wyjaśnienie:**
Użycie maszyn wirtualnych z wywłaszczaniem (preemptible VMs, obecnie znane jako Spot VMs) jest najbardziej efektywną kosztowo strategią dla tego scenariusza. Preemptible VMs oferują znaczące oszczędności (do 91% w porównaniu ze standardowymi VM), a ich czas życia jest ograniczony do 24 godzin i mogą zostać wyłączone w dowolnym momencie. Jednakże, dla obciążenia Sparka, które trwa zaledwie 30 minut i jest uruchamiane raz w tygodniu, ryzyko, że zadanie zostanie całkowicie przerwane i nie ukończone przed wywłaszczeniem, jest stosunkowo niskie. Co więcej, Apache Spark jest odporny na awarie węzłów, potrafiąc odzyskać i wznowić pracę po utracie węzła. To sprawia, że są one idealne do optymalizacji kosztów dla zadań wsadowych, które nie są krytyczne czasowo.

**Dlaczego pozostałe opcje są gorsze/błędne:**
A. Migracja obciążenia do Google Cloud Dataflow byłaby zmianą platformy, a nie optymalizacją obecnego klastra Dataproc, co jest przedmiotem pytania, i wiązałaby się z dodatkowymi kosztami migracji i testowania.
C. Użycie węzłów o większej pamięci w celu przyspieszenia zadania zwiększyłoby koszt jednostkowy maszyn wirtualnych, a skrócenie czasu wykonania niekoniecznie przełoży się na niższe koszty całkowite, jeśli koszt per godzina znacząco wzrośnie.
D. Użycie dysków SSD na węzłach roboczych również zwiększyłoby koszt maszyn wirtualnych, a dla wielu obciążeń Sparka, zwłaszcza tych czytających dane z Google Cloud Storage, lokalne dyski SSD mogą nie przynieść wystarczających korzyści wydajnościowych, aby uzasadnić wzrost kosztów.

**Dodatkowe źródła:**
*   Google Cloud Dataproc Preemptible VMs
*   Compute Engine Spot VMs

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Use pre-emptible virtual machines (VMs) for the cluster (obecnie znane jako Spot VMs).

Dlaczego?

Wymaganie: "optimize the cluster for cost" (optymalizacja kosztów). Charakterystyka zadania: 1. "Workload can run in approximately 30 minutes". 2. "Run this workload weekly". Zadania wsadowe (batch jobs), szczególnie te krótkie (30 min) i odporne na błędy (jak Spark), są idealnym kandydatem dla maszyn wywłaszczalnych (Preemptible/Spot). Kosztują one znacznie mniej (nawet o 60-91%) niż standardowe instancje. Ryzyko wyłączenia maszyny podczas 30-minutowego zadania jest akceptowalne, a Dataproc automatycznie ponowi zadania na innych węzłach.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Migrate to Dataflow: Dataflow jest świetny, ale migracja z Sparka (Scala) wymagałaby przepisania kodu (koszt inżynierski). Pytanie dotyczy optymalizacji istniejącego rozwiązania ("cluster"), a nie zmiany platformy.
  • C. Higher-memory node: Zwiększenie pamięci zazwyczaj zwiększa koszt godzinowy. Może przyspieszyć zadanie, ale rzadko na tyle, by skompensować wyższą stawkę (chyba że zadanie jest memory-bound i spilluje na dysk, ale Spot VMs są pewniejszą oszczędnością).
  • D. SSDs: Dyski SSD są droższe niż standardowe dyski talerzowe (HDD). Choć przyspieszają I/O, w przypadku prostych modeli statystycznych i klastrów Dataproc, rzadko jest to optymalizacja kosztowa (raczej wydajnościowa).
