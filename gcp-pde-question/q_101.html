You need to copy millions of sensitive patient records from a relational database to BigQuery. The total size of the database is 10 TB. You need to design a solution that is secure and time-efficient. What should you do?

  • A. Export the records from the database as an Avro file. Upload the file to GCS using gsutil, and then load the Avro file into BigQuery using the BigQuery web UI in the GCP Console.
  • B. Export the records from the database as an Avro file. Copy the file onto a Transfer Appliance and send it to Google, and then load the Avro file into BigQuery using the BigQuery web UI in the GCP Console. Most Voted
  • C. Export the records from the database into a CSV file. Create a public URL for the CSV file, and then use Storage Transfer Service to move the file to Cloud Storage. Load the CSV file into BigQuery using the BigQuery web UI in the GCP Console.
  • D. Export the records from the database as an Avro file. Create a public URL for the Avro file, and then use Storage Transfer Service to move the file to Cloud Storage. Load the Avro file into BigQuery using the BigQuery web UI in the GCP Console.

Odpowiedź: B

Wyjaśnienie: Opcja B jest najlepsza, ponieważ jest najbardziej efektywna i bezpieczna w przypadku przenoszenia ogromnych ilości danych do BigQuery. Wykorzystanie `gsutil` do uploadu do Google Cloud Storage jest standardową metodą, a następnie użycie BigQuery web UI do ładowania Avro pliku jest proste i intuicyjne. Alternatywy A, C i D są mniej optymalne ze względu na większe wymagania dotyczące czasu, potencjalne problemy z bezpieczeństwem (np. wysyłanie plików do zewnętrznych urządzeń), oraz brak automatyzacji procesu.  `CSV` jako metoda przenoszenia danych jest często mniej wydajna i bardziej podatna na błędy, szczególnie przy dużych zbiorach danych.
Link do dodatkowych żródeł: https://cloud.google.com/bigquery/docs/transfer/avro-file

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Najlepszym rozwiązaniem jest użycie Transfer Appliance (opcja B) do przeniesienia 10 TB wrażliwych danych pacjentów do BigQuery. Kluczowym czynnikiem jest bezpieczeństwo danych. Przesyłanie wrażliwych danych przez publiczny Internet (jak w opcjach C i D) jest ryzykowne. Opcja A, choć popularna, może być nieefektywna dla tak dużego wolumenu danych.

Dyskusja użytkowników potwierdza, że Transfer Appliance jest zaprojektowany do takich scenariuszy. Wielu użytkowników zwraca uwagę na ograniczenia gsutil (choć nowsze wersje gcloud storage obsługują większe pliki, to wciąż Transfer Appliance jest bardziej niezawodny dla 10TB). Użytkownik TNT87 słusznie zauważa, że gsutil ma limit 1TB, co wyklucza opcję A jako pojedynczą operację.

Choć niektórzy użytkownicy (np. spicebits, A_Nasser, Colourseun) argumentują, że gsutil może być szybszy przy odpowiedniej przepustowości, to bezpieczeństwo danych pacjentów jest priorytetem. Transfer Appliance zapewnia fizyczne zabezpieczenie danych podczas transportu. Dodatkowo, czas transportu, choć dłuższy, jest akceptowalny w kontekście bezpieczeństwa i niezawodności.

Użytkownik grshankar9 zwraca uwagę na to, że Storage Transfer Service (STS) jest zalecany dla plików większych niż 1TB, co również wspiera wybór opcji B, ponieważ Transfer Appliance jest często używany w połączeniu z STS do załadowania danych do BigQuery.

Podsumowując, opcja B jest najbardziej bezpieczna, skalowalna i zaprojektowana do obsługi dużych ilości danych, co czyni ją najlepszym wyborem w tym scenariuszu.
