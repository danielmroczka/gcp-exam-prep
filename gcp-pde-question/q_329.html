Your Cloud Storage data lake has raw, processed, and historical data in different buckets. Data older than two years is rarely accessed, and all data must be retained for no longer than seven years. You are concerned about rising storage costs. How should you control costs for the historical data bucket?

  • A. Write a script on a Compute Engine instance, triggered daily by Cloud Scheduler, to scan all objects and delete any older than seven years.
  • B. Configure an Object Lifecycle Management (OLM) rule to transition objects older than two years to the Archive storage class and delete them after seven years.
  • C. Enable the Autoclass feature on your Cloud Storage buckets and select Opt-in to object transitions to Coldline and Archive storage classes.
  • D. Replicate the buckets to a different region with lower storage costs and configure an Object Lifecycle Management rule to delete objects after seven years.
Show Suggested Answer
Suggested Answer: B

Wyjaśnienie

Correct Answer: B

Object Lifecycle Management (OLM) is the native Cloud Storage feature for managing object lifetimes. You can define rules to automatically transition objects to cheaper storage classes (like Archive for data older than 2 years) and to delete objects permanently (after 7 years). This is a managed, no-code solution that is highly cost-effective.

Why other options are incorrect:
A. Running a custom script on Compute Engine is complex, requires maintenance, and incurs Compute Engine costs. OLM is the preferred managed alternative.
C. Autoclass automatically transitions objects based on access patterns, which is great, but it doesn't handle the "delete after 7 years" requirement. A specific OLM rule is needed for deletion.
D. Replicating data just for cost savings usually increases egress and storage costs due to duplication, unless the data is moved entirely. OLM within the existing bucket is simpler.
