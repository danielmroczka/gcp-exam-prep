You are developing an application that uses a recommendation engine on Google Cloud. Your solution should display new videos to customers based on past views. Your solution needs to generate labels for the entities in videos that the customer has viewed. Your design must be able to provide very fast filtering suggestions based on data from other customer preferences on several TB of data. What should you do?

  • A. Build and train a complex classification model with Spark MLlib to generate labels and filter the results. Deploy the models using Cloud Dataproc. Call the model from your application.
  • B. Build and train a classification model with Spark MLlib to generate labels. Build and train a second classification model with Spark MLlib to filter results to match customer preferences. Deploy the models using Cloud Dataproc. Call the models from your application.
  • C. Build an application that calls the Cloud Video Intelligence API to generate labels. Store data in Cloud Bigtable, and filter the predicted labels to match the user's viewing history to generate preferences. Most Voted
  • D. Build an application that calls the Cloud Video Intelligence API to generate labels. Store data in Cloud SQL, and join and filter the predicted labels to match the user's viewing history to generate preferences.

Odpowiedź: C.

**Wyjaśnienie:**
Wybór opcji C jest najbardziej optymalny, ponieważ wykorzystuje gotowe i zarządzane usługi Google Cloud do rozwiązania specyficznych problemów. Cloud Video Intelligence API pozwala na automatyczne generowanie etykiet dla obiektów w wideo bez konieczności czasochłonnego budowania i trenowania własnych modeli uczenia maszynowego (jak w opcjach A i B). Cloud Bigtable jest idealnym wyborem do przechowywania kilku terabajtów danych, ponieważ jest to skalowalna baza danych NoSQL zaprojektowana dla bardzo niskich opóźnień (sub-millisecond) i wysokiej przepustowości, co jest kluczowe dla systemów rekomendacji działających w czasie rzeczywistym. Dzięki takiemu połączeniu, rozwiązanie jest wydajne, szybkie i łatwiejsze w utrzymaniu.

**Dlaczego pozostałe opcje są gorsze/błędne:**
* **A:** Budowanie własnego, złożonego modelu klasyfikacji w Spark MLlib jest nieefektywne, gdy dostępne jest gotowe i zoptymalizowane Cloud Video Intelligence API.
* **B:** Podobnie jak w opcji A, zarządzanie infrastrukturą Dataproc i trenowanie dwóch osobnych modeli wprowadza niepotrzebną złożoność i wyższe koszty operacyjne.
* **C:** (Opcja poprawna).
* **D:** Cloud SQL jako relacyjna baza danych może mieć trudności z zapewnieniem bardzo szybkiego filtrowania (latency) przy danych rzędu wielu terabajtów w porównaniu do Bigtable.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Build an application that calls the Cloud Video Intelligence API... Store data in Cloud Bigtable....

Dlaczego?

1. Generate labels for entities in videos: Zamiast budować własny model (kosztowne, trudne), należy użyć gotowego Cloud Video Intelligence API, które automatycznie rozpoznaje obiekty i etykiety w wideo. 2. Several TB of data i Very fast filtering: To kluczowe wymagania dla bazy danych. Cloud SQL (relacyjna) słabo skaluje się przy terabajtach danych z wymogiem bardzo szybkich odczytów w czasie rzeczywistym. Cloud Bigtable (NoSQL, szerokokolumnowa) jest stworzona do obsługi ogromnych wolumenów danych i zapewnia milisekundowe opóźnienia, co jest idealne dla silników rekomendacji ("recommendation engine").

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i B. Build classification model with Spark: To "reinventing the wheel". Trenowanie własnego modelu na wideo jest bardzo trudne i wymaga ogromnych zasobów. Video Intelligence API robi to "out of the box".
  • D. Store in Cloud SQL: Cloud SQL nie jest zalecane dla "several TB of data" w zastosowaniach wymagających wysokiej przepustowości i niskich opóźnień przy filtrowaniu (chyba że użyjemy sharding, co jest skomplikowane). Bigtable jest natywnym wyborem dla takich wolumenów w GCP.
