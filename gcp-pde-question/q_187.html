The Development and External teams have the project viewer Identity and Access Management (IAM) role in a folder named Visualization. You want the
Development Team to be able to read data from both Cloud Storage and BigQuery, but the External Team should only be able to read data from BigQuery. What should you do?

  • A. Remove Cloud Storage IAM permissions to the External Team on the acme-raw-data project.
  • B. Create Virtual Private Cloud (VPC) firewall rules on the acme-raw-data project that deny all ingress traffic from the External Team CIDR range.
  • C. Create a VPC Service Controls perimeter containing both projects and BigQuery as a restricted API. Add the External Team users to the perimeter's Access Level.
  • D. Create a VPC Service Controls perimeter containing both projects and Cloud Storage as a restricted API. Add the Development Team users to the perimeter's Access Level. Most Voted

Odpowiedź: D.

Rozwiązanie polega na wykorzystaniu VPC Service Controls (VPC SC) do precyzyjnego kontrolowania dostępu do zasobów. Tworząc perymetr VPC SC zawierający oba projekty (acme-raw-data i Visualization) oraz Cloud Storage jako ograniczony API, a następnie dodając użytkowników Development Team do poziomu dostępu perymetru, zapewniamy im dostęp do Cloud Storage. Ponieważ External Team nie jest dodany do poziomu dostępu perymetru dla Cloud Storage, nie będą mieli do niego dostępu, zachowując jednocześnie dostęp do BigQuery, który nie jest ograniczony przez ten perymetr w kontekście tego pytania. VPC SC pozwala na granularną kontrolę dostępu, niezależnie od uprawnień IAM, co jest kluczowe w tym scenariuszu.  Dzięki temu spełniamy wymaganie, aby Development Team miał dostęp do obu zasobów, a External Team tylko do BigQuery.

A. Samo usunięcie uprawnień IAM dla External Team na projekcie acme-raw-data nie uwzględnia folderu Visualization, gdzie już mają uprawnienia.
B. VPC firewall rules kontrolują ruch sieciowy, a nie dostęp do danych na poziomie API.
C. Ograniczenie BigQuery jako API w perymetrze VPC SC nie rozwiązuje problemu dostępu External Team do Cloud Storage.

Sugerowana odpowiedź: D

Poprawną odpowiedzią jest D. Rozwiązanie wykorzystuje VPC Service Controls (VPC SC) do precyzyjnej kontroli dostępu do zasobów. Tworząc perymetr VPC SC zawierający oba projekty oraz Cloud Storage jako ograniczony API, a następnie dodając użytkowników Development Team do poziomu dostępu perymetru, zapewniamy im dostęp do Cloud Storage. Ponieważ External Team nie jest dodany do poziomu dostępu perymetru dla Cloud Storage, nie będą mieli do niego dostępu, zachowując jednocześnie dostęp do BigQuery, który nie jest ograniczony przez ten perymetr w kontekście tego pytania. VPC SC pozwala na granularną kontrolę dostępu, niezależnie od uprawnień IAM, co jest kluczowe w tym scenariuszu.

Dyskusja użytkowników początkowo była podzielona, z silnym poparciem dla odpowiedzi C. Jednakże, po głębszej analizie i uwzględnieniu nowszych komentarzy, odpowiedź D wydaje się bardziej precyzyjna i zgodna z celem pytania. Odpowiedź C ogranicza dostęp do BigQuery, co nie jest wymagane – External Team ma mieć dostęp do BigQuery. Kluczowe jest, że chcemy *ograniczyć* dostęp do Cloud Storage, a nie do BigQuery.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Samo usunięcie uprawnień IAM dla External Team na projekcie acme-raw-data nie uwzględnia folderu Visualization, gdzie już mają uprawnienia.
  • B: VPC firewall rules kontrolują ruch sieciowy, a nie dostęp do danych na poziomie API.
  • C: Ograniczenie BigQuery jako API w perymetrze VPC SC nie rozwiązuje problemu dostępu External Team do Cloud Storage. Co więcej, uniemożliwiłoby dostęp do BigQuery dla Development Team, jeśli nie zostałoby dodane do poziomu dostępu.

Wiele komentarzy (np. od cloudmon, maci_f, zellck) poprawnie argumentuje, że Cloud Storage powinno być ograniczonym API, a Development Team powinien mieć dostęp do niego poprzez perymetr VPC SC.
