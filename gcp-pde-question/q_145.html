You receive data files in CSV format monthly from a third party. You need to cleanse this data, but every third month the schema of the files changes. Your requirements for implementing these transformations include:
✑ Executing the transformations on a schedule
✑ Enabling non-developer analysts to modify transformations
✑ Providing a graphical tool for designing transformations
What should you do?

  • A. Use Dataprep by Trifacta to build and maintain the transformation recipes, and execute them on a scheduled basis Most Voted
  • B. Load each month's CSV data into BigQuery, and write a SQL query to transform the data to a standard schema. Merge the transformed tables together with a SQL query
  • C. Help the analysts write a Dataflow pipeline in Python to perform the transformation. The Python code should be stored in a revision control system and modified as the incoming data's schema changes
  • D. Use Apache Spark on Dataproc to infer the schema of the CSV file before creating a Dataframe. Then implement the transformations in Spark SQL before writing the data out to Cloud Storage and loading into BigQuery

Odpowiedź: A.

Dataprep by Trifacta jest idealnym rozwiązaniem dla tego problemu, ponieważ oferuje interfejs graficzny do projektowania transformacji danych, co umożliwia analitykom bez umiejętności programowania modyfikowanie logiki transformacji. Dodatkowo, Dataprep automatycznie wykrywa zmiany w schemacie danych i proponuje odpowiednie korekty w przepisie transformacji, co jest kluczowe w sytuacji, gdy schemat zmienia się co trzeci miesiąc. Możliwość planowania uruchomień transformacji (scheduling) jest również wbudowana w Dataprep.  Dzięki temu proces czyszczenia i transformacji danych jest efektywny i dostępny dla osób nietechnicznych. Dataprep pozwala na tworzenie "przepisów" (recipes) transformacji, które można łatwo powtarzać i dostosowywać.

A. Jest to najlepsza opcja, ponieważ spełnia wszystkie wymagania zadania.
B. Użycie SQL jest mniej elastyczne w przypadku dynamicznie zmieniających się schematów i wymaga umiejętności programowania SQL.
C. Wymaga od analityków umiejętności programowania w Pythonie i utrzymania kodu, co nie spełnia wymogu dostępności dla osób nietechnicznych.
D. Chociaż Spark może inferować schemat, to implementacja transformacji w Spark SQL nadal wymaga umiejętności programowania i nie oferuje graficznego interfejsu dla analityków.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Odpowiedź A, czyli użycie Dataprep by Trifacta, jest zdecydowanie najlepszym rozwiązaniem dla przedstawionego problemu. Spełnia ona wszystkie podane wymagania:

  • Planowanie (Scheduling): Dataprep umożliwia planowanie uruchomień transformacji danych, co jest kluczowe dla przetwarzania danych otrzymywanych co miesiąc.
  • Dostępność dla analityków nietechnicznych: Dataprep oferuje graficzny interfejs (drag-and-drop), który pozwala analitykom bez umiejętności programowania na projektowanie i modyfikowanie transformacji.
  • Graficzne narzędzie: Interfejs graficzny jest podstawową cechą Dataprep, co ułatwia wizualizację i manipulację danymi.
  • Elastyczność schematu: Dataprep jest zaprojektowany do radzenia sobie ze zmianami w schemacie danych. Potrafi wykrywać zmiany i sugerować odpowiednie korekty w przepisie transformacji.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość komentarzy wskazuje na A jako prawidłową, podkreślając jej zalety, takie jak łatwość użycia dla analityków nietechnicznych i zdolność do radzenia sobie ze zmieniającymi się schematami. Komentarze takie jak te od użytkowników zellck, arpitagrawal, barnac1es i CGS22 szczegółowo wyjaśniają, dlaczego Dataprep jest idealnym rozwiązaniem.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Użycie SQL jest mniej elastyczne w przypadku dynamicznie zmieniających się schematów i wymaga znajomości SQL.
  • C: Wymaga od analityków umiejętności programowania w Pythonie, co nie spełnia wymogu dostępności dla osób nietechnicznych.
  • D: Chociaż Spark może inferować schemat, implementacja transformacji w Spark SQL nadal wymaga umiejętności programowania i nie oferuje graficznego interfejsu dla analityków. Użytkownik SSV początkowo sugerował tę opcję, ale argumentacja jest błędna - Dataprep radzi sobie ze zmianami schematu lepiej niż ręczne modyfikacje w Sparku.
