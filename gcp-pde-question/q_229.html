You currently use a SQL-based tool to visualize your data stored in BigQuery. The data visualizations require the use of outer joins and analytic functions. Visualizations must be based on data that is no less than 4 hours old. Business users are complaining that the visualizations are too slow to generate. You want to improve the performance of the visualization queries while minimizing the maintenance overhead of the data preparation pipeline. What should you do?

  • A. Create materialized views with the allow_non_incremental_definition option set to true for the visualization queries. Specify the max_staleness parameter to 4 hours and the enable_refresh parameter to true. Reference the materialized views in the data visualization tool. Most Voted
  • B. Create views for the visualization queries. Reference the views in the data visualization tool.
  • C. Create a Cloud Function instance to export the visualization query results as parquet files to a Cloud Storage bucket. Use Cloud Scheduler to trigger the Cloud Function every 4 hours. Reference the parquet files in the data visualization tool.
  • D. Create materialized views for the visualization queries. Use the incremental updates capability of BigQuery materialized views to handle changed data automatically. Reference the materialized views in the data visualization tool.

Odpowiedź: A.

Materializowane widoki (materialized views) są idealnym rozwiązaniem w tej sytuacji, ponieważ pre-kompilują wyniki złożonych zapytań, takich jak te wykorzystujące outer joins i funkcje analityczne, co znacznie przyspiesza czas generowania wizualizacji. Ustawienie `max_staleness` na 4 godziny spełnia wymaganie dotyczące wieku danych, a `enable_refresh` automatycznie odświeża widok, minimalizując nakład pracy związany z utrzymaniem. Opcja `allow_non_incremental_definition = true` pozwala na tworzenie widoków, które nie muszą być aktualizowane inkrementalnie, co może być korzystne w przypadku złożonych zapytań, gdzie inkrementalne aktualizacje byłyby zbyt kosztowne. Dzięki temu użytkownicy biznesowi otrzymają szybsze wizualizacje, a zespół inżynierski zminimalizuje obciążenie związane z optymalizacją zapytań i odświeżaniem danych.

A. Jest to najlepsza opcja, ponieważ wykorzystuje wbudowane mechanizmy BigQuery do optymalizacji zapytań i automatycznego odświeżania danych.
B. Widoki (views) nie przechowują pre-kompilowanych danych, więc nie poprawią wydajności zapytań, a jedynie zmienią sposób ich zapisu.
C. Eksportowanie do plików Parquet i używanie Cloud Functions/Scheduler wprowadza dodatkową złożoność i opóźnienia, a także zwiększa koszty związane z przechowywaniem i przetwarzaniem danych.
D. Chociaż inkrementalne aktualizacje są korzystne, nie zawsze są możliwe lub efektywne dla wszystkich typów zapytań, a w tym przypadku `allow_non_incremental_definition = true` sugeruje, że inkrementalne aktualizacje mogą nie być optymalne.

Sugerowana odpowiedź: A

Odpowiedź A jest najbardziej odpowiednia, ponieważ wykorzystuje materializowane widoki BigQuery z opcją allow_non_incremental_definition = true, max_staleness = 4 hours oraz enable_refresh = true. To rozwiązanie pre-kompiluje wyniki złożonych zapytań (outer joins i funkcje analityczne), co znacząco przyspiesza generowanie wizualizacji. Ustawienie max_staleness zapewnia, że dane w wizualizacjach nie będą starsze niż 4 godziny, spełniając wymagania biznesowe. Automatyczne odświeżanie (enable_refresh) minimalizuje nakład pracy związany z utrzymaniem.

Kluczowe jest użycie allow_non_incremental_definition = true. Zgodnie z dokumentacją BigQuery i komentarzem użytkownika ricardovazz, ta opcja pozwala na tworzenie materializowanych widoków dla zapytań, które nie mogą być efektywnie aktualizowane inkrementalnie (np. z powodu użycia outer joins lub funkcji analitycznych). W takim przypadku, pełne odświeżanie w określonych odstępach czasu (kontrolowane przez max_staleness) jest akceptowalnym kompromisem między wydajnością a kosztem.
