You are designing the database schema for a machine learning-based food ordering service that will predict what users want to eat. Here is some of the information you need to store:
✑ The user profile: What the user likes and doesn't like to eat
✑ The user account information: Name, address, preferred meal times
✑ The order information: When orders are made, from where, to whom
The database will be used to store all the transactional data of the product. You want to optimize the data schema. Which Google Cloud Platform product should you use?

  • A. BigQuery Most Voted
  • B. Cloud SQL
  • C. Cloud Bigtable
  • D. Cloud Datastore

Odpowiedź: A. BigQuery

BigQuery to bezserwerowa, wysoce skalowalna hurtownia danych, idealnie dopasowana do analitycznych obciążeń związanych z uczeniem maszynowym. Umożliwia efektywne przechowywanie i wykonywanie zapytań na petabajtach danych transakcyjnych, takich jak profile użytkowników, informacje o kontach i zamówieniach, co jest kluczowe dla budowania systemu predykcyjnego. Dzięki architekturze kolumnowej, BigQuery jest zoptymalizowany do szybkiego przetwarzania złożonych zapytań agregujących i analizujących duże wolumeny danych historycznych, co jest niezbędne do generowania cech dla modeli ML i ich trenowania. Bezproblemowa integracja z BigQuery ML dodatkowo upraszcza proces tworzenia i wdrażania modeli uczenia maszynowego bezpośrednio na danych. Wybór BigQuery optymalizuje schemat danych pod kątem wydajnej analizy i wsparcia dla przewidywań ML.

**Dlaczego pozostałe opcje są gorsze/błędne:**
*   **B. Cloud SQL:** Jest to doskonałe rozwiązanie dla tradycyjnych systemów OLTP (przetwarzania transakcji online) i aplikacji, ale nie jest optymalizowane pod kątem skalowalności i wydajności zapytań analitycznych na dużą skalę, które są wymagane dla uczenia maszynowego.
*   **C. Cloud Bigtable:** To baza danych NoSQL zoptymalizowana pod kątem bardzo wysokiej przepustowości i niskich opóźnień w dostępie do danych operacyjnych, ale nie nadaje się do złożonych, ad-hoc zapytań analitycznych wymaganych do inżynierii cech dla modeli ML.
*   **D. Cloud Datastore:** Jako dokumentowa baza danych NoSQL, jest dobra dla elastycznych schematów i małych transakcji, ale brakuje jej mocy analitycznej i elastyczności zapytań BigQuery do przetwarzania danych na dużą skalę dla ML.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A. BigQuery

Analizując dyskusję użytkowników oraz treść pytania, najbardziej prawdopodobną i poprawną odpowiedzią jest BigQuery. Pytanie podkreśla potrzebę optymalizacji schematu danych dla usługi predykcyjnej opartej na uczeniu maszynowym, która będzie przechowywać dane transakcyjne (profile użytkowników, informacje o kontach, zamówienia). BigQuery, jako wysoce skalowalna hurtownia danych, jest idealnie dopasowana do tego celu.

Choć niektórzy użytkownicy zwracają uwagę na fakt, że BigQuery nie jest tradycyjną bazą danych transakcyjną, to kluczowe jest zrozumienie, że pytanie nie wymaga obsługi transakcji w czasie rzeczywistym (OLTP). Wymaga natomiast efektywnego przechowywania i analizowania dużych ilości danych historycznych, co jest mocną stroną BigQuery. Dodatkowo, wsparcie dla BigQuery ML umożliwia bezpośrednie tworzenie i wdrażanie modeli uczenia maszynowego na przechowywanych danych.

Użytkownik 22c1725 trafnie zauważa, że celem przechowywania danych w tym przypadku nie jest obsługa operacji ACID, a przechowywanie i wykorzystanie danych do uczenia maszynowego. Użytkownik abhaya2608 słusznie wskazuje, że Cloud SQL nie wspiera bezpośrednio uczenia maszynowego.

Nowsze komentarze (np. grshankar9) również potwierdzają, że BigQuery oferuje funkcje optymalizacji schematu danych, takie jak automatyczne wykrywanie schematu i przechowywanie kolumnowe, co jest kluczowe dla efektywnego przetwarzania dużych zbiorów danych.

Dlaczego pozostałe opcje są mniej odpowiednie:

  • Cloud SQL: Dobry dla tradycyjnych aplikacji OLTP, ale nie skaluje się tak dobrze do analizy dużych zbiorów danych jak BigQuery.
  • Cloud Bigtable: Zoptymalizowany pod kątem wysokiej przepustowości i niskich opóźnień, ale nie nadaje się do złożonych zapytań analitycznych.
  • Cloud Datastore: Dobry dla elastycznych schematów i małych transakcji, ale brakuje mu mocy analitycznej BigQuery.
