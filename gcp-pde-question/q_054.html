Your globally distributed auction application allows users to bid on items. Occasionally, users place identical bids at nearly identical times, and different application servers process those bids. Each bid event contains the item, amount, user, and timestamp. You want to collate those bid events into a single location in real time to determine which user bid first. What should you do?

  • A. Create a file on a shared file and have the application servers write all bid events to that file. Process the file with Apache Hadoop to identify which user bid first.
  • B. Have each application server write the bid events to Cloud Pub/Sub as they occur. Push the events from Cloud Pub/Sub to a custom endpoint that writes the bid event information into Cloud SQL.
  • C. Set up a MySQL database for each application server to write bid events into. Periodically query each of those distributed MySQL databases and update a master MySQL database with bid event information.
  • D. Have each application server write the bid events to Google Cloud Pub/Sub as they occur. Use a pull subscription to pull the bid events using Google Cloud Dataflow. Give the bid for each item to the user in the bid event that is processed first. Most Voted

Odpowiedź: D.

Wyjaśnienie:
Opcja D jest najlepszym rozwiązaniem, ponieważ łączy w sobie skalowalność i możliwości przetwarzania strumieniowego w czasie rzeczywistym, co jest kluczowe dla wymagań zadania. Google Cloud Pub/Sub to w pełni zarządzana, globalnie dostępna usługa wiadomości, która doskonale nadaje się do niezawodnego i w czasie rzeczywistym zbierania zdarzeń z rozproszonych serwerów aplikacji. Następnie Google Cloud Dataflow, oparty na Apache Beam, jest idealnym narzędziem do przetwarzania tych strumieniowych danych. Dataflow potrafi w czasie rzeczywistym analizować zdarzenia, stosować logikę windowania i zarządzania stanem, aby efektywnie identyfikować pierwszego użytkownika, który złożył daną ofertę, na podstawie znacznika czasu zdarzenia. Dzięki temu rozwiązanie jest wysoce skalowalne, odporne na błędy i zarządzane, co minimalizuje obciążenie operacyjne.

Dlaczego pozostałe opcje są gorsze/błędne:
*   **A:** Tworzenie pliku na współdzielonym systemie plików jest rozwiązaniem, które szybko stanie się wąskim gardłem dla globalnie rozproszonej aplikacji i dużej liczby jednoczesnych zapisów, a przetwarzanie pliku za pomocą Apache Hadoop jest procesem wsadowym, a nie w czasie rzeczywistym.
*   **B:** Chociaż Pub/Sub jest dobrym wyborem do odbioru zdarzeń, wysyłanie ich do niestandardowego punktu końcowego, który zapisuje pojedyncze zdarzenia do Cloud SQL, może być nieefektywne pod względem wydajności i kosztów dla dużego wolumenu danych, a także wymagałoby samodzielnego zarządzania aplikacją i implementacji skomplikowanej logiki określania "pierwszej oferty" w czasie rzeczywistym.
*   **C:** Konfiguracja oddzielnych baz danych MySQL dla każdego serwera aplikacji tworzy silosy danych i generuje duże koszty operacyjne, a okresowe odpytywanie i aktualizowanie głównej bazy danych nie spełnia wymogu przetwarzania w czasie rzeczywistym.

Dodatkowe źródła:
*   Google Cloud Pub/Sub: https://cloud.google.com/pubsub
*   Google Cloud Dataflow: https://cloud.google.com/dataflow

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Odpowiedź D jest najbardziej odpowiednia, ponieważ łączy w sobie skalowalność, przetwarzanie strumieniowe w czasie rzeczywistym oraz możliwość poprawnego ustalenia kolejności zdarzeń. Google Cloud Pub/Sub zapewnia niezawodny odbiór zdarzeń z rozproszonych serwerów aplikacji. Google Cloud Dataflow, wykorzystując okna sesyjne (session windows) z sortowaniem po znaczniku czasu (timestamp), umożliwia deterministyczne ustalenie, który użytkownik złożył ofertę jako pierwszy, nawet w przypadku potencjalnego braku kolejności dostarczania wiadomości przez Pub/Sub. Kluczowe jest przetwarzanie po czasie zdarzenia (event time), a nie czasie przetwarzania (process time).

Dyskusja użytkowników jest podzielona, ale nowsze komentarze (z ostatnich miesięcy) oraz te z większą liczbą głosów (upvotes) skłaniają się ku odpowiedzi D. Użytkownicy zwracają uwagę na możliwość przetwarzania zdarzeń poza kolejnością w Pub/Sub, ale Dataflow z odpowiednią konfiguracją (session windows i sortowanie po timestamp) rozwiązuje ten problem.

Dlaczego pozostałe odpowiedzi są nieprawidłowe:

  • A: Użycie współdzielonego pliku i Hadoop jest rozwiązaniem wsadowym, a nie w czasie rzeczywistym, co jest kluczowe w tym scenariuszu.
  • B: Zapisywanie do Cloud SQL z niestandardowego punktu końcowego wprowadza dodatkową złożoność, potencjalne wąskie gardła i opóźnienia. Cloud SQL nie jest zoptymalizowany do obsługi dużej liczby zapisów w czasie rzeczywistym.
  • C: Użycie oddzielnych baz danych MySQL dla każdego serwera aplikacji tworzy silosy danych i utrudnia centralne przetwarzanie w czasie rzeczywistym.

Komentarze użytkowników, takie jak ten od theRafael7, wskazują, że Cloud SQL nie jest globalny i nie radzi sobie z kolejnością tak dobrze jak Dataflow, co dyskwalifikuje odpowiedź B.
