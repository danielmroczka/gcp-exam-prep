You are deploying an Apache Airflow directed acyclic graph (DAG) in a Cloud Composer 2 instance. You have incoming files in a Cloud Storage bucket that the DAG processes, one file at a time. The Cloud Composer instance is deployed in a subnetwork with no Internet access. Instead of running the DAG based on a schedule, you want to run the DAG in a reactive way every time a new file is received. What should you do?

  • A. 1. Enable Private Google Access in the subnetwork, and set up Cloud Storage notifications to a Pub/Sub topic.
    2. Create a push subscription that points to the web server URL.
  • B. 1. Enable the Cloud Composer API, and set up Cloud Storage notifications to trigger a Cloud Function.
    2. Write a Cloud Function instance to call the DAG by using the Cloud Composer API and the web server URL.
    3. Use VPC Serverless Access to reach the web server URL.
  • C. 1. Enable the Airflow REST API, and set up Cloud Storage notifications to trigger a Cloud Function instance.
    2. Create a Private Service Connect (PSC) endpoint.
    3. Write a Cloud Function that connects to the Cloud Composer cluster through the PSC endpoint. Most Voted
  • D. 1. Enable the Airflow REST API, and set up Cloud Storage notifications to trigger a Cloud Function instance.
    2. Write a Cloud Function instance to call the DAG by using the Airflow REST API and the web server URL.
    3. Use VPC Serverless Access to reach the web server URL.

Odpowiedź: C.

Opcja C jest poprawna, ponieważ pozwala na reaktywne uruchamianie DAG-ów w Cloud Composer 2 bez konieczności dostępu do Internetu. Cloud Storage notifications wywołują Cloud Function, która następnie, dzięki Private Service Connect (PSC), bezpiecznie łączy się z klastrem Cloud Composer. PSC umożliwia prywatne połączenie z usługami Google bez użycia publicznych adresów IP, co jest kluczowe w środowisku bez dostępu do Internetu. Airflow REST API pozwala na programowe uruchamianie DAG-ów. To rozwiązanie zapewnia zarówno reaktywność, jak i bezpieczeństwo.

A.  Umożliwienie Private Google Access i użycie Pub/Sub do wywołania web serwera nie jest idealne, ponieważ web serwer może nie być dostępny z sieci prywatnej bez dodatkowej konfiguracji.
B.  Użycie Cloud Composer API i Cloud Functions z VPC Serverless Access jest skomplikowane i mniej bezpieczne niż PSC, a także wymaga dostępu do Internetu przez VPC Serverless Access.
D.  Podobnie jak B, użycie Airflow REST API i Cloud Functions z VPC Serverless Access jest mniej bezpieczne i wymaga dostępu do Internetu, a także jest bardziej skomplikowane niż rozwiązanie z PSC.

Sugerowana odpowiedź: C

Odpowiedź C jest najbardziej poprawna i bezpieczna w kontekście zadania. Kluczowym wymaganiem jest uruchamianie DAG-ów w reakcji na pojawianie się nowych plików w Cloud Storage, w środowisku bez dostępu do Internetu. Rozwiązanie to wykorzystuje następujące elementy:

  • Airflow REST API: Umożliwia programowe uruchamianie DAG-ów.
  • Cloud Storage Notifications: Wywołują Cloud Function w momencie pojawienia się nowego pliku.
  • Private Service Connect (PSC): Zapewnia bezpieczne, prywatne połączenie z klastrem Cloud Composer bez użycia publicznych adresów IP, co jest niezbędne w środowisku bez dostępu do Internetu.
  • Cloud Function: Działa jako pośrednik, odbierając powiadomienia z Cloud Storage i uruchamiając DAG poprzez Airflow REST API, wykorzystując połączenie PSC.

Komentarze użytkowników, szczególnie te nowsze (Zek, aditya_ali), potwierdzają, że PSC jest preferowanym i najbezpieczniejszym sposobem na dostęp do Airflow REST API w środowisku prywatnym. Dokumentacja Google Cloud również wspiera to rozwiązanie.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Użycie Private Google Access i Pub/Sub może być problematyczne, ponieważ web serwer Airflow może nie być dostępny z sieci prywatnej bez dodatkowej konfiguracji.
  • B i D: Wykorzystanie VPC Serverless Access, choć możliwe, jest mniej bezpieczne niż PSC i potencjalnie bardziej skomplikowane w konfiguracji. Użytkownik d11379b słusznie zauważa, że Cloud Composer API nie uruchamia DAG-ów poprzez URL web serwera.

Użytkownik STEVE_PEGLEG słusznie zwraca uwagę na dokumentację, która wskazuje, że połączenie Cloud Function z Airflow web serverem nie jest możliwe w konfiguracjach Private IP i VPC Service Controls, co dyskwalifikuje opcje A, B i D.
