You set up a streaming data insert into a Redis cluster via a Kafka cluster. Both clusters are running on Compute Engine instances. You need to encrypt data at rest with encryption keys that you can create, rotate, and destroy as needed. What should you do?

  • A. Create a dedicated service account, and use encryption at rest to reference your data stored in your Compute Engine cluster instances as part of your API service calls.
  • B. Create encryption keys in Cloud Key Management Service. Use those keys to encrypt your data in all of the Compute Engine cluster instances. Most Voted
  • C. Create encryption keys locally. Upload your encryption keys to Cloud Key Management Service. Use those keys to encrypt your data in all of the Compute Engine cluster instances.
  • D. Create encryption keys in Cloud Key Management Service. Reference those keys in your API service calls when accessing the data in your Compute Engine cluster instances.

Odpowiedź: B.

**Wyjaśnienie:**
Wymaganie dotyczące tworzenia, rotowania i niszczenia kluczy szyfrujących najlepiej spełnia usługa Cloud Key Management Service (KMS), która oferuje tzw. klucze zarządzane przez klienta (CMEK). W przypadku klastrów Kafka i Redis działających na Compute Engine, dane "at rest" (w spoczynku) znajdują się na dyskach trwałych (Persistent Disks). Opcja B pozwala na powiązanie klucza z Cloud KMS bezpośrednio z dyskami instancji Compute Engine, co zapewnia automatyczne szyfrowanie bez konieczności modyfikowania kodu aplikacji. Dzięki temu rozwiązaniu administrator ma pełną kontrolę nad cyklem życia klucza, spełniając wszystkie wymienione w pytaniu warunki. Jest to standardowe i zalecane podejście w Google Cloud dla zapewnienia zgodności i bezpieczeństwa infrastruktury.

**Analiza pozostałych opcji:**
*   **A:** Konta serwisowe służą do zarządzania tożsamością i uprawnieniami, a nie do zarządzania cyklem życia kluczy szyfrujących dane na dyskach.
*   **B:** Jest to opcja poprawna, ponieważ Cloud KMS zapewnia natywną integrację z Compute Engine i pełną kontrolę nad kluczami szyfrującymi dyski.
*   **C:** Tworzenie kluczy lokalnie (Customer-Supplied Encryption Keys - CSEK) zwiększa złożoność zarządzania i odpowiedzialność użytkownika, co jest niepotrzebne, gdy Cloud KMS może bezpiecznie generować klucze wewnątrz chmury.
*   **D:** Odwoływanie się do kluczy w wywołaniach API sugeruje szyfrowanie na poziomie warstwy aplikacji, co jest znacznie trudniejsze do zaimplementowania w gotowych narzędziach jak Kafka czy Redis niż szyfrowanie na poziomie dysków.

**Dodatkowe źródła:**
*   Google Cloud KMS - Customer-Managed Encryption Keys (CMEK)
*   Protecting resources with Cloud KMS keys (Compute Engine)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Create encryption keys in Cloud Key Management Service. Use those keys to encrypt your data in all of the Compute Engine cluster instances..

Dlaczego?

Wymagania: 1. Encrypt data at rest: Dane na dyskach muszą być zaszyfrowane. 2. Create, rotate, and destroy keys as needed: Wymaga pełnej kontroli nad cyklem życia kluczy (Key Management). Cloud KMS (Key Management Service) to usługa dedykowana do zarządzania kluczami kryptograficznymi. Google Compute Engine integruje się z KMS poprzez CMEK (Customer-Managed Encryption Keys). Możesz stworzyć klucz w KMS i wskazać go przy tworzeniu dysków dla instancji GCE. Google zajmie się szyfrowaniem/odszyfrowywaniem, ale Ty kontrolujesz klucz (możesz go rotować lub zniszczyć, co skutecznie niszczy dane).

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Service Account: Konta serwisowe służą do autoryzacji (kto może co zrobić), a nie do szyfrowania danych (jak zabezpieczyć bity na dysku).
  • C. Create keys locally (CSEK): Customer-Supplied Encryption Keys. Jest to możliwe, ale trudniejsze w zarządzaniu (musisz sam dbać o bezpieczeństwo kluczy lokalnie). Skoro mamy "rotate and destroy as needed" i chcemy to robić w chmurze, KMS (CMEK) jest rozwiązaniem natywnym i łatwiejszym operacyjnie niż ręczne przesyłanie kluczy CSEK przy każdym starcie maszyny.
  • D. Reference keys in API calls: To sugeruje szyfrowanie po stronie aplikacji (Client-side encryption). Choć możliwe, jest trudne do wdrożenia w gotowym sofcie jak Redis czy Kafka (wymagałoby zmian w kodzie lub konfiguracji). Szyfrowanie dysków (B) jest przezroczyste dla aplikacji i spełnia wymóg "data at rest".
