You work for a shipping company that uses handheld scanners to read shipping labels. Your company has strict data privacy standards that require scanners to only transmit tracking numbers when events are sent to Kafka topics. A recent software update caused the scanners to accidentally transmit recipients' personally identifiable information (PII) to analytics systems, which violates user privacy rules. You want to quickly build a scalable solution using cloud-native managed services to prevent exposure of PII to the analytics systems. What should you do?

  • A. Create an authorized view in BigQuery to restrict access to tables with sensitive data.
  • B. Install a third-party data validation tool on Compute Engine virtual machines to check the incoming data for sensitive information.
  • C. Use Cloud Logging to analyze the data passed through the total pipeline to identify transactions that may contain sensitive information.
  • D. Build a Cloud Function that reads the topics and makes a call to the Cloud Data Loss Prevention (Cloud DLP) API. Use the tagging and confidence levels to either pass or quarantine the data in a bucket for review. Most Voted

Odpowiedź: D

Wyjaśnienie: Najpierw, Cloud Logging jest doskonałym narzędziem do monitorowania danych w systemie. Analiza logów pozwala na identyfikację transakcji, które mogą zawierać PII, nawet jeśli nie są bezpośrednio transmitowane do Analytics.  Kluczowe jest zrozumienie, jakie dane są przetwarzane przez pipeline i jakie są potencjalne punkty wyjścia dla danych.  Wybór między A, B i C jest mniej precyzyjny.  A.  Ograniczenie dostępu do tabel w BigQuery jest dobrym rozwiązaniem, ale nie rozwiązuje problemu źródła danych, który jest problemem w systemie. B.  Weryfikacja przez daty jest pomocna, ale nie identyfikuje *co* jest wrażliwe. C.  Cloud Logging może pomóc w identyfikacji potencjalnych problemów, ale nie jest wystarczającym do rozwiązania problemu, który dotyczy bezpośredniego przekazywania PII.  D.  Budowanie Cloud Function, który czyta dane z topiców i wywołuje Cloud DLP API, jest najbardziej odpowiednim rozwiązaniem, ponieważ pozwala na automatyczne wykrywanie i reagowanie na potencjalne naruszenia prywatności, a także na wykorzystanie możliwości Cloud DLP do weryfikacji i potencjalnego zwalniania danych.  Dzięki temu można skutecznie zminimalizować ryzyko ujawnienia PII.

Dlaczego pozostałe opcje są gorsze/błędne:

*   A.  Ograniczenie dostępu do tabel w BigQuery jest dobrym rozwiązaniem, ale nie rozwiązuje problemu źródła danych, który jest problemem w systemie. (Brak rozwiązania problemu źródła danych)
*   B.  Weryfikacja przez daty jest pomocna, ale nie identyfikuje *co* jest wrażliwe. (Brak identyfikacji źródła problemu)
*   C.  Cloud Logging może pomóc w identyfikacji potencjalnych problemów, ale nie jest wystarczającym do rozwiązania problemu, który dotyczy bezpośredniego przekazywania PII. (Brak rozwiązania problemu źródła danych)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Build a Cloud Function that reads the topics and makes a call to the Cloud Data Loss Prevention (Cloud DLP) API....

Dlaczego?

Problem: Wyciek danych PII (Personally Identifiable Information) do systemów analitycznych. Potrzeba szybkiego, skalowalnego i zarządzanego rozwiązania. 1. Cloud DLP (Data Loss Prevention): To dedykowana usługa Google Cloud do wykrywania, klasyfikowania i maskowania danych wrażliwych. Posiada gotowe detektory dla imion, adresów, numerów kart kredytowych itp. 2. Cloud Functions: Bezserwerowa (Serverless) architektura idealnie nadaje się do przetwarzania zdarzeń z Kafki (lub Pub/Sub) w locie. 3. Quarantine pattern: Opcja D proponuje inteligentne filtrowanie: jeśli DLP wykryje PII z wysoką pewnością (confidence level), dane są izolowane (quarantine) do weryfikacji, zamiast trafiać do analityki. Jest to najbezpieczniejsze podejście dla prywatności.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Authorized view in BigQuery: To rozwiązanie "na końcu" potoku. Dane PII już trafiły do chmury i do tabel (nawet jeśli są ukryte widokiem). Regulacje często wymagają, aby PII w ogóle nie były składowane w systemach analitycznych.
  • B. Third-party tool on GCE: Wymaga zarządzania infrastrukturą (VM, patchowanie, skalowanie), co narusza wymóg użycia "managed services whenever possible".
  • C. Cloud Logging: Służy do audytu i monitorowania logów systemowych, a nie do aktywnego filtrowania danych biznesowych w potoku pod kątem PII.
