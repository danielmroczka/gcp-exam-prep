You have a petabyte of analytics data and need to design a storage and processing platform for it. You must be able to perform data warehouse-style analytics on the data in Google Cloud and expose the dataset as files for batch analysis tools in other cloud providers. What should you do?

  • A. Store and process the entire dataset in BigQuery.
  • B. Store and process the entire dataset in Bigtable.

Odpowiedź: C.

Opcja C jest najlepsza, ponieważ łączy zalety BigQuery i Cloud Storage. BigQuery jest idealny do analiz typu data warehouse, oferując szybkie zapytania i skalowalność. Przechowywanie skompresowanej kopii danych w Cloud Storage umożliwia dostęp do danych przez narzędzia do analizy wsadowej w innych chmurach, spełniając wymagania zadania. Kompresja zmniejsza koszty przechowywania w Cloud Storage. To rozwiązanie zapewnia zarówno wydajność analiz w Google Cloud, jak i elastyczność w dostępie do danych poza platformą.

A. BigQuery jest świetny do analiz, ale przechowywanie petabajtów danych wyłącznie w BigQuery może być kosztowne, szczególnie dla danych rzadko używanych.
B. Bigtable jest bazą danych NoSQL, zoptymalizowaną pod kątem niskich opóźnień i dużego wolumenu zapisu, a nie analiz typu data warehouse.
D. Podział na 80% "ciepłych" i 20% "aktywnych" danych jest arbitralny i nie uwzględnia potrzeby dostępu do pełnego zestawu danych przez narzędzia zewnętrzne.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Store the full dataset in BigQuery, and store a compressed copy ... in a Cloud Storage bucket.

Dlaczego?

Wymagania: 1. Perform data warehouse-style analytics in Google Cloud: BigQuery jest standardem dla hurtowni danych w GCP i oferuje najlepszą wydajność dla zapytań analitycznych na petabajtach danych. 2. Expose the dataset as files for batch analysis tools in other cloud providers: Narzędzia w AWS czy Azure nie mogą bezpośrednio odpytywać BigQuery (bez skomplikowanych mostów). Najprostszym i najtańszym sposobem udostępniania plików dla innych dostawców jest Cloud Storage (GCS). Formaty takie jak Parquet czy Avro w GCS są uniwersalnym standardem w świecie Big Data. Kompresja kopii w GCS pozwala utrzymać niskie koszty udostępniania przy zachowaniu wysokiej wydajności analitycznej w BigQuery.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Entire dataset in BigQuery only: Nie spełnia wymogu udostępniania plików innym dostawcom chmury w sposób natywny i łatwo dostępny.
  • B. Entire dataset in Bigtable: Bigtable to baza NoSQL do szybkich odczytów klucz-wartość, nie nadaje się do analityki typu "data warehouse" (brak SQL, brak agregacji na petabajtach bez pełnego skanu).
  • D. Warm data in GCS, active in BigQuery: To podejście typu "Tiered Storage". Choć oszczędza pieniądze, narusza wymóg udostępnienia całego datasetu jako plików innym dostawcom (oni widzieliby tylko "warm data").
