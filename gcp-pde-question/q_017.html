Your company is migrating their 30-node Apache Hadoop cluster to the cloud. They want to re-use Hadoop jobs they have already created and minimize the management of the cluster as much as possible. They also want to be able to persist data beyond the life of the cluster. What should you do?

  • A. Create a Google Cloud Dataflow job to process the data.
  • B. Create a Google Cloud Dataproc cluster that uses persistent disks for HDFS.
  • C. Create a Hadoop cluster on Google Compute Engine that uses persistent disks.
  • D. Create a Cloud Dataproc cluster that uses the Google Cloud Storage connector. Most Voted
  • E. Create a Hadoop cluster on Google Compute Engine that uses Local SSD disks.

Odpowiedź: D.

Wyjaśnienie: D. Tworzenie klastra Cloud Dataproc z użyciem łącznika Google Cloud Storage (GCS) jest najlepszym rozwiązaniem, ponieważ pozwala na ponowne wykorzystanie istniejących zadań Hadoop, minimalizuje zarządzanie infrastrukturą (Dataproc jest usługą zarządzaną) oraz zapewnia trwałe przechowywanie danych w GCS, niezależnie od cyklu życia klastra. GCS zapewnia wysoką dostępność i skalowalność, co jest kluczowe w chmurze.

A. Google Cloud Dataflow to usługa przetwarzania strumieniowego i wsadowego, która może nie być bezpośrednio kompatybilna z istniejącymi zadaniami Hadoop bez znacznych modyfikacji, a pytanie kładzie nacisk na ponowne wykorzystanie istniejących zadań.
B. Klastry Dataproc domyślnie używają GCS jako systemu plików, a nie trwałych dysków dla HDFS, co oznacza, że dane nie byłyby trwale przechowywane poza życiem klastra, jeśli HDFS byłby używany jako główny magazyn.
C. Tworzenie klastra Hadoop na Google Compute Engine wymagałoby znacznie większego nakładu pracy na zarządzanie infrastrukturą w porównaniu do usługi zarządzanej jak Dataproc.
E. Używanie dysków SSD lokalnych na Google Compute Engine jest dobre dla wydajności, ale te dyski są efemeryczne i dane na nich nie przetrwają cyklu życia klastra, co jest sprzeczne z wymaganiem trwałego przechowywania danych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Create a Cloud Dataproc cluster that uses the Google Cloud Storage connector.

Dlaczego?

Wymagania to: 1. Re-use Hadoop jobs: Wymaga kompatybilności z ekosystemem Hadoop. Cloud Dataproc to zarządzany Hadoop/Spark, więc zapewnia "lift and shift" dla istniejącego kodu. 2. Minimize management: Dataproc jest usługą zarządzaną (Managed Service), w przeciwieństwie do ręcznego stawiania klastra na GCE. 3. Persist data beyond life of cluster: To kluczowy punkt. W architekturze chmurowej oddzielamy obliczenia (Compute) od składowania (Storage). Używając Cloud Storage connector (domyślny w Dataproc), dane są trzymane na GCS (tanie, trwałe), a klaster Dataproc może być efemeryczny (uruchamiany tylko na czas obliczeń i usuwany, co oszczędza pieniądze), nie tracąc danych.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Dataflow: Wymagałoby przepisania kodu (z MapReduce/Spark na Apache Beam), co narusza wymóg "re-use Hadoop jobs".
  • B. Dataproc with Persistent Disks (HDFS): Jeśli użyjesz HDFS na dyskach klastra jako głównego magazynu, usunięcie klastra usunie dane (chyba że zrobisz snapshoty, ale to uciążliwe). Nie jest to "cloud-native" podejście.
  • C & E. Hadoop on GCE: Wymaga ręcznego zarządzania serwerami (patchowanie, konfiguracja), co narusza wymóg "minimize management".
