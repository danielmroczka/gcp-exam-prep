Your organization uses a multi-cloud data storage strategy, storing data in Cloud Storage, and data in Amazon Web Services’ (AWS) S3 storage buckets. All data resides in US regions. You want to query up-to-date data by using BigQuery, regardless of which cloud the data is stored in. You need to allow users to query the tables from BigQuery without giving direct access to the data in the storage buckets. What should you do?

  • A. Setup a BigQuery Omni connection to the AWS S3 bucket data. Create BigLake tables over the Cloud Storage and S3 data and query the data using BigQuery directly. Most Voted
  • B. Set up a BigQuery Omni connection to the AWS S3 bucket data. Create external tables over the Cloud Storage and S3 data and query the data using BigQuery directly.
  • C. Use the Storage Transfer Service to copy data from the AWS S3 buckets to Cloud Storage buckets. Create BigLake tables over the Cloud Storage data and query the data using BigQuery directly.
  • D. Use the Storage Transfer Service to copy data from the AWS S3 buckets to Cloud Storage buckets. Create external tables over the Cloud Storage data and query the data using BigQuery directly.

Odpowiedź: A.

BigQuery Omni umożliwia wykonywanie zapytań na danych przechowywanych w innych chmurach, takich jak AWS S3, bez konieczności kopiowania danych do Google Cloud Storage. Połączenie BigQuery Omni z bucketami S3 pozwala na dostęp do danych w AWS, a tworzenie tabel BigLake nad danymi w Cloud Storage i S3 pozwala na ich jednoczesne i spójne queryowanie bezpośrednio z BigQuery. BigLake zapewnia granularną kontrolę dostępu i bezpieczeństwo danych, eliminując potrzebę dawania użytkownikom bezpośredniego dostępu do bucketów S3. Dzięki temu spełnione są wszystkie wymagania zadania: dostęp do aktualnych danych z obu chmur, queryowanie z BigQuery i brak bezpośredniego dostępu do danych w bucketach.

A. Opcja ta jest poprawna, ponieważ wykorzystuje BigQuery Omni i BigLake, co pozwala na queryowanie danych z różnych chmur bez bezpośredniego dostępu do nich.
B. Użycie zewnętrznych tabel zamiast BigLake ogranicza możliwości kontroli dostępu i zarządzania danymi.
C. Kopiowanie danych za pomocą Storage Transfer Service jest nieefektywne i powoduje opóźnienia w dostępie do aktualnych danych.
D. Podobnie jak w C, kopiowanie danych jest nieefektywne, a użycie zewnętrznych tabel ogranicza możliwości zarządzania danymi.

Sugerowana odpowiedź: A

Poprawną odpowiedzią jest A. Pytanie wymaga znalezienia rozwiązania, które umożliwi queryowanie danych z Cloud Storage oraz AWS S3 przez BigQuery, bez dawania użytkownikom bezpośredniego dostępu do tych danych. BigQuery Omni w połączeniu z BigLake umożliwia właśnie to.

  • BigQuery Omni pozwala na wykonywanie zapytań na danych przechowywanych w innych chmurach (w tym AWS S3) bez konieczności ich kopiowania.
  • BigLake tworzy logiczną abstrakcję (tabelę) nad danymi w Cloud Storage i S3, umożliwiając queryowanie ich za pomocą BigQuery. BigLake zapewnia również granularną kontrolę dostępu.

Dzięki temu połączeniu, użytkownicy mogą queryować dane z obu chmur za pomocą BigQuery, a dostęp do samych bucketów S3 pozostaje ograniczony, spełniając wszystkie wymagania zadania.

Dyskusja użytkowników potwierdza poprawność odpowiedzi A. Użytkownik aoifneofi_ef słusznie zauważa, że tabele zewnętrzne (external tables) mogą być tworzone tylko dla danych w Cloud Storage, BigTable lub Google Drive, co wyklucza opcję B. Użytkownik Ben_oso początkowo błędnie twierdzi, że BigLake nie obsługuje tworzenia tabel z S3, ale późniejsze komentarze (np. chickenwingz) to dementują.

Opcje C i D są nieefektywne, ponieważ wymagają kopiowania danych za pomocą Storage Transfer Service, co wprowadza opóźnienia i zwiększa koszty. Ponadto, użycie tabel zewnętrznych zamiast BigLake ogranicza możliwości kontroli dostępu.
