Your company wants to be able to retrieve large result sets of medical information from your current system, which has over 10 TBs in the database, and store the data in new tables for further query. The database must have a low-maintenance architecture and be accessible via SQL. You need to implement a cost-effective solution that can support data analytics for large result sets. What should you do?

  • A. Use Cloud SQL, but first organize the data into tables. Use JOIN in queries to retrieve data.
  • B. Use BigQuery as a data warehouse. Set output destinations for caching large queries. Most Voted
  • C. Use a MySQL cluster installed on a Compute Engine managed instance group for scalability.
  • D. Use Cloud Spanner to replicate the data across regions. Normalize the data in a series of tables.

Odpowiedź: B.

BigQuery jest idealnym rozwiązaniem dla analizy dużych zbiorów danych, takich jak 10 TB danych medycznych. Jako data warehouse, BigQuery został zaprojektowany do szybkiego przetwarzania zapytań SQL na ogromnych ilościach danych, bez konieczności zarządzania infrastrukturą. Ustawienie miejsc docelowych wyjściowych dla buforowania dużych zapytań dodatkowo optymalizuje wydajność i koszty, umożliwiając ponowne wykorzystanie wyników bez ponownego przetwarzania. BigQuery jest również wysoce skalowalny i oferuje model cenowy oparty na zużyciu, co czyni go rozwiązaniem opłacalnym dla analizy dużych zbiorów danych.  Ponadto, BigQuery automatycznie zarządza replikacją i dostępnością danych, minimalizując nakład pracy administracyjnej.

A. Cloud SQL, mimo że oferuje dostęp SQL, nie jest zoptymalizowany do obsługi zapytań na tak dużą skalę i wymaga więcej ręcznego zarządzania.
C. MySQL cluster na Compute Engine wymaga znacznego nakładu pracy administracyjnej w zakresie skalowania i utrzymania, co stoi w sprzeczności z wymaganiem niskiego nakładu pracy.
D. Cloud Spanner jest doskonały do transakcyjnych obciążeń o niskiej latencji, ale jest droższy niż BigQuery i nie jest optymalny do analizy dużych zbiorów danych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Odpowiedź B, czyli użycie BigQuery jako hurtowni danych z ustawieniem miejsc docelowych wyjściowych dla buforowania dużych zapytań, jest zdecydowanie najbardziej odpowiednią opcją w tym scenariuszu. Kluczowe wymagania to obsługa dużych zbiorów danych (10 TB), niskie koszty utrzymania oraz dostępność poprzez SQL. BigQuery został zaprojektowany specjalnie do takich zadań.

  • Skalowalność: BigQuery jest wysoce skalowalny i bezproblemowo obsługuje zbiory danych o rozmiarze 10 TB i większe.
  • Brak zarządzania infrastrukturą: Jako usługa w pełni zarządzana, BigQuery eliminuje potrzebę zarządzania infrastrukturą, co spełnia wymóg niskiego nakładu pracy administracyjnego.
  • SQL: BigQuery obsługuje standardowy SQL, umożliwiając łatwe wykonywanie zapytań na danych medycznych.
  • Buforowanie: Ustawienie miejsc docelowych wyjściowych dla buforowania zapytań optymalizuje wydajność i koszty, umożliwiając ponowne wykorzystanie wyników bez ponownego przetwarzania.
  • Koszty: Model cenowy BigQuery oparty na zużyciu jest opłacalny dla analizy dużych zbiorów danych.

Dyskusja użytkowników w pełni potwierdza tę odpowiedź. Większość użytkowników (AWSandeep, MaxNRG, zellck, TNT87, ducc) wskazuje odpowiedź B jako prawidłową, a MaxNRG szczegółowo wyjaśnia, dlaczego BigQuery jest najlepszym rozwiązaniem w porównaniu z innymi opcjami.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A (Cloud SQL): Cloud SQL nie jest zoptymalizowany do obsługi zapytań na tak dużą skalę i wymaga więcej ręcznego zarządzania.
  • C (MySQL cluster na Compute Engine): Wymaga znacznego nakładu pracy administracyjnej w zakresie skalowania i utrzymania.
  • D (Cloud Spanner): Jest droższy niż BigQuery i nie jest optymalny do analizy dużych zbiorów danych. Cloud Spanner jest bardziej odpowiedni do transakcyjnych obciążeń o niskiej latencji.
