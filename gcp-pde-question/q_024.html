You have spent a few days loading data from comma-separated values (CSV) files into the Google BigQuery table CLICK_STREAM. The column DT stores the epoch time of click events. For convenience, you chose a simple schema where every field is treated as the STRING type. Now, you want to compute web session durations of users who visit your site, and you want to change its data type to the TIMESTAMP. You want to minimize the migration effort without making future queries computationally expensive. What should you do?

  • A. Delete the table CLICK_STREAM, and then re-create it such that the column DT is of the TIMESTAMP type. Reload the data.
  • B. Add a column TS of the TIMESTAMP type to the table CLICK_STREAM, and populate the numeric values from the column TS for each row. Reference the column TS instead of the column DT from now on.
  • C. Create a view CLICK_STREAM_V, where strings from the column DT are cast into TIMESTAMP values. Reference the view CLICK_STREAM_V instead of the table CLICK_STREAM from now on.
  • D. Add two columns to the table CLICK STREAM: TS of the TIMESTAMP type and IS_NEW of the BOOLEAN type. Reload all data in append mode. For each appended row, set the value of IS_NEW to true. For future queries, reference the column TS instead of the column DT, with the WHERE clause ensuring that the value of IS_NEW must be true.
  • E. Construct a query to return every row of the table CLICK_STREAM, while using the built-in function to cast strings from the column DT into TIMESTAMP values. Run the query into a destination table NEW_CLICK_STREAM, in which the column TS is the TIMESTAMP type. Reference the table NEW_CLICK_STREAM instead of the table CLICK_STREAM from now on. Odpowiedź: E.

Wyjaśnienie: Opcja E jest najlepsza, ponieważ pozwala na przekształcenie istniejących danych bez utraty informacji i bez konieczności ponownego ładowania całego zestawu danych od podstaw. Utworzenie nowej tabeli z przekształconym typem danych dla kolumny DT minimalizuje wysiłek migracyjny, zachowując jednocześnie możliwość wykonywania zapytań bezpośrednio na nowej, poprawnie zdefiniowanej strukturze. Jest to również efektywne pod względem kosztów i wydajności w dłuższej perspektywie, ponieważ unika się narzutu związanego z funkcjami cast w każdym zapytaniu do oryginalnej tabeli.

A. Usunięcie i ponowne utworzenie tabeli wymagałoby pełnego ponownego ładowania danych, co jest czasochłonne i pracochłonne.
B. Dodanie nowej kolumny i jej ręczne wypełnienie jest czasochłonne i podatne na błędy; ponadto sugeruje zastąpienie, a nie konwersję istniejących danych.
C. Utworzenie widoku jest dobrym rozwiązaniem dla tymczasowego dostępu, ale ciągłe rzutowanie typów w widoku może wpłynąć na wydajność przyszłych zapytań, zwłaszcza przy dużych zbiorach danych.
D. Dodanie dwóch kolumn i ponowne ładowanie danych z flagą IS_NEW jest niepotrzebnie skomplikowane i nie rozwiązuje problemu konwersji typu danych w sposób bezpośredni.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: E

Prawidłową odpowiedzią jest E. Construct a query ... Run the query into a destination table NEW_CLICK_STREAM.

Dlaczego?

Problem polega na zmianie typu danych kolumny (ze STRING na TIMESTAMP) w istniejącej tabeli BigQuery. 1. BigQuery nie pozwala na bezpośrednią zmianę typu kolumny (np. ALTER TABLE ... ALTER COLUMN) w sposób, który konwertuje dane w miejscu (in-place) z String na Timestamp. 2. CTAS (Create Table As Select): Rozwiązanie E opisuje wzorzec CTAS. Wybieramy dane, przekształcamy je w locie funkcją TIMESTAMP(DT) i zapisujemy wynik do nowej tabeli. 3. Jest to najbardziej efektywne ("minimize migration effort" - jedna komenda SQL) i wydajne dla przyszłych zapytań ("without making future queries computationally expensive"), ponieważ dane w nowej tabeli będą już natywnym typem TIMESTAMP.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Delete and reload: Ponowne ładowanie danych z plików CSV ("Reload the data") jest bardzo kosztowne czasowo i operacyjnie ("You have spent a few days loading data").
  • B. Add column and update: BigQuery (w momencie pisania pytania i klasycznie) nie obsługuje klasycznego UPDATE dla każdej wiersza w sposób wydajny dla dużych datasetów (DML ma limity i koszty). Dodatkowo, trzymanie starej i nowej kolumny to marnotrawstwo.
  • C. Create a view: Widok (View) obliczałby konwersję (casting) przy każdym zapytaniu. Narusza to wymóg "without making future queries computationally expensive".
  • D. Add columns and append: Bardzo skomplikowane zarządzanie stanem (IS_NEW), dublowanie danych i konieczność filtrowania w każdym zapytaniu.
