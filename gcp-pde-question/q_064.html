You are integrating one of your internal IT applications and Google BigQuery, so users can query BigQuery from the application's interface. You do not want individual users to authenticate to BigQuery and you do not want to give them access to the dataset. You need to securely access BigQuery from your IT application. What should you do?

  • A. Create groups for your users and give those groups access to the dataset
  • B. Integrate with a single sign-on (SSO) platform, and pass each user's credentials along with the query request
  • C. Create a service account and grant dataset access to that account. Use the service account's private key to access the dataset Most Voted
  • D. Create a dummy user and grant dataset access to that user. Store the username and password for that user in a file on the files system, and use those credentials to access the BigQuery dataset

Odpowiedź: C.

Konto serwisowe (service account) to tożsamość, która jest używana przez aplikacje, a nie przez użytkowników końcowych. Utworzenie konta serwisowego i nadanie mu odpowiednich uprawnień dostępu do zbioru danych BigQuery pozwala aplikacji na uwierzytelnianie się w BigQuery w sposób programowy. Dzięki temu indywidualni użytkownicy nie muszą uwierzytelniać się bezpośrednio w BigQuery ani posiadać dostępu do samego zbioru danych, co spełnia kluczowe wymagania zadania. Aplikacja używa klucza prywatnego konta serwisowego (lub innych mechanizmów, takich jak Workload Identity) do bezpiecznego uwierzytelniania i wykonywania zapytań. Jest to standardowa, bezpieczna i rekomendowana praktyka w Google Cloud do integracji aplikacji z usługami.

A. Tworzenie grup dla użytkowników i nadawanie im dostępu do zbioru danych BigQuery jest niezgodne z wymaganiem, aby indywidualni użytkownicy nie mieli dostępu do zbioru danych.
B. Integracja z SSO i przekazywanie poświadczeń każdego użytkownika wymagałoby, aby użytkownicy bezpośrednio uwierzytelniali się w BigQuery (lub aby aplikacja ich impersonowała), co jest sprzeczne z wymaganiem, aby użytkownicy nie uwierzytelniali się w BigQuery i nie mieli dostępu do zbioru danych.
D. Tworzenie "fikcyjnego użytkownika" i przechowywanie jego nazwy użytkownika oraz hasła w pliku w systemie plików jest bardzo niebezpieczną praktyką i nie stanowi bezpiecznego rozwiązania uwierzytelniania aplikacji.

*   Uwierzytelnianie jako konto serwisowe: https://cloud.google.com/docs/authentication/production#service-accounts

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Create a service account and grant dataset access to that account....

Dlaczego?

Wymagania: 1. "Integrate internal IT application and Google BigQuery" (aplikacja wykonuje zapytania). 2. "Do not want individual users to authenticate" (użytkownicy nie mają własnych kont GCP/BQ). 3. "Do not want to give them access to the dataset" (użytkownik nie powinien widzieć danych bezpośrednio, tylko przez aplikację). Konto serwisowe (Service Account) to tożsamość dla aplikacji (maszyny), a nie dla człowieka. Aplikacja używa pliku klucza (JSON) do uwierzytelnienia się jako Service Account. Service Account ma nadane uprawnienia do BigQuery. Użytkownik loguje się do aplikacji (np. przez LDAP firmowy), a aplikacja wykonuje zapytanie do BQ w imieniu swoim (Service Account), a nie użytkownika. To klasyczny wzorzec "Proxy authentication".

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Create groups for users: To dałoby dostęp użytkownikom ("give them access"), co jest sprzeczne z wymaganiami.
  • B. SSO and pass credentials: To oznacza, że użytkownik musi mieć dostęp do BQ (nawet jeśli przez SSO). Narusza wymóg braku bezpośredniego dostępu.
  • D. Dummy user + password file: Przechowywanie hasła w pliku tekstowym to fundamentalny błąd bezpieczeństwa. Konta użytkowników (User Accounts) nie są przeznaczone do automatyzacji aplikacji.
