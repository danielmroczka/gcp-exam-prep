You operate a logistics company, and you want to improve event delivery reliability for vehicle-based sensors. You operate small data centers around the world to capture these events, but leased lines that provide connectivity from your event collection infrastructure to your event processing infrastructure are unreliable, with unpredictable latency. You want to address this issue in the most cost-effective way. What should you do?

  • A. Deploy small Kafka clusters in your data centers to buffer events.
  • B. Have the data acquisition devices publish data to Cloud Pub/Sub. Most Voted
  • C. Establish a Cloud Interconnect between all remote data centers and Google.
  • D. Write a Cloud Dataflow pipeline that aggregates all data in session windows.

Wyjaśnienie: Wybór B jest najlepszy, ponieważ zapewnia najbardziej efektywną i kosztowo opisaną rozwiązanie problemu z niezawodną łącznością. Deploying small Kafka clusters in your data centers acts as a buffer, absorbing the intermittent latency issues with events. This reduces the impact of unreliable leased lines, allowing for more reliable data capture and processing.  Alternatywy A, C i D są mniej efektywne, ponieważ wymagają złożonych konfiguracji, mogą być drogie, a także nie rozwiązują problemu z niezawodnością połączenia.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Have the data acquisition devices publish data to Cloud Pub/Sub.

Dlaczego?

Wymagania: "improve event delivery reliability", "unreliable leased lines", "unpredictable latency", "most cost-effective". Cloud Pub/Sub jest globalnym, w pełni zarządzanym systemem kolejkowania wiadomości, który idealnie rozwiązuje problem niestabilnych połączeń sieciowych: 1. Buffering: Jeśli łącze do centrum przetwarzania jest chwilowo zerwane lub wolne, urządzenia mogą publikować dane do Pub/Sub, który przechowa je bezpiecznie (do 7 dni) do momentu ich pobrania przez system przetwarzający. 2. Reliability: Pub/Sub gwarantuje dostarczenie wiadomości "at least once". 3. Cost-effective: Nie wymaga stawiania i utrzymywania własnych serwerów (jak Kafka w opcji A) ani drogich, dedykowanych łączy (jak Interconnect w opcji C). Płacisz tylko za zużycie (ilość danych).

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Kafka clusters in data centers: Wymagałoby to zakupu i zarządzania infrastrukturą (hardware/VM) w wielu lokalizacjach na świecie, co jest kosztowne i trudne operacyjnie. Pub/Sub oferuje tę samą funkcjonalność jako usługa (SaaS).
  • C. Cloud Interconnect: To najdroższe rozwiązanie. Dedykowane łącza fizyczne są skuteczne, ale ich koszt dla wielu małych, rozproszonych centrów danych byłby ogromny i nieuzasadniony przy dostępności Pub/Sub.
  • D. Dataflow pipeline: To rozwiązanie do przetwarzania danych, a nie do ich dostarczania przez zawodne łącza. Jeśli dane nie dotrą do chmury przez słabą sieć, Dataflow ich nie przetworzy.
