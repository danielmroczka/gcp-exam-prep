You have a query that filters a BigQuery table using a WHERE clause on timestamp and ID columns. By using bq query `"-dry_run you learn that the query triggers a full scan of the table, even though the filter on timestamp and ID select a tiny fraction of the overall data. You want to reduce the amount of data scanned by BigQuery with minimal changes to existing SQL queries. What should you do?

  • A. Create a separate table for each ID.
  • B. Use the LIMIT keyword to reduce the number of rows returned.
  • C. Recreate the table with a partitioning column and clustering column. Most Voted
  • D. Use the bq query --maximum_bytes_billed flag to restrict the number of bytes billed.

Odpowiedź: C

Wyjaśnienie: Użycie `LIMIT` jest dobrym sposobem na ograniczenie liczby zwracanych rekordów w zapytaniu BigQuery, ale nie rozwiązuje problemu z pełnym skanowaniem.  Prawidłowe rozwiązanie polega na zoptymalizowaniu zapytania, aby zmniejszyć ilość danych, które BigQuery musi przetworzyć.  Zamiast pełnego skanowania, można użyć `WHERE` klauzuli, aby filtrować tylko te rekordy, które są istotne dla zapytania.  Ponadto, tworzenie tabeli dla każdego ID jest niepotrzebne, ponieważ BigQuery automatycznie tworzy indeksy na podstawie danych w tabeli.  Ponadto, ograniczenie liczby maksymalnych danych pobranych ( `bq query --maximum_bytes_billed` ) nie jest odpowiednim rozwiązaniem, ponieważ nie wpływa na wydajność zapytania, a jedynie na koszty.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Poprawną odpowiedzią jest C: Recreate the table with a partitioning column and clustering column. Problem polega na tym, że zapytanie wykonuje pełne skanowanie tabeli, mimo że filtry na kolumnach timestamp i ID powinny ograniczać ilość danych. Partycjonowanie i grupowanie (clustering) tabeli w BigQuery pozwala na znaczące zmniejszenie ilości danych skanowanych przez zapytanie. Partycjonowanie dzieli tabelę na mniejsze segmenty na podstawie wartości kolumny partycjonującej (np. data), a grupowanie sortuje dane wewnątrz partycji, co przyspiesza filtrowanie i agregację.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość głosów i komentarzy wskazuje na C jako prawidłowe rozwiązanie, powołując się na dokumentację Google Cloud (https://cloud.google.com/bigquery/docs/partitioned-tables, https://cloud.google.com/bigquery/docs/clustered-tables, https://cloud.google.com/bigquery/docs/best-practices-costs). Użytkownicy słusznie zauważają, że:

  • LIMIT ogranicza tylko liczbę zwracanych wierszy, ale nie zmniejsza ilości danych skanowanych.
  • Tworzenie oddzielnej tabeli dla każdego ID jest niepraktyczne i nieefektywne.
  • Użycie flagi bq query --maximum_bytes_billed ogranicza koszty, ale nie poprawia wydajności zapytania.

Argumenty przeciwko odpowiedzi C, sugerujące, że wymaga ona zmiany tabeli, a nie samego zapytania SQL, są błędne. Ponowne utworzenie tabeli z partycjonowaniem i grupowaniem nie wymaga zmian w istniejących zapytaniach SQL, które odczytują z tej tabeli – zapytania nadal będą działać, ale będą skanować znacznie mniej danych.
