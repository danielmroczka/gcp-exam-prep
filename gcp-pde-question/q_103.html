You have a data stored in BigQuery. The data in the BigQuery dataset must be highly available. You need to define a storage, backup, and recovery strategy of this data that minimizes cost. How should you configure the BigQuery table that have a recovery point objective (RPO) of 30 days?

  • A. Set the BigQuery dataset to be regional. In the event of an emergency, use a point-in-time snapshot to recover the data.
  • B. Set the BigQuery dataset to be regional. Create a scheduled query to make copies of the data to tables suffixed with the time of the backup. In the event of an emergency, use the backup copy of the table.
  • C. Set the BigQuery dataset to be multi-regional. In the event of an emergency, use a point-in-time snapshot to recover the data. Most Voted

Odpowiedź: C

Wyjaśnienie: Strategia zdefiniowana w opcji C jest najlepsza, ponieważ minimalizuje koszty przechowywania danych w BigQuery, zapewniając minimalny Punkt Odbicia Danych (RPO) 30 dni. Użycie punktowego snapshotu w przypadku awarii zapewnia szybką możliwość odzyskania danych, co jest kluczowe dla utrzymania dostępności.  Alternatywy A i B, czyli regionalne ustawienia i skrypty do tworzenia kopii zapasowych, są mniej efektywne, ponieważ wymagają dodatkowego czasu i zasobów, a także mogą prowadzić do zwiększenia kosztów przechowywania danych.  Opcja D, choć również zapewnia możliwość odzyskania danych, jest mniej elastyczna i może być mniej efektywna w przypadku awarii, ponieważ wymaga ręcznego tworzenia kopii zapasowych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią (uwzględniającą wymóg RPO 30 dni) jest B. Set the BigQuery dataset to be regional. Create a scheduled query to make copies of the data....

Dlaczego?

1. RPO of 30 days: To jest kluczowe ograniczenie. BigQuery posiada wbudowaną funkcję Time Travel, która pozwala odpytać dane z przeszłości, ale jej limit to 7 dni. Opcja C i A (point-in-time snapshot / time travel) nie pozwolą na odzyskanie danych sprzed np. 20 dni. Aby zapewnić RPO 30 dni, musisz mieć własną strategię backupu (np. kopie tabel z sufiksem daty). 2. Minimizes cost: Dataset Regionalny jest tańszy w składowaniu i przetwarzaniu niż Multi-regionalny. Skoro priorytetem jest koszt, wybieramy "regional". 3. High Availability: BigQuery nawet w wersji regionalnej oferuje bardzo wysoką dostępność (rezyduje w wielu strefach wewnątrz regionu).

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A i C. Point-in-time snapshot: Jak wspomniano, standardowy mechanizm odzyskiwania BigQuery (snapshots/time travel) ma limit 7 dni. Nie spełnia wymogu RPO 30 dni.
  • D. Multi-regional: Jest droższy niż regionalny, co narusza wymóg minimalizacji kosztów. Choć zapewnia wyższą odporność na awarię całego regionu, regionalny jest wystarczający dla większości standardów HA w chmurze.
