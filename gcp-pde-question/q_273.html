You are creating the CI/CD cycle for the code of the directed acyclic graphs (DAGs) running in Cloud Composer. Your team has two Cloud Composer instances: one instance for development and another instance for production. Your team is using a Git repository to maintain and develop the code of the DAGs. You want to deploy the DAGs automatically to Cloud Composer when a certain tag is pushed to the Git repository. What should you do?

  • A. 1. Use Cloud Build to copy the code of the DAG to the Cloud Storage bucket of the development instance for DAG testing.
    2. If the tests pass, use Cloud Build to copy the code to the bucket of the production instance. Most Voted
  • B. 1. Use Cloud Build to build a container with the code of the DAG and the KubernetesPodOperator to deploy the code to the Google Kubernetes Engine (GKE) cluster of the development instance for testing.
    2. If the tests pass, use the KubernetesPodOperator to deploy the container to the GKE cluster of the production instance.
  • C. 1. Use Cloud Build to build a container and the KubernetesPodOperator to deploy the code of the DAG to the Google Kubernetes Engine (GKE) cluster of the development instance for testing.
    2. If the tests pass, copy the code to the Cloud Storage bucket of the production instance.
  • D. 1. Use Cloud Build to copy the code of the DAG to the Cloud Storage bucket of the development instance for DAG testing.
    2. If the tests pass, use Cloud Build to build a container with the code of the DAG and the KubernetesPodOperator to deploy the container to the Google Kubernetes Engine (GKE) cluster of the production instance.

Odpowiedź: A.

Cloud Composer opiera się na Cloud Storage do przechowywania DAGów. Automatyzacja wdrażania DAGów po wypchnięciu tagu w repozytorium Git najlepiej realizuje się poprzez kopiowanie kodu do odpowiednich bucketów Cloud Storage dla instancji development i production. Cloud Build idealnie nadaje się do automatyzacji tego procesu, uruchamiając skrypty kopiujące kod po wykryciu odpowiedniego tagu. Testy DAGów można przeprowadzić na instancji development, a po ich pomyślnym zakończeniu, kod można skopiować do instancji production. To proste i efektywne rozwiązanie, które wykorzystuje natywne możliwości Cloud Composer.

A. Jest to najprostsze i najbardziej bezpośrednie rozwiązanie, wykorzystujące podstawową funkcjonalność Cloud Composer.
B. Użycie KubernetesPodOperator i GKE jest zbędnie skomplikowane dla wdrażania DAGów w Cloud Composer, który nie wymaga bezpośredniej interakcji z GKE.
C. Mieszanie kopiowania do Cloud Storage z użyciem KubernetesPodOperator jest niekonsekwentne i nie wykorzystuje w pełni możliwości Cloud Composer.
D. Użycie KubernetesPodOperator do wdrożenia na GKE w instancji production jest niepotrzebne i komplikuje proces wdrażania DAGów w Cloud Composer.

Sugerowana odpowiedź: A

Odpowiedź A jest prawidłowa, ponieważ Cloud Composer wykorzystuje Cloud Storage do przechowywania DAGów. Najprostszym i najbardziej efektywnym sposobem automatyzacji wdrażania jest kopiowanie kodu DAGów do odpowiednich bucketów Cloud Storage dla środowisk development i production. Cloud Build idealnie nadaje się do tego zadania, uruchamiając skrypty kopiujące kod po wykryciu odpowiedniego tagu w repozytorium Git. Testy DAGów przeprowadzane są na instancji development, a po ich pomyślnym zakończeniu, kod jest kopiowany do instancji production.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość użytkowników (Matt_108, Sofiia98, Smakyel79, JyoGCP, BIGQUERY_ALT_ALT) wskazuje odpowiedź A jako prawidłową, argumentując, że DAGi są przechowywane w Cloud Storage, a Cloud Build służy do automatyzacji procesu wdrażania. Nawet użytkownik GCP001, który początkowo wskazywał odpowiedź C, ostatecznie przyznał, że A jest poprawna.

Odpowiedzi B, C i D są nieprawidłowe, ponieważ wprowadzają niepotrzebne komplikacje związane z Google Kubernetes Engine (GKE). Cloud Composer nie wymaga bezpośredniej interakcji z GKE do wdrażania DAGów. Użycie KubernetesPodOperator i GKE jest zbędne i nie wykorzystuje w pełni możliwości Cloud Composer.

Link do dokumentacji Cloud Composer i Cloud Build, podany przez Matt_108, potwierdza, że Cloud Build może być używany do automatyzacji wdrażania DAGów poprzez kopiowanie kodu do bucketów Cloud Storage.
