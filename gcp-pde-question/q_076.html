Government regulations in your industry mandate that you have to maintain an auditable record of access to certain types of data. Assuming that all expiring logs will be archived correctly, where should you store data that is subject to that mandate?

  • A. Encrypted on Cloud Storage with user-supplied encryption keys. A separate decryption key will be given to each authorized user.
  • B. In a BigQuery dataset that is viewable only by authorized personnel, with the Data Access log used to provide the auditability. Most Voted
  • C. In Cloud SQL, with separate database user names to each user. The Cloud SQL Admin activity logs will be used to provide the auditability.
  • D. In a bucket on Cloud Storage that is accessible only by an AppEngine service that collects user information and logs the access before providing a link to the bucket.

Odpowiedź: B

Wyjaśnienie: Regulacje rządowe wymagające audytu dostępu do danych są bardzo powszechne w wielu branżach, a ich przestrzeganie wymaga solidnej dokumentacji.  Zapewnienie audytowalnej historii dostępu do danych jest kluczowe dla zgodności z przepisami i zapewnienia bezpieczeństwa.  Przedstawiona opcja B, czyli przechowywanie danych w BigQuery z logami dostępu, jest najbardziej odpowiednia, ponieważ BigQuery jest zaprojektowany do przechowywania i analizowania danych, a logi dostępu są integralną częścią jego funkcjonowania.  Użycie widocznych tylko dla uprawnionych użytkowników i logów dostępu zapewnia, że dostęp do danych jest śledzony i monitorowany, a także pozwala na weryfikację, kto i kiedy uzyskał dostęp do danych.  Opcje A i C nie są odpowiednie, ponieważ nie zapewniają automatycznego audytu dostępu, a opcja D jest nieefektywna i nie spełnia wymagań dotyczących audytu.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. In a BigQuery dataset... with the Data Access log used to provide the auditability.

Dlaczego?

Wymaganie: "maintain an auditable record of access to certain types of data". Cloud Audit Logs (wcześniej Stackdriver) dla BigQuery automatycznie rejestrują zdarzenia "Data Access" (kto czytał dane, jakie zapytanie wykonał). Jest to natywna, wbudowana i niezmienna (immutable) funkcja, która spełnia wymogi regulacyjne (audit compliance). Przechowywanie danych w BQ z włączonym logowaniem (które jest domyślne) jest najprostszym i najbezpieczniejszym sposobem na spełnienie tego wymogu.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Encrypted on GCS with keys: Szyfrowanie to nie audyt. Choć chroni dane, nie daje automatycznego raportu "kto i kiedy uzyskał dostęp" w tak ustrukturyzowany sposób jak logi BQ. GCS ma logi, ale zarządzanie kluczami per user jest trudne operacyjnie.
  • C. Cloud SQL Admin activity logs: "Admin activity logs" rejestrują zmiany konfiguracji (np. create instance), a nie każde zapytanie SELECT (Data Access). Aby audytować każdy odczyt w Cloud SQL, trzeba włączyć dodatkowe flagi (audit plugin), co jest bardziej skomplikowane i wpływa na wydajność. BigQuery robi to natywnie.
  • D. AppEngine middleware: Budowanie własnego middleware ("AppEngine service") do logowania to antywzorzec (tzw. "rolling your own crypto/audit"). Jest podatne na błędy programistyczne i luki bezpieczeństwa. Należy używać wbudowanych funkcji platformy.
