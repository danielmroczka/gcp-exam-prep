You are deploying a MySQL database workload onto Cloud SQL. The database must be able to scale up to support several readers from various geographic regions. The database must be highly available and meet low RTO and RPO requirements, even in the event of a regional outage. You need to ensure that interruptions to the readers are minimal during a database failover. What should you do?

  • A. Create a highly available Cloud SQL instance in region Create a highly available read replica in region B. Scale up read workloads by creating cascading read replicas in multiple regions. Backup the Cloud SQL instances to a multi-regional Cloud Storage bucket. Restore the Cloud SQL backup to a new instance in another region when Region A is down.
  • B. Create a highly available Cloud SQL instance in region A. Scale up read workloads by creating read replicas in multiple regions. Promote one of the read replicas when region A is down.

Odpowiedź: C.

Opcja C jest poprawna, ponieważ łączy w sobie elementy zapewniające wysoką dostępność, skalowalność i minimalne przestoje podczas awarii. Utworzenie wysoce dostępnej instancji Cloud SQL w regionie A zapewnia podstawę dla odporności na awarie w obrębie tego regionu. Dodanie wysoce dostępnej repliki odczytu w regionie B umożliwia obsługę zapytań odczytu z różnych lokalizacji geograficznych i stanowi gotowe zapasowe rozwiązanie w przypadku awarii regionu A.  Kaskadowe repliki odczytu w wielu regionach dodatkowo zwiększają skalowalność i wydajność odczytu. W przypadku awarii regionu A, promowanie repliki odczytu w regionie B do roli głównej minimalizuje przestoje dla aplikacji.

A.  Brak skalowalności odczytu w wielu regionach i ręczne przywracanie z kopii zapasowej powoduje zbyt długi czas przestoju.
B.  Brak kaskadowych replik odczytu ogranicza skalowalność, a promowanie repliki odczytu nie zapewnia automatycznego failoveru.
D.  Replikacja w tym samym regionie nie chroni przed awarią regionalną, a failover do instancji standby nie minimalizuje przestojów tak efektywnie jak promowanie repliki w innym regionie.

Sugerowana odpowiedź: C

Odpowiedź C jest najbardziej kompleksowa i spełnia wszystkie wymagania zadania. Kluczowe jest zapewnienie wysokiej dostępności, skalowalności oraz minimalnego czasu przestoju podczas awarii. Wysoce dostępna instancja Cloud SQL w regionie A zapewnia odporność na awarie w obrębie tego regionu. Dodanie wysoce dostępnej repliki odczytu w regionie B umożliwia obsługę zapytań odczytu z różnych lokalizacji geograficznych i stanowi gotowe zapasowe rozwiązanie w przypadku awarii regionu A. Kaskadowe repliki odczytu w wielu regionach dodatkowo zwiększają skalowalność i wydajność odczytu. Promowanie repliki odczytu w regionie B do roli głównej w przypadku awarii regionu A minimalizuje przestoje.

Dyskusja użytkowników potwierdza poprawność odpowiedzi C, z przewagą głosów (83%). Użytkownicy słusznie wskazują, że opcja C łączy w sobie wysoką dostępność, skalowalność i szybkie przełączanie w przypadku awarii.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Przywracanie z kopii zapasowej jest zbyt czasochłonne i nie spełnia wymagań niskiego RTO (Recovery Time Objective).
  • B: Brak wysoce dostępnej repliki odczytu w regionie B osłabia odporność na awarie. Samo skalowanie odczytu nie gwarantuje wysokiej dostępności.
  • D: Replikacja w tym samym regionie nie chroni przed awarią regionalną. Failover do instancji standby w tym samym regionie nie minimalizuje przestojów tak efektywnie jak promowanie repliki w innym regionie.
