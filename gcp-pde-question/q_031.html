You have Google Cloud Dataflow streaming pipeline running with a Google Cloud Pub/Sub subscription as the source. You need to make an update to the code that will make the new Cloud Dataflow pipeline incompatible with the current version. You do not want to lose any data when making this update. What should you do?

  • A. Update the current pipeline and use the drain flag. Most Voted

  • C. Create a new pipeline that has the same Cloud Pub/Sub subscription and cancel the old pipeline.
  • D. Create a new pipeline that has a new Cloud Pub/Sub subscription and cancel the old pipeline.

Odpowiedź: A.

Wyjaśnienie: Użycie flagi `drain` podczas aktualizacji potoku Dataflow zapewnia, że istniejący potok zakończy przetwarzanie wszystkich elementów znajdujących się w buforze i w transakcjach, zanim zostanie wyłączony. Dzięki temu żadne dane nie zostaną utracone. Po tym, jak stary potok przetworzy wszystko, można uruchomić nowy, niekompatybilny potok. Jest to standardowa procedura Google Cloud do zapewnienia bezstratnych aktualizacji potoków strumieniowych.

A. Opcja A jest poprawna, ponieważ flaga `drain` gwarantuje bezstratne zakończenie działania starego potoku.
B. Opcja B jest błędna; transform mapping JSON jest używany do migracji stanów pomiędzy niekompatybilnymi zmianami, ale `drain` jest prostszym i bardziej bezpośrednim sposobem na zapewnienie, że wszystkie dane zostaną przetworzone.
C. Opcja C jest błędna; jeśli nowy potok używa tej samej subskrypcji Pub/Sub, to dane mogą zostać utracone lub zduplikowane, jeśli nie zostaną prawidłowo zaimplementowane mechanizmy zarządzania stanem i gwarancji dostarczenia.
D. Opcja D jest błędna; utworzenie nowej subskrypcji Pub/Sub spowoduje, że nowy potok nie będzie przetwarzał danych, które zostały wysłane do starej subskrypcji przed jego uruchomieniem, co oznacza utratę danych.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Prawidłową odpowiedzią jest A. Update the current pipeline and use the drain flag.

Dlaczego?

Pytanie dotyczy aktualizacji potoku strumieniowego (Streaming Pipeline) do nowej, niekompatybilnej wersji kodu, przy wymogu braku utraty danych. Flaga --drain (Drain) powoduje, że Dataflow:

  1. Przestaje pobierać nowe dane ze źródła (Pub/Sub).
  2. Dokańcza przetwarzanie wszystkich danych, które są już "w locie" (in-flight) lub zbuforowane w oknach.
  3. Zamyka potok w sposób czysty.

Dzięki temu stary (niekompatybilny) potok kończy pracę bez utraty danych. Następnie można uruchomić nowy potok z nowym kodem. Pub/Sub przechowuje niepobrane wiadomości (do 7 dni), więc nowy potok wznowi pobieranie od momentu, gdzie stary przestał (ack), o ile używa tej samej subskrypcji (lub nowej, jeśli chcemy startować od zera, ale tu chodzi o ciągłość).

Uwaga o opcji B (Transform Mapping): Transform Mapping JSON używany jest przy aktualizacji (--update) kompatybilnego lub lekko zmienionego potoku, aby zmapować nazwy kroków (steps) i zachować stan (state/watermarks). Pytanie mówi wyraźnie, że nowy potok jest niekompatybilny. W takim przypadku aktualizacja in-place (przez --update) często się nie uda. Drain jest bezpieczniejszy dla "breaking changes".

Dlaczego inne odpowiedzi są nieprawidłowe?

  • B. Transform mapping: Służy do aktualizacji działającego potoku (Update), ale przy zmianach niekompatybilnych (np. zmiana topologii grafu, zmiana typu kodowania) update może się nie powieść. Drain jest metodą na "zatrzymaj i uruchom nowy".
  • C. Cancel old pipeline: Cancel natychmiast zabija workerów. Dane będące w trakcie przetwarzania (w pamięci RAM workerów) zostaną utracone (lub w najlepszym razie niepotwierdzone w Pub/Sub i wysłane ponownie, co może powodować duplikaty, ale w Dataflow "Cancel" jest uważane za "twardy stop").
  • D. New subscription: Utworzenie nowej subskrypcji oznacza, że stracimy wiadomości, które napłynęły do starej subskrypcji, a nie zostały jeszcze przetworzone przez stary potok.
