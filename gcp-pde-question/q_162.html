You want to archive data in Cloud Storage. Because some data is very sensitive, you want to use the `Trust No One` (TNO) approach to encrypt your data to prevent the cloud provider staff from decrypting your data. What should you do?

  • A. Use gcloud kms keys create to create a symmetric key. Then use gcloud kms encrypt to encrypt each archival file with the key and unique additional authenticated data (AAD). Use gsutil cp to upload each encrypted file to the Cloud Storage bucket, and keep the AAD outside of Google Cloud.
  • B. Use gcloud kms keys create to create a symmetric key. Then use gcloud kms encrypt to encrypt each archival file with the key. Use gsutil cp to upload each encrypted file to the Cloud Storage bucket. Manually destroy the key previously used for encryption, and rotate the key once.
  • C. Specify customer-supplied encryption key (CSEK) in the .boto configuration file. Use gsutil cp to upload each archival file to the Cloud Storage bucket. Save the CSEK in Cloud Memorystore as permanent storage of the secret.
  • D. Specify customer-supplied encryption key (CSEK) in the .boto configuration file. Use gsutil cp to upload each archival file to the Cloud Storage bucket. Save the CSEK in a different project that only the security team can access. Most Voted

Odpowiedź: D.

Podejście "Trust No One" (TNO) wymaga, aby kontrolę nad kluczami szyfrującymi posiadał wyłącznie klient, uniemożliwiając dostawcy chmury dostęp do danych. Opcja D realizuje to poprzez użycie klucza szyfrującego dostarczonego przez klienta (CSEK) i przechowywanie go w innym projekcie, do którego dostęp ma tylko zespół bezpieczeństwa. Dzięki temu Google nie ma możliwości odszyfrowania danych. Użycie CSEK zapewnia, że szyfrowanie i deszyfrowanie odbywa się wyłącznie po stronie klienta, co jest kluczowe dla TNO. Przechowywanie klucza w oddzielnym projekcie dodatkowo zwiększa bezpieczeństwo, ograniczając potencjalny dostęp do niego.

A. Użycie KMS z AAD nie zapewnia pełnej kontroli klienta nad kluczem, a przechowywanie AAD poza Google Cloud może być problematyczne w zarządzaniu.
B. Zniszczenie i rotacja klucza KMS nie eliminuje możliwości dostępu do danych przez Google w momencie szyfrowania.
C. Przechowywanie CSEK w Cloud Memorystore, mimo że jest bezpieczne, nadal oznacza przechowywanie klucza w infrastrukturze Google, co narusza zasadę TNO.

Dodatkowe źródła:

*   **Google Cloud Storage Encryption Options:** https://cloud.google.com/storage/docs/encryption-options
*   **Customer-Supplied Encryption Keys (CSEK):** https://cloud.google.com/storage/docs/using-cseks
*   **Trust No One (TNO) Security Model:** https://www.akamai.com/blog/security/trust-no-one-security-model (choć nie specyficzne dla Google Cloud, wyjaśnia koncepcję TNO)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Podejście "Trust No One" (TNO) zakłada, że klient ma pełną kontrolę nad kluczami szyfrującymi, uniemożliwiając dostawcy chmury dostęp do odszyfrowanych danych. Opcja D najlepiej realizuje tę zasadę poprzez wykorzystanie kluczy szyfrujących dostarczonych przez klienta (CSEK) i przechowywanie ich w oddzielnym projekcie, do którego dostęp ma wyłącznie zespół bezpieczeństwa. W ten sposób Google nie ma możliwości odszyfrowania danych, ponieważ nie posiada klucza. Użycie CSEK zapewnia, że szyfrowanie i deszyfrowanie odbywa się wyłącznie po stronie klienta.

Dyskusja użytkowników w większości skłania się ku odpowiedzi D, szczególnie nowsze komentarze (z ostatnich kilku miesięcy) oraz te z większą liczbą głosów. Użytkownicy podkreślają, że CSEK jest kluczowy dla TNO, ponieważ klucze nie są przechowywane w infrastrukturze Google. Argumenty te są zgodne z definicją TNO i dokumentacją Google Cloud.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Użycie KMS z AAD, choć zwiększa bezpieczeństwo, nadal polega na infrastrukturze Google do zarządzania kluczami. AAD samo w sobie nie zapewnia pełnej kontroli klienta nad kluczem, a przechowywanie AAD poza Google Cloud może być problematyczne w zarządzaniu i nie eliminuje ryzyka dostępu Google do klucza KMS.
  • B: Rotacja klucza KMS nie eliminuje możliwości dostępu do danych przez Google w momencie szyfrowania.
  • C: Przechowywanie CSEK w Cloud Memorystore, mimo że jest to usługa bezpieczna, nadal oznacza przechowywanie klucza w infrastrukturze Google, co narusza zasadę TNO.

Podsumowując, odpowiedź D jest jedyną, która w pełni spełnia wymagania podejścia "Trust No One", zapewniając, że Google nie ma dostępu do kluczy szyfrujących i nie może odszyfrować danych.
