You are building new real-time data warehouse for your company and will use Google BigQuery streaming inserts. There is no guarantee that data will only be sent in once but you do have a unique ID for each row of data and an event timestamp. You want to ensure that duplicates are not included while interactively querying data. Which query type should you use?

  • A. Include ORDER BY DESK on timestamp column and LIMIT to 1.
  • B. Use GROUP BY on the unique ID column and timestamp column and SUM on the values.
  • C. Use the LAG window function with PARTITION by unique ID along with WHERE LAG IS NOT NULL.
  • D. Use the ROW_NUMBER window function with PARTITION by unique ID along with WHERE row equals 1. Most Voted

Odpowiedź: D.

Wyjaśnienie: Użycie funkcji okienkowej `ROW_NUMBER()` z partycjonowaniem po unikalnym identyfikatorze i wyborem pierwszego wiersza (`WHERE row_number = 1`) pozwala na zidentyfikowanie i zachowanie tylko jednej instancji każdej unikalnej danej, skutecznie eliminując duplikaty. Jest to standardowe i wydajne podejście do usuwania duplikatów w zapytaniach SQL, szczególnie gdy mamy unikalny identyfikator. Ta metoda zapewnia, że dla każdego unikalnego ID wybierana jest konkretna wiersz, co w połączeniu z partycjonowaniem zapobiega usunięciu wartościowych danych.

A. `ORDER BY` i `LIMIT` mogą usunąć duplikaty, ale nie gwarantują wybrania prawidłowego lub najnowszego wpisu, jeśli dane nie są idealnie posortowane.
B. `GROUP BY` z funkcją `SUM` agreguje wartości, a nie usuwa duplikaty wierszy, co nie jest celem w tym przypadku.
C. Funkcja `LAG` służy do porównywania obecnego wiersza z poprzednim i nie jest bezpośrednio przeznaczona do usuwania duplikatów w ten sposób, jak `ROW_NUMBER()`.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Prawidłową odpowiedzią jest D. Use the ROW_NUMBER window function....

Dlaczego?

BigQuery podczas `streaming inserts` nie gwarantuje unikalności (możliwe są duplikaty "at-least-once delivery"). Aby usunąć duplikaty w czasie odczytu (deduplication at query time), standardowym wzorcem jest użycie funkcji analitycznej (okienkowej) ROW_NUMBER().

Zapytanie wyglądałoby mniej więcej tak:

SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER(PARTITION BY unique_id ORDER BY event_timestamp DESC) as rn
  FROM dataset.table
)
WHERE rn = 1

PARTITION BY unique_id grupuje te same rekordy, ORDER BY event_timestamp DESC (domyślny w praktyce, choć w odpowiedzi nie wspomniany, jest kluczowy dla wybrania najnowszego) ustawia je w kolejności, a ROW_NUMBER() numeruje. Wybranie wiersza numer 1 daje nam jeden, unikalny rekord.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. ORDER BY and LIMIT 1: Zwróci tylko jeden wiersz z całej tabeli, a nie jeden unikalny wiersz dla każdego ID.
  • B. GROUP BY and SUM: To agregacja (sumowanie wartości), a nie deduplikacja (wybranie jednego reprezentanta). Zmieni to kształt danych.
  • C. LAG function: Służy do pobierania wartości z poprzedniego wiersza (np. do obliczania różnic delta). Nie jest przeznaczona do filtrowania duplikatów w prosty sposób.
