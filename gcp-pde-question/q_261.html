You want to migrate your existing Teradata data warehouse to BigQuery. You want to move the historical data to BigQuery by using the most efficient method that requires the least amount of programming, but local storage space on your existing data warehouse is limited. What should you do?

  • A. Use BigQuery Data Transfer Service by using the Java Database Connectivity (JDBC) driver with FastExport connection. Most Voted
  • B. Create a Teradata Parallel Transporter (TPT) export script to export the historical data, and import to BigQuery by using the bq command-line tool.
  • C. Use BigQuery Data Transfer Service with the Teradata Parallel Transporter (TPT) tbuild utility.
  • D. Create a script to export the historical data, and upload in batches to Cloud Storage. Set up a BigQuery Data Transfer Service instance from Cloud Storage to BigQuery.

Odpowiedź: A.

BigQuery Data Transfer Service (DTS) z JDBC i FastExport to najbardziej efektywny sposób migracji danych z Teradata do BigQuery, spełniający podane wymagania. FastExport jest zoptymalizowany do szybkiego eksportu danych z Teradata, a DTS automatyzuje proces transferu do BigQuery, minimalizując potrzebę pisania skryptów. JDBC driver umożliwia bezpośrednie połączenie z Teradata, a DTS zajmuje się zarządzaniem transferem i potencjalnymi problemami z siecią.  Ponieważ lokalna przestrzeń dyskowa jest ograniczona, eksport bezpośrednio do BigQuery przez DTS jest lepszy niż tworzenie plików tymczasowych.  DTS oferuje również monitorowanie i ponawianie prób w przypadku błędów, co zwiększa niezawodność migracji.

B. Użycie TPT i bq command-line tool wymaga więcej ręcznej konfiguracji i skryptów, co jest sprzeczne z wymaganiem minimalnej ilości programowania.
C. DTS nie współpracuje bezpośrednio z TPT tbuild utility, a próba integracji byłaby bardziej skomplikowana niż użycie JDBC.
D. Eksport do Cloud Storage w partiach i następnie import do BigQuery przez DTS jest mniej efektywny niż bezpośredni transfer z Teradata przez DTS z JDBC i FastExport, ponieważ dodaje dodatkowy krok i potencjalne opóźnienia.

Sugerowana odpowiedź: A

Odpowiedź A, wykorzystująca BigQuery Data Transfer Service (DTS) z JDBC i FastExport, jest najbardziej efektywna i spełnia wszystkie wymagania zadania. Kluczowym aspektem jest ograniczenie przestrzeni dyskowej na serwerze Teradata. FastExport umożliwia strumieniowe przesyłanie danych bezpośrednio z Teradata do BigQuery, bez potrzeby tworzenia dużych plików tymczasowych na dysku lokalnym. DTS automatyzuje proces transferu, minimalizując potrzebę pisania skryptów, co również jest zgodne z wymaganiami pytania.

Dyskusja użytkowników potwierdza tę opinię. Wielu użytkowników (raaad, AllenChen123, Matt_108, CGS22) wskazuje na dokumentację Google Cloud, która wyraźnie zaleca użycie JDBC z FastExport w przypadku ograniczeń przestrzeni dyskowej. Użytkownik raaad podkreśla, że FastExport eliminuje potrzebę lokalnej przestrzeni dyskowej.

Odpowiedź C, choć popularna wśród niektórych użytkowników (desertlotus1211, Parandhaman_Margan, ToiToi, kurayish), jest mniej optymalna. Dokumentacja Google Cloud (link podany przez AllenChen123) wskazuje, że JDBC z FastExport jest preferowane, gdy nie można użyć TPT. Użytkownik kurayish zauważa, że JDBC z FastExport wymaga mniej programowania niż rozwiązanie z TPT.

Odpowiedź B i D są mniej efektywne, ponieważ wymagają więcej ręcznej konfiguracji i skryptów, co stoi w sprzeczności z wymaganiem minimalnej ilości programowania. Odpowiedź D dodatkowo wprowadza pośredni krok zapisu do Cloud Storage, co zwiększa opóźnienia i złożoność.

Podsumowując, odpowiedź A jest najbardziej bezpośrednia, efektywna i zgodna z dokumentacją Google Cloud, co czyni ją najlepszym rozwiązaniem dla tego scenariusza.
