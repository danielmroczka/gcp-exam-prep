Your company needs to upload their historic data to Cloud Storage. The security rules don't allow access from external IPs to their on-premises resources. After an initial upload, they will add new data from existing on-premises applications every day. What should they do?

  • A. Execute gsutil rsync from the on-premises servers. Most Voted
  • B. Use Dataflow and write the data to Cloud Storage.
  • C. Write a job template in Dataproc to perform the data transfer.
  • D. Install an FTP server on a Compute Engine VM to receive the files and move them to Cloud Storage.

Odpowiedź: A

Wyjaśnienie: Najpierw, należy zidentyfikować, jak najszybciej i efektywnie przesyłać dane z on-premises zasobów do Cloud Storage, biorąc pod uwagę ograniczenia bezpieczeństwa.  `gsutil rsync` jest idealnym narzędziem do tego, ponieważ jest zaprojektowane do synchronizacji danych między lokalnymi serwerami i Cloud Storage, a jego działanie nie wymaga zewnętrznych połączeń.  Użycie Dataflow, choć potencjalnie przydatne, wymaga więcej konfiguracji i może być mniej wydajne niż bezpośrednie przesyłanie danych przez `gsutil`.  Wybór Job Template w Dataproc jest bardziej złożony i mniej bezpośredni niż `gsutil rsync`, a instalacja FTP serwera na Compute Engine VM nie jest standardową metodą przesyłania danych do Cloud Storage.  Ostatecznie, `gsutil rsync` jest najszybszym i najprostszym sposobem na automatyczne przesyłanie danych z on-premises zasobów do Cloud Storage, spełniając ograniczenia bezpieczeństwa.

Link do dodatkowych żródeł: https://cloud.google.com/storage/docs/transfer

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Prawidłową odpowiedzią jest A. Execute gsutil rsync from the on-premises servers.

Dlaczego?

Wymagania i ograniczenia: 1. Security: No access from external IPs to on-premises resources. To oznacza, że żadna usługa w chmurze (jak Dataflow czy Cloud Storage Transfer Service) nie może "sięgnąć" do Twojego centrum danych (połączenie Inbound jest zablokowane). 2. Daily updates: Potrzebna synchronizacja nowych danych każdego dnia. gsutil rsync uruchomione lokalnie (Outbound connection) rozwiązuje oba problemy. Ponieważ połączenie jest inicjowane z wnętrza centrum danych na zewnątrz (do API Google), przechodzi ono przez firewall (o ile ruch outbound do Google jest dozwolony). rsync jest idealny do codziennych zadań, ponieważ przesyła tylko nowe lub zmienione pliki, oszczędzając czas i pasmo.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • B. Dataflow: Jako usługa zarządzana w chmurze, Dataflow musiałby połączyć się ze źródłem on-premise. Narusza to wymóg braku dostępu z zewnątrz.
  • C. Dataproc job template: Podobnie jak B, Dataproc musiałby mieć dostęp do serwerów lokalnych.
  • D. FTP server on GCE: Wymagałoby to wystawienia serwera FTP na GCE (zewnętrzny IP) i przesyłania tam danych. Choć technicznie możliwe (outbound z on-prem), jest to znacznie bardziej skomplikowane, mniej bezpieczne i mniej wydajne niż natywne gsutil rsync.
