Your organization stores customer data in an on-premises Apache Hadoop cluster in Apache Parquet format. Data is processed on a daily basis by Apache Spark jobs that run on the cluster. You are migrating the Spark jobs and Parquet data to Google Cloud. BigQuery will be used on future transformation pipelines so you need to ensure that your data is available in BigQuery. You want to use managed services, while minimizing ETL data processing changes and overhead costs. What should you do?

  • A. Migrate your data to Cloud Storage and migrate the metadata to Dataproc Metastore (DPMS). Refactor Spark pipelines to write and read data on Cloud Storage, and run them on Dataproc Serverless. Most Voted
  • B. Migrate your data to Cloud Storage and register the bucket as a Dataplex asset. Refactor Spark pipelines to write and read data on Cloud Storage, and run them on Dataproc Serverless.
  • C. Migrate your data to BigQuery. Refactor Spark pipelines to write and read data on BigQuery, and run them on Dataproc Serverless.
  • D. Migrate your data to BigLake. Refactor Spark pipelines to write and read data on Cloud Storage, and run them on Dataproc on Compute Engine.

Odpowiedź: A.

Migracja danych do Cloud Storage i metadanych do Dataproc Metastore (DPMS) jest najbardziej efektywnym rozwiązaniem, ponieważ minimalizuje zmiany w istniejących potokach Spark. Cloud Storage zapewnia skalowalną i ekonomiczną przestrzeń do przechowywania danych w formacie Parquet. DPMS pozwala na zachowanie metadanych, co ułatwia odnajdywanie i zarządzanie danymi. Użycie Dataproc Serverless pozwala na uruchomienie potoków Spark bez konieczności zarządzania infrastrukturą, a refaktoryzacja do odczytu i zapisu z Cloud Storage jest minimalna w porównaniu do zmiany formatu lub docelowego systemu. To podejście łączy korzyści z zarządzanych usług z minimalnym wpływem na istniejący kod.

A. Jest to poprawna odpowiedź, ponieważ minimalizuje zmiany w kodzie Spark i wykorzystuje zarządzane usługi.
B. Dataplex jest bardziej odpowiedni do zarządzania danymi w wielu lokalizacjach, a nie tylko w Cloud Storage, i nie rozwiązuje problemu metadanych tak efektywnie jak DPMS.
C. Bezpośrednia migracja do BigQuery i refaktoryzacja potoków Spark do zapisu i odczytu z BigQuery wymagałaby znaczących zmian w kodzie, co zwiększa koszty i ryzyko.
D. Użycie Dataproc na Compute Engine wymaga zarządzania infrastrukturą, co jest sprzeczne z wymaganiem wykorzystania zarządzanych usług.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Najlepszym rozwiązaniem jest migracja danych do Cloud Storage i metadanych do Dataproc Metastore (DPMS), a następnie uruchomienie zrefaktoryzowanych potoków Spark na Dataproc Serverless. Kluczowe jest minimalizowanie zmian w istniejącym kodzie ETL oraz wykorzystanie zarządzanych usług Google Cloud. Cloud Storage jest efektywnym i kompatybilnym z Spark rozwiązaniem do przechowywania danych w formacie Parquet. DPMS pozwala na zachowanie metadanych, co ułatwia zarządzanie i odnajdywanie danych. Dataproc Serverless eliminuje potrzebę zarządzania infrastrukturą, a refaktoryzacja potoków Spark do odczytu i zapisu z Cloud Storage jest minimalna.

Dyskusja użytkowników potwierdza tę opinię. Wielu użytkowników (e70ea9e, aoifneofi_ef, josech, Anudeep58, JamesKarianis) wskazuje, że opcja A minimalizuje zmiany w kodzie Spark, jest kosztowo efektywna i umożliwia integrację z BigQuery poprzez zewnętrzne tabele. Użytkownik raaad podkreśla kompatybilność Cloud Storage ze Sparkiem i możliwość późniejszego załadowania danych do BigQuery.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Dataplex jest bardziej odpowiedni do zarządzania danymi w wielu lokalizacjach, a nie tylko w Cloud Storage. Dodatkowo, nie rozwiązuje problemu metadanych tak efektywnie jak DPMS.
  • C: Bezpośrednia migracja do BigQuery i refaktoryzacja potoków Spark do zapisu i odczytu z BigQuery wymagałaby znaczących zmian w kodzie, co zwiększa koszty i ryzyko. Chociaż niektórzy użytkownicy (hrishi19, 52ed0e5, Moss2011, JyoGCP) początkowo wskazywali C jako poprawną odpowiedź, argumentowali to głównie potrzebą zapewnienia dostępności danych w BigQuery, pomijając wymóg minimalizacji zmian w ETL.
  • D: Użycie Dataproc na Compute Engine wymaga zarządzania infrastrukturą, co jest sprzeczne z wymaganiem wykorzystania zarządzanych usług. Ponadto, plum21 słusznie zauważa, że Dataproc Serverless jest usługą zarządzaną, a nie Dataproc na Compute Engine. Positron75 podważa również tę interpretację.
