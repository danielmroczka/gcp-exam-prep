You designed an application to use BigQuery streaming inserts for individual postings. Your application also performs data aggregations right after the streaming inserts. You discover that the queries after streaming inserts do not exhibit strong consistency, and reports from the queries might miss in-flight data. How can you adjust your application design to handle this?

  • A. Re-write the application to load accumulated data every 2 minutes using load jobs.
  • B. Convert the streaming insert code to batch load for individual messages.
  • C. Load the original message to Google Cloud SQL, and export the table every hour to BigQuery.
  • D. Estimate the average latency for data availability after streaming inserts, and always run queries after waiting twice as long.
Show Suggested Answer
Suggested Answer: D

Wyjaśnienie

Correct Answer: D

BigQuery streaming inserts provide high-throughput ingestion, but data may take some time (usually seconds to a few minutes) to become available for all types of queries, especially those involving complex aggregations. This is known as "data availability latency." Waiting for a sufficient amount of time (option D) is a common strategy to ensure that the streaming buffer has been processed and is consistent for reporting.

Why other options are incorrect:
A. Loading data every 2 minutes (batch) might solve consistency but increases latency for real-time reporting.
B. Batch loading individual messages is highly inefficient and would hit BigQuery load job quotas very quickly.
C. Exporting from Cloud SQL to BigQuery every hour introduces too much lag for a system that was originally designed for streaming.
