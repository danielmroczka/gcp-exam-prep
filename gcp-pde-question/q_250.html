Your company's data platform ingests CSV file dumps of booking and user profile data from upstream sources into Cloud Storage. The data analyst team wants to join these datasets on the email field available in both the datasets to perform analysis. However, personally identifiable information (PII) should not be accessible to the analysts. You need to de-identify the email field in both the datasets before loading them into BigQuery for analysts. What should you do?

  • A. 1. Create a pipeline to de-identify the email field by using recordTransformations in Cloud Data Loss Prevention (Cloud DLP) with masking as the de-identification transformations type.
    2. Load the booking and user profile data into a BigQuery table.
  • B. 1. Create a pipeline to de-identify the email field by using recordTransformations in Cloud DLP with format-preserving encryption with FFX as the de-identification transformation type.
    2. Load the booking and user profile data into a BigQuery table. Most Voted
  • C. 1. Load the CSV files from Cloud Storage into a BigQuery table, and enable dynamic data masking.
    2. Create a policy tag with the email mask as the data masking rule.
    3. Assign the policy to the email field in both tables. A
    4. Assign the Identity and Access Management bigquerydatapolicy.maskedReader role for the BigQuery tables to the analysts.
  • D. 1. Load the CSV files from Cloud Storage into a BigQuery table, and enable dynamic data masking.
    2. Create a policy tag with the default masking value as the data masking rule.
    3. Assign the policy to the email field in both tables.
    4. Assign the Identity and Access Management bigquerydatapolicy.maskedReader role for the BigQuery tables to the analysts

Odpowiedź: B.

Opcja B jest poprawna, ponieważ format-preserving encryption (FPE) z FFX (Format-Preserving Encryption) jest odpowiednią metodą de-identyfikacji PII, taką jak adresy e-mail. FPE zachowuje format danych (w tym przypadku długość i strukturę adresu e-mail), co może być ważne dla niektórych analiz, jednocześnie uniemożliwiając bezpośrednią identyfikację użytkowników. Użycie `recordTransformations` w Cloud DLP pozwala na transformację danych w pipeline przed załadowaniem ich do BigQuery. Po de-identyfikacji danych, można je bezpiecznie załadować do BigQuery dla analityków.

A: Maskowanie w Cloud DLP ukrywa dane, ale nie zapewnia tak silnej ochrony jak szyfrowanie, a format danych zostaje zmieniony.
C: Dynamiczne maskowanie w BigQuery wymaga uprawnień i może nie być wystarczające do ochrony PII, szczególnie jeśli analitycy mają dostęp do metadanych.
D: Użycie domyślnej wartości maskującej w dynamicznym maskowaniu jest mniej bezpieczne niż FPE, ponieważ nie zapewnia żadnej ochrony formatu danych i może być łatwo odwrócone.

Dodatkowe źródła:

*   **Cloud DLP Documentation:** https://cloud.google.com/dlp
*   **Format-Preserving Encryption (FPE):** https://cloud.google.com/dlp/docs/format-preserving-encryption
*   **BigQuery Dynamic Data Masking:** https://cloud.google.com/bigquery/docs/dynamic-data-masking

Sugerowana odpowiedź: B

Odpowiedź B jest najtrafniejsza, ponieważ spełnia wszystkie wymagania zadania. Kluczowe jest de-identyfikowanie danych przed załadowaniem ich do BigQuery, a jednocześnie umożliwienie analitykom wykonywania operacji JOIN na tych danych. Format-preserving encryption (FPE) z FFX jest idealnym rozwiązaniem, ponieważ zachowuje format danych (długość i strukturę adresu e-mail), co pozwala na poprawne łączenie danych, jednocześnie chroniąc PII.

Użycie recordTransformations w Cloud DLP pozwala na zautomatyzowanie tego procesu w pipeline przetwarzania danych, co jest zgodne z najlepszymi praktykami w architekturze danych.

Dyskusja użytkowników w większości potwierdza tę odpowiedź. Wiele osób zwraca uwagę na konieczność zachowania formatu danych dla operacji JOIN (lipa31, chrissamharris, JyoGCP, ML6, task_7). Dodatkowo, argumenty przeciwko innym odpowiedziom są przekonujące:

  • Odpowiedź A: Maskowanie nie zachowuje formatu danych, co uniemożliwia poprawne łączenie.
  • Odpowiedź C i D: Dynamiczne maskowanie w BigQuery odbywa się po załadowaniu danych do BigQuery, a zadanie wymaga de-identyfikacji przed załadowaniem. Ponadto, role bigquerydatapolicy.maskedReader nie gwarantują pełnej ochrony PII, ponieważ dostęp do danych w surowej postaci może być możliwy w pewnych scenariuszach.

Chociaż początkowo pojawiły się głosy za odpowiedzią C (Matt_108, Sofiia98, GCP001, raaad), to przeważająca większość argumentów i głosów wskazuje na odpowiedź B jako prawidłową.
