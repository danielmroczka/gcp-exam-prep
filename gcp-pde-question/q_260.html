You have two projects where you run BigQuery jobs:
• One project runs production jobs that have strict completion time SLAs. These are high priority jobs that must have the required compute resources available when needed. These jobs generally never go below a 300 slot utilization, but occasionally spike up an additional 500 slots.
• The other project is for users to run ad-hoc analytical queries. This project generally never uses more than 200 slots at a time. You want these ad-hoc queries to be billed based on how much data users scan rather than by slot capacity.

You need to ensure that both projects have the appropriate compute resources available. What should you do?

  • A. Create a single Enterprise Edition reservation for both projects. Set a baseline of 300 slots. Enable autoscaling up to 700 slots.
  • B. Create two reservations, one for each of the projects. For the SLA project, use an Enterprise Edition with a baseline of 300 slots and enable autoscaling up to 500 slots. For the ad-hoc project, configure on-demand billing. Most Voted
  • C. Create two Enterprise Edition reservations, one for each of the projects. For the SLA project, set a baseline of 300 slots and enable autoscaling up to 500 slots. For the ad-hoc project, set a reservation baseline of 0 slots and set the ignore idle slots flag to False.
  • D. Create two Enterprise Edition reservations, one for each of the projects. For the SLA project, set a baseline of 800 slots. For the ad-hoc project, enable autoscaling up to 200 slots.

Odpowiedź: B.

Opcja B jest poprawna, ponieważ pozwala na optymalne zarządzanie kosztami i wydajnością dla obu projektów. Dla projektu z SLA, rezerwacja Enterprise Edition z baseline 300 slotów i autoscalingiem do 500 slotów zapewnia stałą dostępność zasobów potrzebnych do spełnienia umów SLA, a jednocześnie pozwala na elastyczne zwiększenie mocy obliczeniowej w razie potrzeby. Dla projektu ad-hoc, konfiguracja on-demand billingu pozwala na płacenie tylko za faktycznie przeskanowane dane, co jest bardziej opłacalne, biorąc pod uwagę niskie wykorzystanie zasobów. To rozwiązanie zapewnia, że projekt z SLA ma gwarantowane zasoby, a projekt ad-hoc płaci tylko za to, czego używa.

A. Tworzenie jednej rezerwacji dla obu projektów z baseline 300 i autoscalingiem do 700 slotów nie uwzględnia potrzeby rozliczeń na podstawie przeskanowanych danych dla projektu ad-hoc.
C. Ustawienie baseline 0 slotów i wyłączenie ignorowania nieużywanych slotów w projekcie ad-hoc nie gwarantuje rozliczeń na podstawie przeskanowanych danych i może prowadzić do niepotrzebnych kosztów.
D. Ustawienie baseline 800 slotów dla projektu SLA jest zbyt wysokie i nieefektywne kosztowo, a autoscaling do 200 slotów dla projektu ad-hoc jest niewystarczający.

Sugerowana odpowiedź: B

Odpowiedź B jest prawidłowa, ponieważ optymalnie spełnia wymagania zadania dotyczące obu projektów. Projekt z umowami SLA (Service Level Agreement) wymaga gwarantowanej dostępności zasobów obliczeniowych, aby zapewnić terminowe wykonywanie zadań. Rezerwacja Enterprise Edition z baseline 300 slotów i autoscalingiem do 500 slotów zapewnia stałą dostępność 300 slotów, a w przypadku nagłych wzrostów obciążenia, system automatycznie zwiększy liczbę slotów do 800. Dzięki temu zadania SLA będą wykonywane w ramach określonego czasu.

Drugi projekt, przeznaczony do ad-hoc analiz, charakteryzuje się zmiennym obciążeniem i niskim wykorzystaniem zasobów (do 200 slotów). W tym przypadku, konfiguracja rozliczeń on-demand (płać za użycie) jest najbardziej opłacalna, ponieważ użytkownicy płacą tylko za faktycznie przeskanowane dane, a nie za zarezerwowane sloty. To rozwiązanie minimalizuje koszty, gdy obciążenie jest niskie.

Dyskusja użytkowników potwierdza poprawność odpowiedzi B. Wielu użytkowników, w tym osoby pracujące z BigQuery na co dzień (np. ToiToi), wskazuje, że rozliczenia on-demand są kluczowe dla projektów ad-hoc. Użytkownik SamuelTsch podaje link do dokumentacji Google Cloud, która potwierdza, że baseline i autoscaling to dwa różne elementy, co dodatkowo wzmacnia argument za odpowiedzią B.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Tworzenie jednej rezerwacji dla obu projektów nie uwzględnia potrzeby rozliczeń na podstawie przeskanowanych danych dla projektu ad-hoc.
  • C: Ustawienie baseline 0 slotów i wyłączenie ignorowania nieużywanych slotów w projekcie ad-hoc nie gwarantuje rozliczeń na podstawie przeskanowanych danych i może prowadzić do niepotrzebnych kosztów.
  • D: Ustawienie baseline 800 slotów dla projektu SLA jest zbyt wysokie i nieefektywne kosztowo, a autoscaling do 200 slotów dla projektu ad-hoc jest niewystarczający. Dodatkowo, jak wskazują użytkownicy (np. ToiToi), rezerwowanie 800 slotów dla sporadycznych pików jest nieopłacalne.
