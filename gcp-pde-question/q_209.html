A shipping company has live package-tracking data that is sent to an Apache Kafka stream in real time. This is then loaded into BigQuery. Analysts in your company want to query the tracking data in BigQuery to analyze geospatial trends in the lifecycle of a package. The table was originally created with ingest-date partitioning. Over time, the query processing time has increased. You need to copy all the data to a new clustered table. What should you do?

  • A. Re-create the table using data partitioning on the package delivery date.
  • B. Implement clustering in BigQuery on the package-tracking ID column. Most Voted
  • C. Implement clustering in BigQuery on the ingest date column.
  • D. Tier older data onto Cloud Storage files and create a BigQuery table using Cloud Storage as an external data source.

Odpowiedź: B.

Implementacja clusteringu w BigQuery na kolumnie identyfikatora śledzenia paczki (package-tracking ID) jest najlepszym rozwiązaniem, ponieważ clustering optymalizuje wydajność zapytań poprzez fizyczne grupowanie danych o podobnych wartościach w tej kolumnie. W przypadku analizy trendów geoprzestrzennych w cyklu życia paczki, identyfikator śledzenia paczki jest kluczowy, ponieważ pozwala na szybkie filtrowanie i agregację danych dla konkretnych paczek.  Ponieważ oryginalna tabela była partycjonowana po dacie załadowania (ingest-date), a wydajność zapytań spadła, clustering na kolumnie istotnej dla analizy (tracking ID) przyniesie większe korzyści niż zmiana partycjonowania. Clustering pozwala na efektywne skanowanie tylko tych bloków danych, które są istotne dla zapytania, co znacznie przyspiesza przetwarzanie.

A. Zmiana partycjonowania na datę dostawy paczki może nie przynieść oczekiwanej poprawy wydajności, jeśli zapytania często dotyczą pojedynczych paczek.
C. Clustering na dacie załadowania (ingest-date) nie jest odpowiedni, ponieważ nie odzwierciedla danych istotnych dla analizy trendów geoprzestrzennych.
D. Przeniesienie starszych danych do Cloud Storage i użycie ich jako zewnętrznego źródła danych pogorszy wydajność zapytań, ponieważ BigQuery będzie musiał odczytywać dane z wolniejszego medium.

Sugerowana odpowiedź: B

Implementacja clusteringu w BigQuery na kolumnie identyfikatora śledzenia paczki (package-tracking ID) jest najefektywniejszym rozwiązaniem. Analiza dotyczy trendów geoprzestrzennych w cyklu życia pojedynczych paczek. Clustering na identyfikatorze śledzenia paczki fizycznie grupuje wszystkie dane związane z daną paczką, co znacząco przyspiesza zapytania, które filtrują lub agregują dane po tym identyfikatorze. To pozwala na efektywne skanowanie tylko tych bloków danych, które są istotne dla zapytania, zamiast skanowania całej tabeli.

Komentarze użytkowników w dużej mierze potwierdzają tę odpowiedź. Większość użytkowników (w tym ci z nowszymi datami i większą liczbą głosów) wybrała odpowiedź B i uzasadniła ją w podobny sposób. Użytkownik e70ea9e szczególnie dobrze wyjaśnił, dlaczego clustering jest lepszy niż partycjonowanie w tym przypadku, podkreślając, że clustering optymalizuje organizację danych pod kątem konkretnych wzorców zapytań.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Zmiana partycjonowania na datę dostawy paczki może nie przynieść oczekiwanej poprawy, jeśli zapytania często dotyczą pojedynczych paczek lub analizy w określonym przedziale czasu dla wielu paczek.
  • C: Clustering na dacie załadowania (ingest-date) nie jest odpowiedni, ponieważ data załadowania nie jest bezpośrednio związana z danymi istotnymi dla analizy trendów geoprzestrzennych.
  • D: Przeniesienie starszych danych do Cloud Storage i użycie ich jako zewnętrznego źródła danych znacznie pogorszy wydajność zapytań, ponieważ BigQuery będzie musiał odczytywać dane z wolniejszego medium.

Dodatkowo, komentarz użytkownika apoio.certificacoes.closer zwraca uwagę na ważną kwestię: clustering nie wymaga migracji danych, jak to ma miejsce w przypadku partycjonowania. Można go zaimplementować na istniejącej tabeli, choć dane muszą być ponownie zapisane, aby skorzystać z clusteringu.
