You are migrating a large number of files from a public HTTPS endpoint to Cloud Storage. The files are protected from unauthorized access using signed URLs. You created a TSV file that contains the list of object URLs and started a transfer job by using Storage Transfer Service. You notice that the job has run for a long time and eventually failed. Checking the logs of the transfer job reveals that the job was running fine until one point, and then it failed due to HTTP 403 errors on the remaining files. You verified that there were no changes to the source system. You need to fix the problem to resume the migration process. What should you do?

  • A. Set up Cloud Storage FUSE, and mount the Cloud Storage bucket on a Compute Engine instance. Remove the completed files from the TSV file. Use a shell script to iterate through the TSV file and download the remaining URLs to the FUSE mount point.
  • B. Renew the TLS certificate of the HTTPS endpoint. Remove the completed files from the TSV file and rerun the Storage Transfer Service job.
  • C. Create a new TSV file for the remaining files by generating signed URLs with a longer validity period. Split the TSV file into multiple smaller files and submit them as separate Storage Transfer Service jobs in parallel. Most Voted
  • D. Update the file checksums in the TSV file from using MD5 to SHA256. Remove the completed files from the TSV file and rerun the Storage Transfer Service job.

Odpowiedź: C.

Problem polega na tym, że Storage Transfer Service napotkał błędy 403 (Forbidden) po pewnym czasie działania, co sugeruje, że wygenerowane wcześniej signed URL-e wygasły. Signed URL-e mają ograniczony czas ważności, a ponieważ migracja trwała długo, niektóre z nich stały się nieważne zanim transfer został ukończony. Opcja C rozwiązuje ten problem poprzez wygenerowanie nowych signed URL-i z dłuższym okresem ważności, co pozwoli na pomyślne dokończenie transferu. Dodatkowo, podzielenie pliku TSV na mniejsze części i uruchomienie zadań równolegle może przyspieszyć proces i zmniejszyć ryzyko ponownego wygaśnięcia URL-i.

A. Użycie Cloud Storage FUSE i skryptu shell jest nieefektywne i bardziej skomplikowane niż wykorzystanie Storage Transfer Service do tego zadania.
B. Odnowienie certyfikatu TLS nie rozwiąże problemu wygasłych signed URL-i, ponieważ problem leży po stronie autoryzacji, a nie połączenia.
D. Zmiana algorytmu sum kontrolnych nie ma wpływu na problem z autoryzacją i wygasłymi URL-ami.

Sugerowana odpowiedź: C

Odpowiedź C jest prawidłowa i najlepiej rozwiązuje problem. Błędy 403 (Forbidden) po pewnym czasie działania transferu wskazują na wygaśnięcie signed URL-i. Signed URL-e mają ograniczony czas ważności, a długotrwały proces transferu mógł spowodować, że niektóre z nich stały się nieważne zanim transfer został ukończony. Generowanie nowych signed URL-i z dłuższym okresem ważności rozwiązuje ten problem.

Dodatkowo, podzielenie pliku TSV na mniejsze części i uruchomienie zadań równolegle (również zawarte w odpowiedzi C) może przyspieszyć proces transferu i zmniejszyć ryzyko ponownego wygaśnięcia URL-i, a także poprawić odporność na potencjalne problemy z transferem dużych plików.

Komentarze użytkowników, w szczególności te nowsze (np. iooj, który potwierdza pojawienie się tego pytania na egzaminie w sierpniu 2024) oraz te z większą liczbą głosów (np. raaad, srivastavas08, Pime13, Matt_108), jednoznacznie wskazują na odpowiedź C jako prawidłową i uzasadniają ją wygaśnięciem signed URL-i.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Użycie Cloud Storage FUSE i skryptu shell jest nieefektywne i bardziej skomplikowane niż wykorzystanie Storage Transfer Service. Wymaga ręcznej interwencji i nie rozwiązuje problemu wygasłych URL-i.
  • B: Odnowienie certyfikatu TLS nie rozwiąże problemu wygasłych signed URL-i. Problem leży po stronie autoryzacji (ważności URL-i), a nie połączenia.
  • D: Zmiana algorytmu sum kontrolnych (MD5 na SHA256) nie ma wpływu na problem z autoryzacją i wygasłymi URL-ami.
