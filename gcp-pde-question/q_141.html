Data Analysts in your company have the Cloud IAM Owner role assigned to them in their projects to allow them to work with multiple GCP products in their projects. Your organization requires that all BigQuery data access logs be retained for 6 months. You need to ensure that only audit personnel in your company can access the data access logs for all projects. What should you do?

  • A. Enable data access logs in each Data Analyst's project. Restrict access to Stackdriver Logging via Cloud IAM roles.
  • B. Export the data access logs via a project-level export sink to a Cloud Storage bucket in the Data Analysts' projects. Restrict access to the Cloud Storage bucket.
  • C. Export the data access logs via a project-level export sink to a Cloud Storage bucket in a newly created projects for audit logs. Restrict access to the project with the exported logs.
  • D. Export the data access logs via an aggregated export sink to a Cloud Storage bucket in a newly created project for audit logs. Restrict access to the project that contains the exported logs. Most Voted

Odpowiedź: D.

Eksportowanie logów BigQuery za pomocą zagregowanego sinka do dedykowanego projektu dla logów audytowych jest najlepszym rozwiązaniem, ponieważ pozwala na scentralizowane przechowywanie i kontrolę dostępu do logów z wielu projektów. Agregowany sink umożliwia zbieranie logów z całej organizacji w jednym miejscu, co ułatwia zarządzanie i analizę. Następnie, poprzez ograniczenie dostępu do projektu zawierającego logi audytowe za pomocą Cloud IAM, zapewniamy, że tylko uprawniony personel audytowy będzie miał do nich dostęp. To rozwiązanie spełnia wymagania dotyczące retencji logów (6 miesięcy) i ograniczenia dostępu.

A. Włączenie logów w każdym projekcie Data Analysta utrudnia scentralizowane zarządzanie i kontrolę dostępu.
B. Przechowywanie logów w projektach Data Analystów narusza zasadę najmniejszych uprawnień i nie zapewnia odpowiedniej ochrony.
C. Eksport do osobnych projektów dla logów audytowych jest mniej efektywny niż agregacja, ponieważ wymaga zarządzania wieloma projektami.

Sugerowana odpowiedź: D

Odpowiedź D jest prawidłowa, ponieważ wykorzystuje zagregowane sinki logów (aggregated export sinks) do scentralizowanego zbierania logów dostępu do BigQuery z wielu projektów w dedykowanym projekcie dla logów audytowych. Pozwala to na efektywne zarządzanie logami i ograniczenie dostępu wyłącznie do uprawnionego personelu audytowego poprzez kontrolę dostępu na poziomie projektu. Zagregowane sinki są kluczowe, ponieważ pytanie dotyczy dostępu do logów z wielu projektów, a standardowe sinki działają na poziomie pojedynczego projektu.

Komentarze użytkowników, szczególnie te nowsze i z większą liczbą głosów (np. barnac1es, zellck, Chelseajcole, MaxNRG), konsekwentnie potwierdzają poprawność odpowiedzi D, odwołując się do dokumentacji Google Cloud i podkreślając zalety zagregowanych sinków w kontekście scentralizowanego logowania i kontroli dostępu.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Włączenie logów w każdym projekcie Data Analysta nie zapewnia scentralizowanego zarządzania i utrudnia kontrolę dostępu.
  • B: Przechowywanie logów w projektach Data Analystów narusza zasadę najmniejszych uprawnień i umożliwia dostęp do logów osobom nieuprawnionym.
  • C: Chociaż eksport do osobnych projektów dla logów audytowych jest lepszy niż przechowywanie ich w projektach Data Analystów, nie jest tak efektywny jak agregacja. Zarządzanie wieloma projektami logów audytowych jest bardziej skomplikowane niż zarządzanie jednym zagregowanym sinkiem. Dodatkowo, jak zauważono w dyskusji (septiandy, FP77), różnica między C i D sprowadza się do użycia zagregowanego sinka, który jest kluczowy dla efektywnego zarządzania logami z wielu projektów.
