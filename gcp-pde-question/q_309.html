You work for an airline and you need to store weather data in a BigQuery table. Weather data will be used as input to a machine learning model. The model only uses the last 30 days of weather data. You want to avoid storing unnecessary data and minimize costs. What should you do?

  • A. Create a BigQuery table where each record has an ingestion timestamp. Run a scheduled query to delete all the rows with an ingestion timestamp older than 30 days.
  • B. Create a BigQuery table partitioned by datetime value of the weather date. Set up partition expiration to 30 days. Most Voted
  • C. Create a BigQuery table partitioned by ingestion time. Set up partition expiration to 30 days.
  • D. Create a BigQuery table with a datetime column for the day the weather data refers to. Run a scheduled query to delete rows with a datetime value older than 30 days.

Odpowiedź: B.

Partycjonowanie tabel w BigQuery według daty (w tym przypadku daty pogodowej) jest najbardziej efektywnym sposobem zarządzania danymi, które mają ograniczony okres ważności. Ustawienie wygasania partycji na 30 dni automatycznie usuwa starsze dane, eliminując potrzebę ręcznego usuwania rekordów za pomocą zapytań. To rozwiązanie minimalizuje koszty przechowywania, ponieważ BigQuery przechowuje metadane tylko dla istniejących partycji, a usunięcie partycji jest operacją metadanych, a nie kopiowania danych. Dzięki temu proces jest szybki i tani. Partycjonowanie pozwala również na szybsze wykonywanie zapytań, ponieważ BigQuery skanuje tylko te partycje, które zawierają potrzebne dane.

A. Używanie sygnatury czasowej wstawiania do filtrowania danych jest mniej efektywne, ponieważ BigQuery musi skanować całą tabelę, aby znaleźć rekordy do usunięcia.
C. Partycjonowanie według czasu wstawiania nie jest idealne, ponieważ dane pogodowe odnoszą się do konkretnej daty, a nie do momentu ich wstawienia do BigQuery.
D. Podobnie jak w opcji A, usuwanie wierszy na podstawie daty pogodowej za pomocą zaplanowanego zapytania wymaga skanowania całej tabeli, co jest kosztowne i wolne.

Sugerowana odpowiedź: B

Najlepszym rozwiązaniem jest partycjonowanie tabeli BigQuery według daty pogodowej i ustawienie wygaśnięcia partycji na 30 dni. To podejście jest najbardziej efektywne kosztowo i wydajnościowo, ponieważ BigQuery zarządza partycjami jako jednostkami metadanych. Usunięcie partycji jest operacją metadanych, a nie kopiowania danych, co jest szybkie i tanie.

Komentarze użytkowników, szczególnie te nowsze (np. sierpień 2024, iooj) potwierdzają, że pytanie pojawiło się na egzaminie i poprawna odpowiedź to B. Użytkownik joao_01 słusznie zauważa, że granularność partycji to dni, a nie sekundy, co obala argument chambg o przekroczeniu limitu 4000 partycji.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A i D: Usuwanie wierszy za pomocą zaplanowanych zapytań wymaga skanowania całej tabeli, co jest kosztowne i czasochłonne.
  • C: Partycjonowanie według czasu wstawiania (ingestion time) nie jest optymalne, ponieważ interesuje nas data pogodowa, a nie moment wstawienia danych do BigQuery. Dane mogą być wstawiane z opóźnieniem, a my potrzebujemy danych za ostatnie 30 dni kalendarzowych.

Dodatkowe źródło (link podany przez d11379b) potwierdza możliwość partycjonowania po datach z granularnością dzienną.
