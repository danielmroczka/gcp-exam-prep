Your software uses a simple JSON format for all messages. These messages are published to Google Cloud Pub/Sub, then processed with Google Cloud
Dataflow to create a real-time dashboard for the CFO. During testing, you notice that some messages are missing in the dashboard. You check the logs, and all messages are being published to Cloud Pub/Sub successfully. What should you do next?

  • A. Check the dashboard application to see if it is not displaying correctly.
  • B. Run a fixed dataset through the Cloud Dataflow pipeline and analyze the output. Most Voted
  • C. Use Google Stackdriver Monitoring on Cloud Pub/Sub to find the missing messages.
  • D. Switch Cloud Dataflow to pull messages from Cloud Pub/Sub instead of Cloud Pub/Sub pushing messages to Cloud Dataflow.

Odpowiedź: B.

Obecne problemy z brakującymi danymi sugerują, że dane docierają do Pub/Sub, ale giną lub są błędnie przetwarzane w Dataflow, zanim trafią do pulpitu nawigacyjnego. Uruchomienie ustalonego zestawu danych przez potok Dataflow i analiza jego wyjścia pozwoli na zlokalizowanie miejsca występowania problemu w całym procesie przetwarzania danych. Pozwoli to zidentyfikować, czy problem leży w samym potoku Dataflow, jego konfiguracji, czy też w sposobie przetwarzania danych.

Opcja A jest mniej prawdopodobna, ponieważ problem wydaje się dotyczyć brakujących danych, a nie błędów w wyświetlaniu istniejących.
Opcja C koncentruje się na źródle (Pub/Sub), a logi wskazują, że wiadomości tam docierają, więc problem prawdopodobnie leży dalej w potoku.
Opcja D sugeruje zmianę mechanizmu dostarczania, co nie rozwiązuje problemu braku danych, jeśli występuje on podczas przetwarzania w Dataflow.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Run a fixed dataset through the Cloud Dataflow pipeline and analyze the output.

Dlaczego?

Sytuacja: Dane wchodzą do Pub/Sub (potwierdzone logami), ale nie pojawiają się w dashboardzie (wynik Dataflow). Oznacza to, że problem leży w logice przetwarzania wewnątrz Dataflow (np. złe okna czasowe, odrzucanie "late data", błędy parsowania JSON). Aby zdebugować logikę strumieniową, najlepszą praktyką jest "izolacja problemu". Użycie statycznego, znanego zbioru danych (fixed dataset) pozwala na powtarzalne testy i sprawdzenie, czy pipeline poprawnie przetwarza te konkretne rekordy, bez szumu związanego z siecią czy opóźnieniami strumienia.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Check dashboard: Dashboard pokazuje wynik. Skoro brakuje danych, sprawdzenie dashboardu nie powie nam DLACZEGO ich brakuje w logice pipeline'u.
  • C. Stackdriver Monitoring on Pub/Sub: Wiemy już z treści pytania, że "all messages are being published to Cloud Pub/Sub successfully". Monitoring Pub/Sub potwierdzi tylko to, co już wiemy.
  • D. Switch to pull messages: Dataflow używa mechanizmu pull/streaming wewnętrznie. Ręczna zmiana modelu subskrypcji nie naprawi błędów w logice biznesowej pipeline'u (np. błędnego filtra).
