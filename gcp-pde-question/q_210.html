You are designing a data mesh on Google Cloud with multiple distinct data engineering teams building data products. The typical data curation design pattern consists of landing files in Cloud Storage, transforming raw data in Cloud Storage and BigQuery datasets, and storing the final curated data product in BigQuery datasets. You need to configure Dataplex to ensure that each team can access only the assets needed to build their data products. You also need to ensure that teams can easily share the curated data product. What should you do?

  • A. 1. Create a single Dataplex virtual lake and create a single zone to contain landing, raw, and curated data.
    2. Provide each data engineering team access to the virtual lake.
  • B. 1. Create a single Dataplex virtual lake and create a single zone to contain landing, raw, and curated data.
    2. Build separate assets for each data product within the zone.
    3. Assign permissions to the data engineering teams at the zone level.
  • C. 1. Create a Dataplex virtual lake for each data product, and create a single zone to contain landing, raw, and curated data.
    2. Provide the data engineering teams with full access to the virtual lake assigned to their data product.
  • D. 1. Create a Dataplex virtual lake for each data product, and create multiple zones for landing, raw, and curated data.
    2. Provide the data engineering teams with full access to the virtual lake assigned to their data product. Most Voted

Odpowiedź: D.

Opcja D jest poprawna, ponieważ w architekturze data mesh kluczowe jest zapewnienie autonomii zespołom data engineering. Tworzenie oddzielnych Dataplex virtual lakes dla każdego produktu danych oraz stref (zones) dla poszczególnych etapów przetwarzania (landing, raw, curated) pozwala na izolację danych i kontrolę dostępu. Zespoły otrzymują pełny dostęp do swojego virtual lake, co umożliwia im niezależne budowanie i zarządzanie swoimi produktami danych. Jednocześnie, Dataplex umożliwia łatwe udostępnianie gotowych, wyselekcjonowanych produktów danych innym zespołom, zachowując kontrolę nad tym, co jest udostępniane.

A. Stworzenie pojedynczego virtual lake i strefy nie zapewnia izolacji danych i autonomii zespołów, a dostęp do całego lake'a dla każdego zespołu jest zbyt szeroki.
B. Budowanie oddzielnych zasobów w jednej strefie nie rozwiązuje problemu kontroli dostępu i izolacji danych tak efektywnie jak oddzielne virtual lakes.
C. Stworzenie wielu virtual lakes z pojedynczą strefą nie wykorzystuje w pełni możliwości Dataplex do zarządzania etapami przetwarzania danych i kontroli dostępu.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Odpowiedź D jest prawidłowa i najlepiej odzwierciedla zasady architektury data mesh oraz możliwości Dataplex. Kluczowym elementem data mesh jest decentralizacja i autonomia zespołów. Tworzenie oddzielnych virtual lakes dla każdego produktu danych oraz stref (zones) dla poszczególnych etapów przetwarzania (landing, raw, curated) zapewnia izolację danych i kontrolę dostępu. Każdy zespół ma pełną kontrolę nad swoim virtual lake, co umożliwia niezależne budowanie i zarządzanie swoimi produktami danych.

Dodatkowo, Dataplex umożliwia łatwe udostępnianie gotowych, wyselekcjonowanych produktów danych innym zespołom poprzez mechanizmy udostępniania między jeziorami (cross-lake sharing), zachowując kontrolę nad tym, co jest udostępniane. Komentarze użytkowników, szczególnie te nowsze i z większą liczbą głosów (np. MaxNRG, datapassionate, Helinia), potwierdzają tę interpretację, wskazując na konieczność oddzielnych jezior dla każdego produktu danych i stref dla różnych etapów przetwarzania.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Stworzenie pojedynczego virtual lake i strefy nie zapewnia izolacji danych i autonomii zespołów. Dostęp do całego jeziora dla każdego zespołu jest zbyt szeroki i narusza zasadę minimalnych uprawnień.
  • B: Budowanie oddzielnych zasobów w jednej strefie nie rozwiązuje problemu kontroli dostępu i izolacji danych tak efektywnie jak oddzielne virtual lakes. Utrudnia to również zarządzanie etapami przetwarzania danych.
  • C: Stworzenie wielu virtual lakes z pojedynczą strefą nie wykorzystuje w pełni możliwości Dataplex do zarządzania etapami przetwarzania danych i kontroli dostępu. Brak podziału na strefy uniemożliwia efektywne zarządzanie cyklem życia danych.

Użytkownik MaxNRG w swoim komentarzu trafnie podsumowuje, że należy skonfigurować udostępnianie między jeziorami (cross-lake sharing) dla wyselekcjonowanych danych, co dodatkowo potwierdza poprawność odpowiedzi D.
