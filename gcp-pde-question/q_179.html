You are building a real-time prediction engine that streams files, which may contain PII (personal identifiable information) data, into Cloud Storage and eventually into BigQuery. You want to ensure that the sensitive data is masked but still maintains referential integrity, because names and emails are often used as join keys.
How should you use the Cloud Data Loss Prevention API (DLP API) to ensure that the PII data is not accessible by unauthorized individuals?

  • A. Create a pseudonym by replacing the PII data with cryptogenic tokens, and store the non-tokenized data in a locked-down button.
  • B. Redact all PII data, and store a version of the unredacted data in a locked-down bucket.
  • C. Scan every table in BigQuery, and mask the data it finds that has PII.
  • D. Create a pseudonym by replacing PII data with a cryptographic format-preserving token. Most Voted

Odpowiedź: D.

Opcja D jest poprawna, ponieważ format-preserving tokenization (FPT) pozwala na zastąpienie danych PII tokenami, które zachowują oryginalny format danych (np. długość i typ). Dzięki temu, dane mogą być nadal używane jako klucze do łączenia tabel w BigQuery, zachowując referencyjną integralność, jednocześnie maskując wrażliwe informacje. FPT wykorzystuje kryptografię do generowania tokenów, co zapewnia bezpieczeństwo danych. To rozwiązanie minimalizuje wpływ na istniejące procesy analityczne, które polegają na tych kluczach.  DLP API oferuje funkcjonalność FPT, umożliwiając automatyzację tego procesu w potoku danych.

A. Pseudonimizacja z użyciem cryptogenic tokens i przechowywanie oryginalnych danych w bezpiecznym bucketcie jest mniej praktyczna, ponieważ wymaga zarządzania dodatkowym kluczem i potencjalnie zwiększa złożoność systemu.
B. Redakcja PII uniemożliwia wykorzystanie tych danych jako kluczy do łączenia, co narusza wymóg zachowania referencyjnej integralności.
C. Skanowanie istniejących tabel w BigQuery jest reaktywne, a nie proaktywne i nie chroni danych w momencie ich wprowadzania do systemu, a także może być kosztowne i czasochłonne.

Sugerowana odpowiedź: D

Odpowiedź D jest prawidłowa, ponieważ spełnia wszystkie wymagania zadania. Kluczowe jest zachowanie referencyjnej integralności danych PII (imię i nazwisko, adres e-mail) używanych jako klucze łączące w BigQuery. Format-preserving tokenization (FPT) pozwala na zastąpienie danych PII tokenami, które zachowują oryginalny format (długość, typ danych), co umożliwia dalsze wykorzystanie tych tokenów w operacjach JOIN bez utraty możliwości łączenia rekordów. DLP API oferuje funkcjonalność FPT, co pozwala na automatyzację tego procesu w potoku danych.

Komentarze użytkowników, szczególnie te nowsze i z większą liczbą głosów, konsekwentnie wskazują na odpowiedź D jako prawidłową, podkreślając znaczenie zachowania referencyjnej integralności. Użytkownik musumusu słusznie zauważa, że kluczowym słowem jest "referential integrity" i format-preserving option to umożliwia.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Przechowywanie oryginalnych, nietokenizowanych danych w zabezpieczonym bucketcie zwiększa złożoność systemu i ryzyko naruszenia bezpieczeństwa. Dodatkowo, nie jest to standardowa praktyka.
  • B: Redakcja danych PII uniemożliwia ich wykorzystanie jako kluczy łączących, co narusza wymóg zachowania referencyjnej integralności.
  • C: Skanowanie istniejących tabel w BigQuery jest działaniem reaktywnym, a nie proaktywnym. Nie chroni danych w momencie ich wprowadzania do systemu i może być kosztowne oraz czasochłonne.

Warto zauważyć, że użytkownik cetanx początkowo wskazywał na odpowiedź B, ale po uwzględnieniu warunku zachowania referencyjnej integralności zmienił zdanie. Użytkownik jkhong podnosi ważny argument dotyczący ograniczeń ASCII w FPT, jednak w kontekście zadania, gdzie wymagane jest zachowanie referencyjnej integralności, odpowiedź D pozostaje najbardziej adekwatna.
