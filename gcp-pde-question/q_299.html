You have an upstream process that writes data to Cloud Storage. This data is then read by an Apache Spark job that runs on Dataproc. These jobs are run in the us-central1 region, but the data could be stored anywhere in the United States. You need to have a recovery process in place in case of a catastrophic single region failure. You need an approach with a maximum of 15 minutes of data loss (RPO=15 mins). You want to ensure that there is minimal latency when reading the data. What should you do?

  • A. 1. Create two regional Cloud Storage buckets, one in the us-central1 region and one in the us-south1 region.
    2. Have the upstream process write data to the us-central1 bucket. Use the Storage Transfer Service to copy data hourly from the us-central1 bucket to the us-south1 bucket.
    3. Run the Dataproc cluster in a zone in the us-central1 region, reading from the bucket in that region.
    4. In case of regional failure, redeploy your Dataproc clusters to the us-south1 region and read from the bucket in that region instead.
  • B. 1. Create a Cloud Storage bucket in the US multi-region.
    2. Run the Dataproc cluster in a zone in the us-central1 region, reading data from the US multi-region bucket.
    3. In case of a regional failure, redeploy the Dataproc cluster to the us-central2 region and continue reading from the same bucket.
  • C. 1. Create a dual-region Cloud Storage bucket in the us-central1 and us-south1 regions.
    2. Enable turbo replication.
    3. Run the Dataproc cluster in a zone in the us-central1 region, reading from the bucket in the us-south1 region.
    4. In case of a regional failure, redeploy your Dataproc cluster to the us-south1 region and continue reading from the same bucket.
  • D. 1. Create a dual-region Cloud Storage bucket in the us-central1 and us-south1 regions.
    2. Enable turbo replication.
    3. Run the Dataproc cluster in a zone in the us-central1 region, reading from the bucket in the same region.
    4. In case of a regional failure, redeploy the Dataproc clusters to the us-south1 region and read from the same bucket. Most Voted

Odpowiedź: D.

Opcja D jest poprawna, ponieważ wykorzystuje dual-region Cloud Storage z turbo replication, co zapewnia wysoką dostępność i odporność na awarie. Dual-region replikuje dane do dwóch regionów, a turbo replication minimalizuje opóźnienia w dostępie do danych w regionie awaryjnym. Uruchomienie Dataproc w us-central1 i odzyskiwanie w us-south1, czytając z tego samego bucketu, spełnia wymaganie RPO 15 minut (turbo replication zapewnia szybką synchronizację) i minimalizuje opóźnienia, ponieważ Dataproc będzie czytać z najbliższego regionu. W przypadku awarii, redeploy Dataproc do us-south1 i kontynuacja odczytu z tego samego bucketu zapewnia ciągłość działania.

A. Użycie Storage Transfer Service z godzinowym opóźnieniem nie spełnia RPO 15 minut.
B. Multi-region bucket nie gwarantuje tak niskich opóźnień jak dual-region z turbo replication.
C. Czytanie z us-south1 podczas normalnej pracy zwiększa opóźnienia, a turbo replication nie jest w pełni wykorzystane.

Sugerowana odpowiedź: D

Odpowiedź D jest prawidłowa, ponieważ spełnia wszystkie wymagania zadania dotyczące odzyskiwania po awarii (RPO 15 minut) i minimalizacji opóźnień. Kluczowe elementy to:

  • Dual-region Cloud Storage: Replikacja danych do dwóch regionów (us-central1 i us-south1) zapewnia odporność na awarie pojedynczego regionu.
  • Turbo Replication: Turbo replication zapewnia niemal natychmiastową synchronizację danych między regionami, co pozwala na spełnienie wymagania RPO wynoszącego 15 minut.
  • Lokalny dostęp do danych: Uruchomienie klastra Dataproc w us-central1 i odczytywanie danych z tego samego regionu minimalizuje opóźnienia podczas normalnej pracy.
  • Płynne przełączenie po awarii: W przypadku awarii regionu us-central1, klastry Dataproc mogą zostać ponownie wdrożone w us-south1 i kontynuować odczytywanie danych z tego samego bucketu, bez przestojów.

Dyskusja użytkowników w pełni potwierdza poprawność odpowiedzi D. Użytkownicy raaad, JyoGCP, Matt_108 i scaenruy jednogłośnie wskazują na tę opcję, podkreślając korzyści z turbo replication, minimalnych opóźnień i płynnego przełączania po awarii.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Użycie Storage Transfer Service z godzinowym opóźnieniem nie spełnia wymogu RPO wynoszącego 15 minut.
  • B: Multi-region bucket, choć zapewnia wysoką dostępność, nie gwarantuje tak niskich opóźnień jak dual-region z turbo replication.
  • C: Czytanie z us-south1 podczas normalnej pracy zwiększa opóźnienia, a turbo replication nie jest w pełni wykorzystane, ponieważ dane są odczytywane z regionu, do którego nie są natywnie kierowane.
