The data analyst team at your company uses BigQuery for ad-hoc queries and scheduled SQL pipelines in a Google Cloud project with a slot reservation of 2000 slots. However, with the recent introduction of hundreds of new non time-sensitive SQL pipelines, the team is encountering frequent quota errors. You examine the logs and notice that approximately 1500 queries are being triggered concurrently during peak time. You need to resolve the concurrency issue. What should you do?

  • A. Increase the slot capacity of the project with baseline as 0 and maximum reservation size as 3000.
  • B. Update SQL pipelines to run as a batch query, and run ad-hoc queries as interactive query jobs. Most Voted
  • C. Increase the slot capacity of the project with baseline as 2000 and maximum reservation size as 3000.
  • D. Update SQL pipelines and ad-hoc queries to run as interactive query jobs.

Odpowiedź: B.

Wyjaśnienie: Problem polega na zbyt dużej liczbie współbieżnych zapytań (1500) w stosunku do dostępnej puli slotów (2000). Zmiana sposobu uruchamiania zapytań na zadania wsadowe (batch) dla SQL pipelines i interaktywne dla zapytań ad-hoc jest najlepszym rozwiązaniem. Zadania wsadowe są kolejkowane i uruchamiane w miarę dostępności slotów, co zapobiega przekroczeniu limitu. Interaktywne zapytania, choć uruchamiane natychmiast, są zazwyczaj krótsze i mniej zasobożerne, co pozwala na efektywniejsze wykorzystanie slotów. To podejście pozwala na lepsze zarządzanie zasobami i uniknięcie błędów związanych z przekroczeniem limitu.

A. Zwiększenie slotów bez zmiany sposobu uruchamiania zapytań tylko odroczy problem, a koszty będą wyższe.
C. Zwiększenie slotów z baseline 2000 nie rozwiąże problemu współbieżności, ponieważ nadal będzie więcej zapytań niż dostępnych slotów w szczycie.
D. Uruchamianie *wszystkich* zapytań jako interaktywne pogorszy problem, ponieważ interaktywne zapytania nie są kolejkowane i mogą prowadzić do jeszcze częstszych błędów.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Problem polega na przekroczeniu limitu współbieżności zapytań w BigQuery, pomimo posiadania rezerwacji 2000 slotów. Obecnie uruchomionych jest 1500 zapytań, co sugeruje, że problem nie leży w braku slotów *per se*, ale w sposobie ich wykorzystania. Odpowiedź B jest poprawna, ponieważ optymalizuje wykorzystanie dostępnych zasobów poprzez rozróżnienie priorytetów zapytań.

  • Zadania wsadowe (batch queries) dla SQL pipelines: Są to zapytania nieterminalne, które mogą być kolejkowane i uruchamiane w miarę dostępności slotów. To zapobiega natychmiastowemu przekroczeniu limitu współbieżności.
  • Zapytania interaktywne dla zapytań ad-hoc: Są to zapytania wymagające szybkiej odpowiedzi, które powinny mieć priorytetowy dostęp do zasobów.

To podejście pozwala na efektywne zarządzanie zasobami i uniknięcie błędów związanych z przekroczeniem limitu, bez konieczności zwiększania kosztów poprzez zakup dodatkowych slotów.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A i C: Zwiększenie liczby slotów (nawet z autoskalowaniem) nie rozwiązuje problemu współbieżności. Jeśli liczba zapytań przekracza dostępną liczbę slotów, problem nadal będzie występował, a jedynie koszty będą wyższe. Komentarze użytkowników, takie jak ToiToi, trafnie wskazują, że pytanie nie dotyczy zwiększania slotów.
  • D: Uruchamianie *wszystkich* zapytań jako interaktywne pogorszyłoby sytuację. Interaktywne zapytania nie są kolejkowane, co mogłoby prowadzić do jeszcze częstszych błędów przekroczenia limitu.

Dyskusja użytkowników w większości popiera odpowiedź B, a komentarze, takie jak te od raaad, judy_data i LP_PDE, potwierdzają zasadność tego rozwiązania. Użytkownik CGS22 początkowo sugerował A, ale później sam skorygował swoje stanowisko, argumentując, że zapytania wsadowe nie rozwiązują problemu współbieżności, co jest błędne – to właśnie kolejkowanie zapytań wsadowych jest kluczem do rozwiązania problemu.
