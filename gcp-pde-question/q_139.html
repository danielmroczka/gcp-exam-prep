You are building a new data pipeline to share data between two different types of applications: jobs generators and job runners. Your solution must scale to accommodate increases in usage and must accommodate the addition of new applications without negatively affecting the performance of existing ones. What should you do?

  • A. Create an API using App Engine to receive and send messages to the applications
  • B. Use a Cloud Pub/Sub topic to publish jobs, and use subscriptions to execute them Most Voted
  • C. Create a table on Cloud SQL, and insert and delete rows with the job information
  • D. Create a table on Cloud Spanner, and insert and delete rows with the job information

Odpowiedź: B.

Użycie Cloud Pub/Sub to idealne rozwiązanie dla tego scenariusza, ponieważ zapewnia asynchroniczną, skalowalną i niezawodną komunikację między aplikacjami. Publikowanie zadań do tematu Pub/Sub pozwala na odseparowanie generatorów zadań od ich wykonawców, co zwiększa elastyczność i odporność systemu. Subskrypcje umożliwiają niezależne skalowanie aplikacji wykonujących zadania, bez wpływu na generatory. Dodatkowo, łatwo można dodać nowe aplikacje jako subskrybentów, bez modyfikacji istniejącego kodu. Pub/Sub jest zaprojektowany do obsługi dużych wolumenów danych i wysokiej przepustowości, co jest kluczowe dla skalowalności.

A. App Engine API może stać się wąskim gardłem i nie skaluje się tak dobrze jak Pub/Sub przy dużym obciążeniu.
C. Cloud SQL nie jest przeznaczone do obsługi dużych wolumenów danych i może nie skalować się wystarczająco szybko dla rosnącego obciążenia.
D. Cloud Spanner jest potężną bazą danych, ale jest bardziej skomplikowana i kosztowna niż Pub/Sub, a w tym przypadku jego możliwości są zbędne.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Odpowiedź B, czyli użycie Cloud Pub/Sub do publikowania zadań i subskrypcji do ich wykonywania, jest zdecydowanie najlepszym rozwiązaniem w tym scenariuszu. Kluczowe wymagania to skalowalność i możliwość dodawania nowych aplikacji bez wpływu na istniejące. Cloud Pub/Sub został zaprojektowany specjalnie do tego typu zadań.

  • Skalowalność: Pub/Sub jest wysoce skalowalną usługą, która może obsłużyć duże wolumeny wiadomości i subskrybentów.
  • Odseparowanie (Decoupling): Pub/Sub odseparowuje generatory zadań od ich wykonawców, co zwiększa elastyczność i odporność systemu.
  • Łatwe dodawanie nowych aplikacji: Dodanie nowych aplikacji jako subskrybentów jest proste i nie wymaga modyfikacji istniejącego kodu.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość osób poprawnie identyfikuje Pub/Sub jako idealne rozwiązanie, podkreślając jego zalety w zakresie skalowalności i odseparowania. Komentarze użytkowników, takie jak te od jkhong, barnac1es, juliorevk, musumusu i zellck, trafnie wyjaśniają dlaczego Pub/Sub jest najlepszym wyborem.

Pozostałe odpowiedzi są mniej odpowiednie:

  • A (App Engine API): Może stać się wąskim gardłem przy dużym obciążeniu i nie skaluje się tak dobrze jak Pub/Sub.
  • C i D (Cloud SQL/Spanner): Bazy danych, choć potężne, są bardziej skomplikowane i kosztowne niż Pub/Sub w tym konkretnym przypadku. Nie są one zaprojektowane do obsługi strumieniowego przesyłania danych i asynchronicznej komunikacji w taki sposób, jak Pub/Sub.
