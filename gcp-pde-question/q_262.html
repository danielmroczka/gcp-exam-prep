You are on the data governance team and are implementing security requirements. You need to encrypt all your data in BigQuery by using an encryption key managed by your team. You must implement a mechanism to generate and store encryption material only on your on-premises hardware security module (HSM). You want to rely on Google managed solutions. What should you do?

  • A. Create the encryption key in the on-premises HSM, and import it into a Cloud Key Management Service (Cloud KMS) key. Associate the created Cloud KMS key while creating the BigQuery resources.
  • B. Create the encryption key in the on-premises HSM and link it to a Cloud External Key Manager (Cloud EKM) key. Associate the created Cloud KMS key while creating the BigQuery resources. Most Voted
  • C. Create the encryption key in the on-premises HSM, and import it into Cloud Key Management Service (Cloud HSM) key. Associate the created Cloud HSM key while creating the BigQuery resources.
  • D. Create the encryption key in the on-premises HSM. Create BigQuery resources and encrypt data while ingesting them into BigQuery.

Odpowiedź: B.

Wyjaśnienie: Pytanie dotyczy szyfrowania danych w BigQuery z wykorzystaniem klucza generowanego i przechowywanego na własnym sprzęcie HSM, przy jednoczesnym korzystaniu z zarządzanych rozwiązań Google. Cloud External Key Manager (Cloud EKM) jest zaprojektowany właśnie do tego celu – umożliwia wykorzystanie kluczy przechowywanych w zewnętrznych HSM, takich jak te znajdujące się w infrastrukturze on-premises. Tworząc klucz w HSM i linkując go do klucza Cloud EKM, można korzystać z BigQuery do szyfrowania danych, a kontrola nad kluczem pozostaje w rękach zespołu, na ich własnym sprzęcie. Cloud EKM działa jako pośrednik, umożliwiając BigQuery korzystanie z klucza bez jego kopiowania.

A. Cloud KMS nie pozwala na bezpośredni import kluczy z HSM, a jedynie na generowanie kluczy wewnątrz KMS.
C. Cloud HSM służy do przechowywania kluczy *w* chmurze Google, a nie do integracji z zewnętrznymi HSM.
D. Samo szyfrowanie danych podczas ingestii nie spełnia wymogu zarządzania kluczem przez zespół na własnym HSM i korzystania z zarządzanych rozwiązań Google.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Poprawną odpowiedzią jest B. Pytanie dotyczy sytuacji, w której klucz szyfrujący musi być generowany i przechowywany na własnym sprzęcie HSM (Hardware Security Module), a jednocześnie chcemy korzystać z zarządzanych usług Google. Cloud External Key Manager (Cloud EKM) jest zaprojektowany specjalnie do tego celu. Umożliwia on wykorzystanie kluczy przechowywanych w zewnętrznych systemach zarządzania kluczami, takich jak on-premises HSM, przez usługi Google Cloud, w tym BigQuery.

Dzięki Cloud EKM, klucz pozostaje pod pełną kontrolą zespołu, na ich własnym sprzęcie, a BigQuery może go używać do szyfrowania danych bez konieczności kopiowania klucza do chmury Google. Cloud EKM działa jako bezpieczny pośrednik.

Komentarze użytkowników, szczególnie te od raaad i f74ca0c, trafnie wyjaśniają dlaczego odpowiedź B jest prawidłowa i dlaczego pozostałe odpowiedzi są niepoprawne. Użytkownik raaad szczegółowo wyjaśnia działanie Cloud EKM i dlaczego nie można użyć Cloud HSM (C), ponieważ jest to rozwiązanie oparte na chmurze, a klucze nie są przechowywane on-premises.

Dlaczego pozostałe odpowiedzi są niepoprawne:

  • A: Cloud KMS nie umożliwia bezpośredniego importu kluczy z HSM.
  • C: Cloud HSM przechowuje klucze w chmurze Google, a nie integruje się z zewnętrznymi HSM.
  • D: Szyfrowanie danych podczas ingestii nie spełnia wymogu zarządzania kluczem przez zespół na własnym HSM i korzystania z zarządzanych rozwiązań Google.

Warto zauważyć, że początkowo użytkownik scaenruy wybrał odpowiedź C, ale późniejsze komentarze i głosowania wyraźnie wskazują na poprawność odpowiedzi B.
