You are selecting services to write and transform JSON messages from Cloud Pub/Sub to BigQuery for a data pipeline on Google Cloud. You want to minimize service costs. You also want to monitor and accommodate input data volume that will vary in size with minimal manual intervention. What should you do?

  • A. Use Cloud Dataproc to run your transformations. Monitor CPU utilization for the cluster. Resize the number of worker nodes in your cluster via the command line.
  • B. Use Cloud Dataproc to run your transformations. Use the diagnose command to generate an operational output archive. Locate the bottleneck and adjust cluster resources.
  • C. Use Cloud Dataflow to run your transformations. Monitor the job system lag with Stackdriver. Use the default autoscaling setting for worker instances. Most Voted
  • D. Use Cloud Dataflow to run your transformations. Monitor the total execution time for a sampling of jobs. Configure the job to use non-default Compute Engine machine types when needed.

Odpowiedź: C.

**Wyjaśnienie:**
Cloud Dataflow jest usługą w pełni zarządzaną i bezserwerową, co czyni ją idealnym wyborem do przetwarzania strumieniowego danych z Pub/Sub do BigQuery. Dzięki wbudowanej funkcji automatycznego skalowania (autoscaling), usługa samodzielnie dostosowuje liczbę zasobów obliczeniowych do aktualnego natężenia ruchu, co eliminuje potrzebę ręcznej interwencji i optymalizuje koszty. Monitorowanie opóźnienia systemowego (system lag) w Stackdriver pozwala na bieżąco śledzić wydajność potoku i upewnić się, że dane są przetwarzane bez opóźnień. Wybór Dataflow zamiast Dataproc w tym scenariuszu jest lepszy, ponieważ Dataflow został zaprojektowany z myślą o dynamicznym skalowaniu bez konieczności zarządzania klastrami. Takie podejście najlepiej spełnia wymagania dotyczące minimalizacji kosztów oraz automatyzacji procesów przy zmiennym wolumenie danych wejściowych.

**Analiza pozostałych opcji:**
*   **A:** Wykorzystanie Cloud Dataproc z ręcznym skalowaniem liczby węzłów przez linię komend narusza wymóg minimalnej interwencji manualnej.
*   **B:** Diagnozowanie wąskich gardeł i ręczne korygowanie zasobów klastra jest procesem czasochłonnym i nieefektywnym przy dynamicznie zmieniającym się obciążeniu.
*   **D:** Monitorowanie czasu wykonania wybranych zadań i ręczna zmiana typów maszyn jest rozwiązaniem znacznie mniej zautomatyzowanym i mniej precyzyjnym niż natywny autoscaling w Dataflow.

Sugerowana odpowiedź: C

Prawidłową odpowiedzią jest C. Use Cloud Dataflow... Monitor the job system lag... Use the default autoscaling....

Dlaczego?

Wymagania: 1. Data pipeline (Pub/Sub to BigQuery): Dataflow jest do tego stworzony (ETL). 2. Minimize service costs & minimal manual intervention: Dataflow jest w pełni zarządzany (serverless) i posiada Autoscaling. Autoscaling automatycznie dodaje lub usuwa workery w zależności od obciążenia, co optymalizuje koszty (płacisz tylko za to, czego używasz) i eliminuje potrzebę ręcznego skalowania. 3. Monitor volume: Metryka "System Lag" (opóźnienie systemu) w Dataflow/Stackdriver jest kluczowym wskaźnikiem zdrowia pipeline'u strumieniowego. Jeśli rośnie, oznacza to, że system nie nadąża.

Dlaczego inne odpowiedzi są nieprawidłowe?

  • A. Dataproc + manual resize: Wymaga ręcznej interwencji ("via command line"), co narusza wymagania.
  • B. Dataproc + diagnose: Diagnostyka to nie rozwiązanie produkcyjne na "variable input volume". To podejście reaktywne i manualne.
  • D. Dataflow + manual machine types: Sugeruje ręczne dobieranie typów maszyn na podstawie próbkowania. Autoscaling robi to lepiej i automatycznie.
