You are deploying a batch pipeline in Dataflow. This pipeline reads data from Cloud Storage, transforms the data, and then writes the data into BigQuery. The security team has enabled an organizational constraint in Google Cloud, requiring all Compute Engine instances to use only internal IP addresses and no external IP addresses. What should you do?

  • A. Ensure that your workers have network tags to access Cloud Storage and BigQuery. Use Dataflow with only internal IP addresses.
  • B. Ensure that the firewall rules allow access to Cloud Storage and BigQuery. Use Dataflow with only internal IPs.
  • C. Create a VPC Service Controls perimeter that contains the VPC network and add Dataflow, Cloud Storage, and BigQuery as allowed services in the perimeter. Use Dataflow with only internal IP addresses.
  • D. Ensure that Private Google Access is enabled in the subnetwork. Use Dataflow with only internal IP addresses. Most Voted

Odpowiedź: D.

Organizacyjna restrykcja wymuszająca użycie wyłącznie wewnętrznych adresów IP dla instancji Compute Engine wpływa na sposób, w jaki Dataflow komunikuje się z innymi usługami Google Cloud. Dataflow używa instancji Compute Engine jako workerów do przetwarzania danych. Aby Dataflow mógł komunikować się z Cloud Storage i BigQuery bez użycia zewnętrznych adresów IP, należy włączyć Private Google Access w podsieci, w której uruchamiane są workery Dataflow. Private Google Access umożliwia instancjom bez zewnętrznych adresów IP dostęp do usług Google Cloud API i usług, takich jak Cloud Storage i BigQuery, poprzez sieć Google. Użycie wyłącznie wewnętrznych adresów IP w Dataflow jest zgodne z restrykcją organizacyjną, a Private Google Access zapewnia niezbędną łączność.

A. Sieciowe tagi nie rozwiązują problemu braku dostępu do usług Google Cloud z instancji bez zewnętrznych IP.
B. Reguły zapory kontrolują ruch sieciowy, ale nie umożliwiają dostępu do usług Google Cloud z instancji bez zewnętrznych IP.
C. VPC Service Controls to potężne narzędzie do kontroli dostępu, ale nie jest konieczne do spełnienia tej konkretnej restrykcji organizacyjnej i jest bardziej złożone niż Private Google Access.

Dodatkowe źródła:

*   **Private Google Access:** https://cloud.google.com/vpc/docs/private-google-access
*   **Dataflow i VPC:** https://cloud.google.com/dataflow/docs/guides/using-vpc
*   **Organizational Policies:** https://cloud.google.com/resource-manager/docs/organization-policy/overview

Sugerowana odpowiedź: D

Poprawną odpowiedzią jest D – włączenie Private Google Access w podsieci, w której uruchamiane są workery Dataflow. Organizacyjna restrykcja wymaga użycia wyłącznie wewnętrznych adresów IP dla instancji Compute Engine. Dataflow wykorzystuje te instancje jako workery. Private Google Access umożliwia instancjom bez zewnętrznych adresów IP dostęp do usług Google Cloud, takich jak Cloud Storage i BigQuery, poprzez sieć Google. Jest to najprostsze i najbardziej bezpośrednie rozwiązanie spełniające wymagania restrykcji organizacyjnej.

Dyskusja użytkowników potwierdza tę odpowiedź. Wielu użytkowników, w tym GCP001 i Tryolabs, wskazuje na dokumentację Google Cloud, która wyraźnie potwierdza, że Private Google Access umożliwia dostęp do usług Google Cloud z instancji posiadających tylko wewnętrzne adresy IP. Użytkownik BIGQUERY_ALT_ALT również podkreśla, że Private Google Access jest prostszym i bardziej odpowiednim rozwiązaniem niż VPC Service Controls w tym konkretnym przypadku.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • A: Sieciowe tagi nie zapewniają dostępu do usług Google Cloud z instancji bez zewnętrznych adresów IP.
  • B: Reguły zapory kontrolują ruch sieciowy, ale nie umożliwiają dostępu do usług Google Cloud z instancji bez zewnętrznych IP.
  • C: VPC Service Controls to potężne narzędzie do kontroli dostępu, ale jest bardziej złożone niż Private Google Access i nie jest konieczne do spełnienia tej konkretnej restrykcji organizacyjnej. Jak zauważają użytkownicy, VPC Service Controls służą do kontrolowania dostępu, a nie do zapewnienia łączności w przypadku braku zewnętrznych adresów IP.

Podsumowując, Private Google Access jest najprostszym, najbardziej efektywnym i zgodnym z dokumentacją rozwiązaniem problemu.
