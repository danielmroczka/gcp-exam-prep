You are designing storage for very large text files for a data pipeline on Google Cloud. You want to support ANSI SQL queries. You also want to support compression and parallel load from the input locations using Google recommended practices. What should you do?

  • A. Transform text files to compressed Avro using Cloud Dataflow. Use BigQuery for storage and query.
  • B. Transform text files to compressed Avro using Cloud Dataflow. Use Cloud Storage and BigQuery permanent linked tables for query. Most Voted
  • C. Compress text files to gzip using the Grid Computing Tools. Use BigQuery for storage and query.
  • D. Compress text files to gzip using the Grid Computing Tools. Use Cloud Storage, and then import into Cloud Bigtable for query.

Odpowiedź: B.

**Wyjaśnienie:**
Wybór opcji B jest podyktowany najlepszymi praktykami Google Cloud dotyczącymi przetwarzania dużych zbiorów danych. Format Avro jest formatem binarnym, który w przeciwieństwie do Gzip jest "podzielny" (splittable), co umożliwia w pełni równoległe czytanie i ładowanie danych przez BigQuery oraz Cloud Dataflow. Użycie Cloud Dataflow pozwala na skalowalną transformację ogromnych plików tekstowych, a przechowywanie ich w Cloud Storage z wykorzystaniem tabel zewnętrznych (permanent linked tables) umożliwia natychmiastowe wykonywanie zapytań ANSI SQL bez konieczności kosztownego i czasochłonnego importu danych do natywnego formatu BigQuery. Taka architektura pozwala na zachowanie elastyczności potoku danych (data pipeline) i separację warstwy składowania od warstwy analitycznej. Jest to rozwiązanie optymalne kosztowo i wydajnościowo przy pracy z bardzo dużymi plikami, które muszą być dostępne dla standardowego SQL-a.

**Analiza pozostałych opcji:**
*   **A:** Choć BigQuery oferuje świetną wydajność, wymuszone ładowanie wszystkich danych do natywnego magazynu BigQuery może być mniej efektywne w dynamicznym potoku danych niż bezpośrednie zapytania do plików Avro w Cloud Storage.
*   **C:** Kompresja Gzip nie pozwala na równoległe przetwarzanie plików (nie jest splittable), co drastycznie spowalnia proces ładowania i analizy bardzo dużych zbiorów danych.
*   **D:** Cloud Bigtable to baza danych NoSQL, która nie obsługuje natywnie zapytań ANSI SQL, co dyskwalifikuje tę opcję w świetle wymagań egzaminacyjnych.

**Dodatkowe źródła:**
*   BigQuery - Introduction to external data sources
*   Google Cloud - Avro as a preferred format for BigQuery

Analiza i Wyjaśnienie

Sugerowana odpowiedź: B

Prawidłową odpowiedzią jest B. Transform text files to compressed Avro... Use Cloud Storage and BigQuery permanent linked tables....

Dlaczego?

Wymagania: 1. Very large text files: Tekst (np. CSV/JSON) zajmuje dużo miejsca i jest wolny w parsowaniu. 2. Support compression and parallel load: To kluczowe. GZIP na pliku tekstowym (CSV.GZ) nie jest splittable (niepodzielny). Oznacza to, że BigQuery/Dataflow musi czytać taki plik jednym wątkiem, co zabija wydajność przy "very large files". 3. Avro: Jest to format binarny, który wspiera kompresję (np. Snappy, Deflate) i jest splittable (można go czytać równolegle wieloma workerami). 4. ANSI SQL Queries: BigQuery obsługuje SQL. 5. Designing storage... input locations: Opcja B sugeruje trzymanie danych w GCS (jako Avro) i odpytywanie ich przez BigQuery External Tables (Linked Tables). Jest to tzw. "Data Lake" pattern – tanie składowanie w GCS, moc SQL z BigQuery, bez konieczności pełnego ładowania (ingest) do natywnego storage BQ (choć opcja A też jest dobra, B jest często preferowana dla elastyczności i "storage design" w starszych egzaminach).

Dlaczego inne odpowiedzi są nieprawidłowe?

  • C i D. Compress to gzip: GZIP nie jest podzielny (not splittable) dla plików tekstowych. Nie spełnia wymogu "parallel load".
  • A. Use BigQuery for storage: To też bardzo dobre rozwiązanie (Native Storage jest szybsze niż External), ale kluczowa różnica między A i B w kontekście tego pytania to często nacisk na "designing storage... input locations" i elastyczność. Jednak najważniejsze jest odrzucenie C i D przez brak parallelizmu GZIP. Między A i B, B jest często wybierane jako "Google recommended practice" dla Data Lake na GCS + Federated Query.
