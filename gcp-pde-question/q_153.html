You operate an IoT pipeline built around Apache Kafka that normally receives around 5000 messages per second. You want to use Google Cloud Platform to create an alert as soon as the moving average over 1 hour drops below 4000 messages per second. What should you do?

  • A. Consume the stream of data in Dataflow using Kafka IO. Set a sliding time window of 1 hour every 5 minutes. Compute the average when the window closes, and send an alert if the average is less than 4000 messages. Most Voted
  • B. Consume the stream of data in Dataflow using Kafka IO. Set a fixed time window of 1 hour. Compute the average when the window closes, and send an alert if the average is less than 4000 messages.
  • C. Use Kafka Connect to link your Kafka message queue to Pub/Sub. Use a Dataflow template to write your messages from Pub/Sub to Bigtable. Use Cloud Scheduler to run a script every hour that counts the number of rows created in Bigtable in the last hour. If that number falls below 4000, send an alert.
  • D. Use Kafka Connect to link your Kafka message queue to Pub/Sub. Use a Dataflow template to write your messages from Pub/Sub to BigQuery. Use Cloud Scheduler to run a script every five minutes that counts the number of rows created in BigQuery in the last hour. If that number falls below 4000, send an alert.

Odpowiedź: A.

Opcja A jest poprawna, ponieważ Dataflow z Kafka IO pozwala na bezpośrednie przetwarzanie strumienia danych z Kafki. Użycie sliding time window o długości 1 godziny i częstotliwości 5 minut umożliwia obliczanie średniej ruchomej w sposób ciągły i efektywny. Po zamknięciu okna, obliczana jest średnia, a jeśli jest ona poniżej 4000 wiadomości na sekundę, wysyłany jest alert. To podejście zapewnia szybką reakcję na spadek przepustowości, co jest kluczowe w monitoringu IoT.  Sliding windows są idealne do obliczania ruchomych średnich, ponieważ nie wymagają ponownego przetwarzania danych dla każdego okna.

A. Jest to najbardziej efektywne i bezpośrednie rozwiązanie do monitorowania średniej ruchomej w czasie rzeczywistym.
B. Użycie fixed time window nie pozwala na ciągłe monitorowanie średniej ruchomej, a jedynie na obliczanie średniej za pełną godzinę, co opóźnia reakcję na spadek przepustowości.
C. Użycie Kafka Connect, Pub/Sub, Bigtable i Cloud Scheduler jest zbyt skomplikowane i wprowadza niepotrzebne opóźnienia w procesie alertowania.
D. Podobnie jak w C, użycie Kafka Connect, Pub/Sub, BigQuery i Cloud Scheduler jest bardziej złożone i mniej efektywne niż bezpośrednie przetwarzanie strumienia w Dataflow.

Sugerowana odpowiedź: A

Odpowiedź A jest prawidłowa, ponieważ oferuje najbardziej efektywne i bezpośrednie rozwiązanie do monitorowania średniej ruchomej w czasie rzeczywistym. Dataflow z Kafka IO umożliwia bezpośrednie przetwarzanie strumienia danych z Kafki. Kluczowe jest użycie sliding time window (okna czasowego z przesuwem) o długości 1 godziny i częstotliwości 5 minut. Pozwala to na ciągłe obliczanie średniej ruchomej bez konieczności ponownego przetwarzania danych dla każdego okna. W momencie zamknięcia okna, obliczana jest średnia, a jeśli jest ona poniżej 4000 wiadomości na sekundę, wysyłany jest alert.

Dyskusja użytkowników w dużej mierze potwierdza tę odpowiedź. Większość użytkowników wskazuje A jako prawidłową, argumentując, że Kafka IO i Dataflow stanowią optymalne połączenie, a sliding window jest niezbędny do obliczania ruchomej średniej. Użytkownik zellck trafnie odnosi się do dokumentacji Dataflow, podkreślając rolę hopping windows (sliding windows w Apache Beam) w obliczaniu ruchomych średnich.

Dlaczego inne odpowiedzi są nieprawidłowe:

  • B: Użycie fixed time window (okna czasowego o stałej długości) nie pozwala na ciągłe monitorowanie średniej ruchomej. Obliczenia są wykonywane tylko po upływie pełnej godziny, co opóźnia reakcję na spadek przepustowości.
  • C i D: Użycie Kafka Connect, Pub/Sub, Bigtable/BigQuery i Cloud Scheduler jest nadmiernie skomplikowane i wprowadza niepotrzebne opóźnienia. Przetwarzanie danych przez kilka usług zwiększa złożoność i potencjalne punkty awarii, a także wydłuża czas reakcji na zmiany w przepustowości.

Podsumowując, odpowiedź A jest najbardziej efektywna, skalowalna i bezpośrednia, co czyni ją najlepszym rozwiązaniem dla tego scenariusza.
