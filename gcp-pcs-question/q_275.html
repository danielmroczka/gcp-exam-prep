Question 275 / 361

Your organization is migrating business critical applications to Google Cloud across multiple projects. You only have the required IAM permission at the Google Cloud organization level. You want to grant project access to support engineers from two partner organizations using their existing identity provider (IdP) credentials. What should you do?
  • A. Create two single sign-on (SSO) profiles for the internal and partner IdPs by using SSO for Cloud Identity.
  • B. Create users manually by using the Google Cloud console. Assign the users to groups.
  • C. Create two workforce identity pools for the partner IdPs.
  • D. Sync user identities from their existing IdPs to Cloud Identity by using Google Cloud Directory Sync (GCDS).
Show Suggested Answer Hide Answer
Suggested Answer: C

Wyjaśnienie

Odpowiedź: C.

Wyjaśnienie (Dlaczego poprawna):
C. Workforce identity pools: Jest to nowa funkcja Google Cloud (następca tradycyjnej federacji dla pracowników), która pozwala na przyznanie dostępu do konsoli i API Google Cloud bezpośrednio użytkownikom z zewnętrznych systemów tożsamości (Azure AD, Okta, itp.) BEZ konieczności synchronizowania ich kont do Cloud Identity (no-sync model). Jest to idealne dla partnerów, ponieważ ich konta żyją tylko w ich własnym systemie.

Dlaczego inne są błędne:
A. SSO for Cloud Identity: Wymaga wcześniejszego 'istnienia' (zsynchronizowania) konta w Cloud Identity, co przeczy wymogowi 'using their existing credentials' bez nadmiarowej pracy.
B. Manual users: Nie skaluje się i nie zapewnia bezpieczeństwa (brak powiązania z IdP partnera).
D. GCDS: Wymaga fizycznej synchronizacji kont, co jest przeciwieństwem nowoczesnego modelu federacyjnego typu workforce pool.

Sugerowana odpowiedź: C

Workforce Identity Federation pozwala na dostęp do GCP bez tworzenia 'cieni' użytkowników (shadow accounts) w Cloud Identity.
