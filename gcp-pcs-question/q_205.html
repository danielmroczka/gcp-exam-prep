Question 205 / 361

You are running applications outside Google Cloud that need access to Google Cloud resources. You are using workload identity federation to grant external identities Identity and Access Management (IAM) roles to eliminate the maintenance and security burden associated with service account keys. You must protect against attempts to spoof another user's identity and gain unauthorized access to Google Cloud resources. What should you do? (Choose two.)
  • A. Enable data access logs for IAM APIs.
  • B. Limit the number of external identities that can impersonate a service account.
  • C. Use a dedicated project to manage workload identity pools and providers.
  • D. Use immutable attributes in attribute mappings.
  • E. Limit the resources that a service account can access.
Show Suggested Answer Hide Answer
Suggested Answer: CD

Wyjaśnienie

Odpowiedź: CD.

Wyjaśnienie (Dlaczego poprawna):
C. Dedicated project for pools: Centralizacja zarządzania federacją w dedykowanym, mocno zabezpieczonym projekcie ogranicza 'blast radius'. Tylko nieliczni administratorzy mają tam dostęp, co zapobiega nieuprawnionym zmianom w mapowaniu tożsamości.
D. Immutable attributes in mappings: Przy mapowaniu tożsamości zewnętrznej (np. z AWS czy Azure) do GCP, należy używać atrybutów, których użytkownik nie może sam zmienić (np. unikalny ID subskrypcji lub stały ID użytkownika). Jeśli użyjesz atrybutów zmiennych (np. nazwa grupy), napastnik po przejęciu kontroli w systemie zewnętrznym mógłby zmienić swój atrybut na 'admin' i w ten sposób sfałszować tożsamość w GCP (spoofing).

Dlaczego inne są błędne:
A. Data access logs: Pomagają w audycie, ale same w sobie nie blokują 'spoofingu'.
B. Limit number of identities: Nie zapobiega to sytuacji, gdy JEDNA z tych tożsamości jest sfałszowana.
E. Limit resources: Dobra praktyka (Least Privilege), ale nie adresuje bezpośrednio ryzyka sfałszowania tożsamości (identyfikacji).
