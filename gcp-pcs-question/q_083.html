Question 83 / 361

You need to optimize Cloud DLP usage to reduce cost. Data is in GCS and BigQuery. Which options should you recommend?
  • A. Set rowsLimit on BQ outside US, bytesLimitPerFile on multiregional GCS.
  • B. Set rowsLimit on BQ outside US, minimize transformation units.
  • C. Use rowsLimit and bytesLimitPerFile to sample data and use CloudStorageRegexFileSet.
  • D. Use FindingLimits and TimespanConfig.
Show Suggested Answer Hide Answer
Suggested Answer: C

Wyjaśnienie

Odpowiedź: C.

Wyjaśnienie (Dlaczego poprawna):
C. Sampling and Filtering: Największy wpływ na koszt DLP ma ilość skanowanych danych. Użycie `rowsLimit` (dla BigQuery) i `bytesLimitPerFile` (dla GCS) pozwala na przeskanowanie reprezentatywnej PRÓBKI danych zamiast całości. `CloudStorageRegexFileSet` dodatkowo ogranicza skanowanie tylko do konkretnych typów plików (np. tylko logi .txt).

Dlaczego inne są błędne:
A, B. Regiony (outside US) nie mają bezpośredniego wpływu na optymalizację kosztów skanowania jako taką (cennik jest zazwyczaj spójny lub różnice są minimalne w porównaniu do redukcji wolumenu danych).
D. FindingLimits: Ogranicza liczbę raportowanych znalezisk, ale DLP i tak musi przeskanować cały plik, aby je znaleźć, więc nie redukuje to kosztów operacji skanowania.

Konsensus społeczności ExamTopics: Zdecydowana większość na C. Próbkowanie (sampling) to kluczowa technika optymalizacji kosztów w Big Data.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Głęboka analiza: Jeśli masz 1TB logów, skanowanie 100MB z każdego pliku (sampling) zazwyczaj wystarczy, aby wykryć, czy dany proces zaczął omyłkowo zapisywać PII, a kosztuje ułamek ceny pełnego skanu.
