Question 222 / 361

Minimize service account keys for on-prem Windows apps. Implement WIF with IdP on-prem. What to do?
  • A. Workload identity pool with ADFS, rule for impersonation.
  • B. Workload identity pool with ADFS, all principals impersonate.
  • C. OIDC on same machine.
  • D. OIDC, all principals impersonate.
Show Suggested Answer Hide Answer
Suggested Answer: A

Wyjaśnienie

Odpowiedź: A.

Wyjaśnienie (Dlaczego poprawna):
A. WIF with ADFS: Windows Server natywnie używa ADFS (Active Directory Federation Services), który wspiera SAML/OIDC. Workload Identity Federation pozwala na wymianę tokena ADFS na token GCP. Dodanie reguły (A) zapewnia, że tylko konkretne maszyny/aplikacje mogą przejąć tożsamość konta serwisowego w chmurze (zasada najniższych uprawnień).

Dlaczego inne są błędne:
B, D. All principals: Skrajnie niebezpieczne. Każdy użytkownik Twojego AD mógłby wtedy przejąć uprawnienia administratora w chmurze.
C. OIDC on same machine: Sama obecność protokołu nie rozwiązuje problemu autoryzacji i wymiany tożsamości bez mechanizmu WIF.

Konsensus społeczności ExamTopics: 100% na A. To standardowa architektura eliminacji kluczy JSON w środowisku Windows/AD.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Głęboka analiza: WIF używa tzw. 'Security Token Service' (STS) Google, który weryfikuje podpis Twojego lokalnego ADFS i na tej podstawie wydaje token OAuth2.
