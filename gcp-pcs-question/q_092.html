Question 92 / 361

Web app in staging on GKE includes user data without validation. Attackers could execute commands in victim's browser. How to prevent and fix?
  • A. Cloud IAP based on IP.
  • B. HTTPS LB and Cloud Armor.
  • C. Web Security Scanner for outdated libraries.
  • D. Web Security Scanner in staging to simulate XSS and use contextual auto-escaping.
Show Suggested Answer Hide Answer
Suggested Answer: D

Wyjaśnienie

Odpowiedź: D.

Wyjaśnienie (Dlaczego poprawna):
D. WSS and Auto-escaping: Opisany atak to Cross-Site Scripting (XSS). Web Security Scanner potrafi symulować takie ataki w środowisku stagingowym i je wykrywać. Rozwiązaniem programistycznym ('fix') jest użycie mechanizmów auto-escaping w silniku szablonów, co uniemożliwia interpretację danych użytkownika jako kodu JS.

Dlaczego inne są błędne:
A. IAP: Chroni przed nieautoryzowanym wejściem, ale nie naprawia błędów w kodzie samej aplikacji (XSS zadziała dla zalogowanego użytkownika).
B. Cloud Armor: Może blokować próby ataku, ale pytanie prosi o 'prevent and FIX' (naprawę), a naprawa musi nastąpić w kodzie.
C. Outdated libraries: To tylko jeden z wielu powodów XSS. WSS szuka błędów logicznych w wyświetlaniu danych, a nie tylko wersji bibliotek.

Konsensus społeczności ExamTopics: 100% na D. Połączenie testów DAST i poprawnych praktyk kodowania to jedyna droga.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: D

Głęboka analiza: XSS polega na wstrzyknięciu złośliwego skryptu, który uruchamia się u innego użytkownika. Kontekstowy auto-escaping sprawia, że znaki takie jak `<` stają się bezpiecznym `<`.
