Question 59 / 361

A patch for a vulnerability has been released, and a DevOps team needs to update their running containers in Google Kubernetes Engine (GKE). How should the DevOps team accomplish this?
  • A. Use Puppet or Chef to push out the patch to the running container.
  • B. Verify that auto upgrade is enabled; if so, Google will upgrade the nodes in a GKE cluster.
  • C. Update the application code or apply a patch, build a new image, and redeploy it.
  • D. Configure containers to automatically upgrade when the base image is available in Container Registry.
Show Suggested Answer Hide Answer
Suggested Answer: C

Wyjaśnienie

Odpowiedź: C.

Wyjaśnienie (Dlaczego poprawna):
C. Build a new image and redeploy: Kontenery z założenia są efemeryczne i niezmienne (immutable). Nie patchuje się działających kontenerów 'na żywo'. Prawidłowy proces to: poprawka w kodzie/Dockerfile -> nowy obraz -> nowe wdrożenie (np. rolling update).

Dlaczego inne są błędne:
A. Puppet/Chef: Narzędzia do konfiguracji OS, nie stosuje się ich do patchowania wnętrza kontenerów.
B. Auto upgrade nodes: Aktualizuje system operacyjny węzłów (worker nodes), ale nie aplikacje/biblioteki wewnątrz Twoich kontenerów.
D. Auto upgrade on registry change: GKE nie posiada natywnej funkcji, która automatycznie restartuje pody tylko dlatego, że pojawił się nowy tag w Container Registry (wymaga to akcji CI/CD lub kontrolera).

Sugerowana odpowiedź: C

Zasada 'Continer Immutability' to fundament bezpieczeństwa i powtarzalności operacji w Kubernetes.
