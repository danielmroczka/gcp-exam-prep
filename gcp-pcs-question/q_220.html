Question 220 / 361

You manage one of your organization's Google Cloud projects (Project A). A VPC Service Control (SC) perimeter is blocking API access requests to this project, including Pub/Sub. A resource running under a service account in another project (Project B) needs to collect messages from a Pub/Sub topic in your project. Project B is not included in a VPC SC perimeter. You need to provide access from Project B to the Pub/Sub topic in Project A using the principle of least privilege. What should you do?
  • A. Configure an ingress policy for the perimeter in Project A, and allow access for the service account in Project B to collect messages.
  • B. Create an access level that allows a developer in Project B to subscribe to the Pub/Sub topic that is located in Project A.
  • C. Create a perimeter bridge between Project A and Project B to allow the required communication between both projects.
  • D. Remove the Pub/Sub API from the list of restricted services in the perimeter configuration for Project A.
Show Suggested Answer Hide Answer
Suggested Answer: A

Wyjaśnienie

Odpowiedź: A.

Wyjaśnienie (Dlaczego poprawna):
A. Ingress policy for the perimeter: VPC Service Controls pozwala na definiowanie precyzyjnych reguł 'Ingress'. Reguła ta pozwala na dostęp do usług wewnątrz obwodu (Project A) tożsamościom z zewnątrz (Project B). Jest to zgodne z zasadą najmniejszych uprawnień, ponieważ zezwalasz tylko na konkretną operację dla konkretnego konta serwisowego.

Dlaczego inne są błędne:
B. Access Level: Służy do wpuszczania na podstawie IP/Urządzenia, ale nie rozwiązuje problemu blokady między-projektowej dla usług API wewnątrz VPC SC bez reguły ingress.
C. Perimeter Bridge: Łączy dwa obwody. Tutaj Project B nie jest w żadnym obwodzie, więc nie ma czego 'mostkować'.
D. Remove Pub/Sub: Całkowicie osłabia bezpieczeństwo obwodu dla wszystkich innych zasobów.
