Question 345 / 361

dedicated Cloud Interconnect. Private access from on-prem hosts to GCS/BQ without public internet. What to do?
  • A. Shared VPC + PGA.
  • B. VPC peering + Cloud Firewall.
  • C. PGA for on-premises + private.googleapis.com DNS.
  • D. Cloud NAT + 199.36.153.8/30 DNS.
Show Suggested Answer Hide Answer
Suggested Answer: C

Wyjaśnienie

Odpowiedź: C.

Wyjaśnienie (Dlaczego poprawna):
C. PGA for on-premises + private VIP: Aby hosty lokalne (on-prem) mogły 'rozmawiać' z usługami Google przez Interconnect, muszą kierować zapytania DNS do zakresu IP `199.36.153.8/30` (private.googleapis.com). Google Cloud zajmuje się wtedy routingiem tego ruchu przez prywatne łącze zamiast przez internet.

Dlaczego inne są błędne:
A. PGA: Dotyczy ruchu Z chmury, PGA 'for on-premises' to osobna, choć powiązana funkcja.
B. VPC Peering: Nie łączy on-prem z chmurą i nie ma wpływu na dostęp do publicznych API Google (GCS/BQ).
D. Cloud NAT: Służy do WYJŚCIA na internet, a tu chcemy go UNIKNĄĆ.

Konsensus społeczności ExamTopics: 100% poparcia dla C. To standardowa konfiguracja 'Hybrid Private Access'.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: C

Głęboka analiza: Wybór między `private` (wszystkie API) a `restricted` (tylko VPC SC) zależy od Twojej polityki bezpieczeństwa, ale technicznie oba rozwiązania PGA for on-prem działają podobnie.
