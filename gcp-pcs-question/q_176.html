Question 176 / 361

You need to enforce a security policy in your Google Cloud organization that prevents users from exposing objects in their buckets externally. There are currently no buckets in your organization. Which solution should you implement proactively to achieve this goal with the least operational overhead?
  • A. Create an hourly cron job to run a Cloud Function that finds public buckets and makes them private.
  • B. Enable the constraints/storage.publicAccessPrevention constraint at the organization level.
  • C. Enable the constraints/storage.uniformBucketLevelAccess constraint at the organization level.
  • D. Create a VPC Service Controls perimeter that protects the storage.googleapis.com service in your projects that contains buckets. Add any new project that contains a bucket to the perimeter.
Show Suggested Answer Hide Answer
Suggested Answer: B

Wyjaśnienie

Odpowiedź: B.

Wyjaśnienie (Dlaczego poprawna):
B. constraints/storage.publicAccessPrevention: Jest to polityka organizacji zaprojektowana dokładnie do tego celu. Po jej włączeniu, nikt w organizacji nie może nadać bucketowi (lub obiektowi) uprawnień publicznych (allUsers). Jest to rozwiązanie 'proaktywne' i o 'najniższym nakładzie', ponieważ działa automatycznie na wszystkie nowe i istniejące zasoby.

Dlaczego inne są błędne:
A. Hourly cron job: To rozwiązanie 'reaktywne' (przez godzinę dane mogą być publiczne) i zwiększa 'operational overhead'.
C. Uniform bucket-level access: Bardzo dobre, ale samo w sobie nie blokuje nadania uprawnienia `allUsers` na poziomie bucketu.
D. VPC Service Controls: Zbyt skomplikowane do tak prostego celu i nie blokuje udostępnienia danych przez API dla kogoś z zewnątrz, jeśli IAM na to pozwoli.
