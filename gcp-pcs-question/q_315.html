Question 315 / 361

Your organization is building a real-time recommendation engine using ML models that process live user activity data stored in BigQuery and Cloud Storage. Each new model developed is saved to Artifact Registry. This new system deploys models to Google Kubernetes Engine, and uses Pub/Sub for message queues. Recent industry news have been reporting attacks exploiting ML model supply chains. You need to enhance the security in this serverless architecture, specifically against risks to the development and deployment pipeline. What should you do?
  • A. Enable container image vulnerability scanning during development and pre-deployment. Enforce Binary Authorization on images deployed from Artifact Registry to your continuous integration and continuous deployment (CI/CD) pipeline.
  • B. Thoroughly sanitize all training data prior to model development to reduce risk of poisoning attacks. Use IAM for authorization, and apply role-based restrictions to code repositories and cloud services.
  • C. Limit external libraries and dependencies that are used for the ML models as much as possible. Continuously rotate encryption keys that are used to access the user data from BigQuery and Cloud Storage.
  • D. Develop strict firewall rules to limit external traffic to Cloud Run instances. Integrate intrusion detection systems (IDS) for real-time anomaly detection on Pub/Sub message flows.
Show Suggested Answer Hide Answer
Suggested Answer: A

Wyjaśnienie

Odpowiedź: A.

Wyjaśnienie (Dlaczego poprawna):
A. Vulnerability scanning + Binary Authorization: Pytanie dotyczy 'supply chain attacks' (ataków na łańcuch dostaw). Najlepszą obroną jest: 1. Automatyczne skanowanie obrazów w Artifact Registry pod kątem malware i luk (CVE). 2. Binary Authorization, które blokuje uruchomienie jakiegokolwiek modelu/kontenera, który nie został podpisany (atestowany) jako bezpieczny przez Twój system budowania.

Dlaczego inne są błędne:
B. Training data sanitization: Chroni przed 'poisoning' (zatruciem modelu), ale nie chroni samego potoku deploymentu (pipeline) przed wstrzyknięciem złośliwego kodu do obrazu kontenera.
C. Rotate keys: Dobra praktyka, ale nie adresuje ryzyka 'supply chain' (czyli zawartości obrazów).
D. Firewall/IDS: Chronią system po uruchomieniu (runtime), ale nie zapobiegają wdrożeniu złośliwego artefaktu zbudowanego w Twoim pipeline.

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Ochrona łańcucha dostaw (Supply Chain Security) wymaga weryfikacji artefaktów przed ich dopuszczeniem do środowiska wykonawczego.
