Question 8 / 361

A customer implements Cloud Identity-Aware Proxy for their ERP system hosted on Compute Engine. Their security team wants to add a security layer so that the ERP systems only accept traffic from Cloud Identity-Aware Proxy. What should the customer do to meet these requirements?
  • A. Make sure that the ERP system can validate the JWT assertion in the HTTP requests.
  • B. Make sure that the ERP system can validate the identity headers in the HTTP requests.
  • C. Make sure that the ERP system can validate the x-forwarded-for headers in the HTTP requests.
  • D. Make sure that the ERP system can validate the user's unique identifier headers in the HTTP requests.
Show Suggested Answer Hide Answer
Suggested Answer: A

Wyjaśnienie

Odpowiedź: A.

Wyjaśnienie (Dlaczego poprawna):
A. JWT assertion: Cloud IAP do każdego żądania dołącza nagłówek 'x-goog-iap-jwt-assertion' zawierający podpisany cyfrowo token JWT. Aplikacja powinna go zweryfikować za pomocą publicznych kluczy Google, aby mieć 100% pewności, że żądanie przyszło przez IAP.

Dlaczego inne są błędne:
B, D: Nagłówki tożsamości (np. x-goog-authenticated-user-email) są dodawane przez IAP, ale bez weryfikacji JWT mogłyby zostać sfałszowane, gdyby ktoś uzyskał dostęp do sieci wewnętrznej z pominięciem proxy.
C. x-forwarded-for: Ten nagłówek zawiera adres IP klienta, który łatwo sfałszować i nie stanowi on dowodu autentyczności żądania w modelu Zero Trust.

Konsensus społeczności ExamTopics: Zdecydowana większość wskazuje na A. Weryfikacja JWT to standardowy krok implementacji IAP zapobiegający atakom typu 'bypass'.

Analiza i Podsumowanie Dyskusji (AI)

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

Głęboka analiza: IAP chroni wejście, ale 'Defense in depth' nakazuje aplikacji sprawdzić, czy 'bileter' (IAP) rzeczywiście podpisał bilet. Google regularnie rotuje klucze do tych podpisów.
