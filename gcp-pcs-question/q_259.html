Question 259 / 361

Your team maintains 1PB of sensitive data within BigQuery that contains personally identifiable information (PII). You need to provide access to this dataset to another team within your organization for analysis purposes. You must share the BigQuery dataset with the other team while protecting the PII. What should you do?
  • A. Utilize BigQuery's row-level access policies to mask PII columns based on the other team's user identities.
  • B. Export the BigQuery dataset to Cloud Storage. Create a VPC Service Control perimeter and allow only their team's project access to the bucket.
  • C. Implement data pseudonymization techniques to replace the PII fields with non-identifiable values. Grant the other team access to the pseudonymized dataset.
  • D. Create a filtered copy of the dataset and replace the sensitive data with hash values in a separate project. Grant the other team access to this new project.
Show Suggested Answer Hide Answer
Suggested Answer: A

Wyjaśnienie

Odpowiedź: A.

Wyjaśnienie (Dlaczego poprawna):
A. Row-level access policies (and Dynamic Masking): BigQuery pozwala na definiowanie polityk dostępu na poziomie wierszy oraz kolumn (Dynamic Data Masking). Możesz udostępnić ten sam dataset drugiemu zespołowi, ale zdefiniować regułę, która dla ich tożsamości automatycznie maskuje kolumny PII (np. zamienia email na 'XXXXX'). Jest to najbardziej elastyczne rozwiązanie, nie wymagające kopiowania ogromnych ilości danych (1PB).

Dlaczego inne są błędne:
B. Export to GCS: Bardzo kosztowne i powolne przy 1PB danych, tworzy niepotrzebną kopię.
C. Pseudonymization of whole dataset: Wymagałoby stworzenia nowej tabeli 1PB, co jest nieefektywne kosztowo.
D. Filtered copy: Ponownie, tworzenie fizycznej kopii 1PB danych jest niezalecane, jeśli można to zrobić logicznie przez polityki dostępu (A).

Analiza i Wyjaśnienie

Sugerowana odpowiedź: A

BigQuery Row/Column-level security pozwala na bezpieczne współdzielenie danych (Multi-tenancy) bez ich duplikacji.
